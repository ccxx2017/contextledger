# MAINTENANCE_LOOP · 折叠过程规格

本文件管**过程**：谁在什么时候、按什么条件、调用什么、失败怎么办。
内容规则（什么住哪、怎么写）在 README，执行规格（折叠时遵循的规则）在 maintenance_prompt.md。本文件不复制那两份的内容，只指路。

当前阶段：**人在 loop**。每节先写"人怎么做"，再写"agent 接手时这一格变成什么"。
两栏不一致时以"人怎么做"为准——它是正在被验证的真相，agent 栏是待实现的草稿。

---

## 一、触发条件：什么时候该折叠

**人怎么做：**
- 触发阈值见 maintenance_prompt.md（全局唯一真源），目前累计 N 轮一次。
- 「一轮」的定义：________（待你填：一问一答？一个完整 task？一次 session？）
- 轮次计数现在存在哪：________（待填：人脑记 / 某个计数文件 / 对话条数）
- 边界判断：当一轮跨越多次往返、或一次往返含多主题时，怎么算？________

**agent 接手时：**
- 计数必须落成可观测状态（如 `run/round_counter`），不得隐式存于人脑。
- 触发器读计数 + 读阈值，返回 should_fold: bool。这一步是确定性逻辑，不走 LLM。

---

## 二、校验契约：折叠产出怎么算合格

**人怎么做：**
- 铁律2 机械动作：数「长期记忆」节与「脊柱」节行数，新 ≥ 旧，唯一例外是显式 `[DEPRECATED]`。
- 现在谁来数：________（待填：人肉数 / 已有脚本？）
- 还核对什么人话判据：________（待填：分流是否归错节？技能卡是否重复？锚点格式对不对？）

**agent 接手时：**
- 行数核对必须是确定性校验脚本，**不交给 LLM 自检**（LLM 数数不可靠）。
- 脚本契约：
  - 输入：旧版 memory_pack、新版 memory_pack
  - 检查：两节行数比、DEPRECATED 例外、锚点正则合规
  - 输出：pass: bool + 失败明细（哪节少了几行）
- 「分流归错」「技能卡重复」这类语义判据，看是否能脚本化；不能的部分明确标注为"仍需 LLM 评审"，不要假装它已自动化。

---

## 三、编排回路：谁调用谁、失败怎么办

**人怎么做（现在你就是这个控制器）：**
1. 判断该折了（第一节）
2. 备齐 4 份输入，喂折叠 LLM
3. 拿到产物A/产物B
4. 跑校验（第二节）
5. 不合格 →________（待填：你现在怎么做？重喂？手改？打回让模型重来？）
6. 合格 →________（待填：怎么落盘？memory_pack 覆盖写？skills 手动 append？raw 怎么归档？）

**agent 接手时：**
- 需要一个显式编排层（现在这层由你扮演，README/prompt 里都没有它的位置）：
  - 调用顺序、重试次数上限、重试仍失败的升级路径（升级给谁/停机等人？）
  - 落盘的原子性：产物A覆盖 memory_pack 与产物B append skills 是否要同成功同失败？
  - 每轮快照写入 run/（呼应你 TaskGraph 那边"脚本自动生成轮次快照、不手动复制"的偏好）

## 四、迁移检查表（什么时候算"可以交给 agent 了"）

逐项打勾，全绿才切：
- [ ] 轮次计数已落成可观测状态，不依赖人脑
- [ ] 行数校验已是独立脚本，且脱离 LLM
- [ ] 失败重试 / 升级路径已写死
- [ ] 落盘原子性已定义
- [ ] 仍需 LLM 评审的语义判据已被显式列出（哪些是 agent 也得调模型判的，别假装全自动）
- [ ] 手动期至少完整跑过 ____ 次且无静默丢失
