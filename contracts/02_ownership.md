# 单一正文归属契约 [STABLE]

## 铁律(不变)
任一条知识,其正文(canonical text)只能存在于以下三处之一:

  1. graph 节点        —— 事实、状态、约束等"可被机械失效的命题"
                          (含 Constraint;其失效表现为 status=superseded,见 03)
  2. pack 叙事         —— 给人读的概览/项目故事
  3. pack 技能(skills)—— 图装不下的软料:方法论、偏好、技能

## 推论
- 另外两处若需提及该知识,只能引用,不得重述正文
- 引用方式:graph 用 node_id,pack 用 raw_id 或 node_id
- 事实与状态一律归 graph;pack 叙事即便复述了某事实,也只是"渲染",
  不是正文,装配时以 graph 为准(见 04)

## 判定口诀(修订:分界不是"会不会变",而是"失效能否机械裁定")
注意:graph 与 pack 的内容【都会】随时间被推翻——
     graph 的事实会被新事实失效,pack 的偏好/方法也会被后续推翻(降级规则)。
     所以"会不会被推翻"不能作为分界。真正的分界是:

  "它的失效是不是客观、可机械裁定的状态命题?"
    是 —— 有明确实体(entity_ref)、有离散状态(state)、靠互斥矩阵判真假
         -> 归 graph(失效=status 置 superseded,机械裁决,见 03)
         例:工单状态、接口是否上线、约束阈值、文件版本

    否 —— 软性命题,无客观互斥,失效靠人/叙事软标记
         -> 归 pack(失效=软标记 [DEPRECATED],非机械裁决)
         例:编码风格偏好、调试方法论、个人习惯

## 边界澄清(防误判)
- 偏好/方法被推翻时,不进 graph 做机械失效,仍留 pack 内按降级规则标记。
- 一条知识若"既像约束又像偏好",问:它有没有可枚举的离散状态?
  有(在force/失效) -> Constraint,归 graph
  没有(只是倾向)   -> 偏好,归 pack
