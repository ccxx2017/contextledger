# 长对话压缩的二次压缩方法讨论

用户提出，现有的一般性对话压缩方法（保留逻辑链条、去掉对话体、每轮加标题）会导致压缩内容越积越长，不适合长期作为上下文。询问能否在已有压缩基础上进行继续精简，即二次压缩。

GPT-5.5建议采用二次压缩，核心思想是将“一轮压缩”视为保留过程（回答“之前发生过什么”），而“二次压缩”应转为保留状态（回答“现在继续对话时AI必须知道什么”）。具体方案如下：

## 三层压缩机制

1. **轮次压缩**：保留每轮逻辑链条、有标题、去掉对话体，适合归档但不适合反复喂给大模型。
2. **主题级压缩**：当某个主题积累较长时，按主题合并成摘要，包含项目目标、当前代码状态、已解决问题、关键决策、废弃方案、仍需注意的问题、下一步、重要文件与参数。
3. **当前上下文交接单**：最适合下次直接提供给大模型的版本，不追求完整历史，而是保证继续工作不中断。结构包括：当前目标、当前有效状态、已完成修复、关键约束、废弃方案、当前未解决问题、重要文件与关键修改点、下次继续时应优先做什么。

## 二次压缩的核心原则

- 从“历史叙述”转为“当前状态”：只保留最终结论和注意事项，删除中间猜测、多次试错、已不影响后续的内容及大段代码。
- 对已解决的问题只保留“现象-根因-最终修复”。
- 单独列出“废弃方案”以防AI重犯错误。
- 对代码只保留文件名、函数名、职责和关键修改点。

## 推荐使用的二次压缩提示词

**首次生成交接单**：
```
接下来请你对已有压缩内容做“二次压缩”。目标不是重新复述历史，而是生成一个适合下次继续对话时提供给大模型的“当前上下文交接单”。要求：保留继续工作必须知道的信息（当前目标、有效状态、关键约束、已完成修复、废弃方案、未解决问题、重要文件、下一步）；删除过程细节和代码主体；区分有效、废弃和待解决问题；输出固定结构。
```

**在已有交接单基础上更新**：
```
接下来请你基于“已有上下文交接单”和“新增压缩内容”，生成新的上下文交接单。要求：合并、去重、更新状态；若新内容推翻旧内容，将旧内容移至“废弃方案”；已解决的问题移至“已完成修复”；输出固定结构。
```

## 结论

轮次压缩用于归档，二次压缩（上下文交接单）用于继续工作。定期将长压缩稿重写成交接单，可大幅减少token消耗，维持对话连续性。

# 一般性对话压缩的改进方案讨论

用户询问Claude-Opus-4.8对GPT-5.5提出的二次压缩方案的意见。Claude-Opus-4.8基本认同“过程归档、状态续作”的核心判断，但指出三点不足并提出改进方案。

## 主要问题

1. **模板过于技术化**：GPT-5.5的交接单结构（代码状态、重要文件等）适合Minecraft调试，但不适合用户明确提出的“一般性对话”（如课程构思、写作讨论、方案权衡）。建议分两类：技术类用原模板，一般类只用三块——当前讨论什么、已达成共识、悬而未决的分歧。

2. **牺牲了“脉络”**：用户要求压缩要保留“整体的脉络”，而GPT-5.5的“只留状态、删掉过程”恰恰删除了脉络。纯状态压缩有界但不保留脉络，保留脉络的压缩天然会增长。解决方案：保留一条“细脊柱”——每个大阶段只留一行转折（如“先按A做，因X放弃，转向B”），这条脊柱增长是次线性的，其余细节才压成状态。

3. **周期性重写不能根治增长**：GPT-5.5的“隔段时间做二次压缩”只是偶尔变短，长期仍会再涨。建议改成有界结构：
   - **稳定区**（上限固定）：目标、关键约束、已闭环结论、细脊柱
   - **滚动区**（只保留最近N轮）：近几轮的详细压缩。每加一轮新内容，就把最旧的一轮折叠进稳定区（只抽出脊柱一行+新结论），总长度趋于常数。

## 额外提醒：压缩要“可重入”

AI在已压缩内容上再压时，常会反向展开或加说明，导致不降反升。提示词必须明确：输入若已是压缩稿，只做合并去重，禁止复述、禁止扩写、禁止加说明。

## 推荐通用型二次压缩提示词

```text
请对已有压缩内容做二次精简，生成“可长期续用的上下文底稿”。

规则：
1. 已闭环的内容只留结论，不留推导和试错过程。
2. 每个大阶段保留一行“转折说明”（做了什么→为何转向），用于维持脉络。
3. 输入若已是压缩稿，只做合并、去重、更新；禁止复述、扩写、添加解释。
4. 区分三类信息：仍然有效 / 已被推翻 / 仍待解决。
5. 输出分两部分：【稳定区】目标、关键约束、已定结论、脉络脊柱（每阶段一行）；【近况区】最近几轮的详细要点。
6. 将【近况区】中已闭环的旧轮次折叠进【稳定区】，只保留其脊柱一行。
7. 只输出底稿本身。
```

## 结论

GPT-5.5方向正确，但模板偏技术且会牺牲脉络。真正需要的是“固定稳定区 + 滚动近况区”的有界结构，并保留一条次线性增长的脉络脊柱。
# 对话原稿的保存与检索策略

用户询问：一次压缩会存档，但原始对话稿是否需要保存？折叠或存档后的内容在必要时是否需要检索？

GPT-5.5建议：原稿必须保存，但作为冷存档的证据库，不默认喂给大模型；一次压缩稿保存为过程档案；二次压缩稿作为日常上下文；折叠内容应设计为可检索，但只在少数场景按需召回。

## 四层管理方案

1. **原始对话稿**：保存为“证据库”，用于压缩摘要不确定、结论有争议、需要还原细节时的最终依据。日常不给AI。建议格式：`archive/raw/日期_主题.md`，并标注对应的一次压缩和二次压缩文件。

2. **一次压缩稿**：保存为“过程档案”，比原稿轻但保留完整逻辑链条，适合回顾讨论过程。日常不必喂给AI，仅当二次压缩不够用时回查。

3. **二次压缩稿/上下文底稿**：日常默认使用，采用“稳定区+脉络脊柱+近况区”的有界结构。

4. **折叠/存档内容**：需要设计为可检索，但不需要经常回查。典型需要回查的场景包括：当前摘要表述模糊、AI重复犯旧错、需要恢复被删细节、结论受挑战需确认依据、重新做更高质量的二次压缩。

## 推荐文件体系

```
project_context/
  current/
    context_base.md              # 当前二次压缩稿
  rolling/
    recent_rounds.md             # 最近N轮的一次压缩摘要
  archive/
    compressed/                  # 一次压缩稿存档
    raw/                         # 原稿存档
  index.md                       # 索引
```

## 索引示例

```markdown
# 对话资料索引
## Minecraft AI Agent 调试
- 当前底稿：current/context_base.md
- 一次压缩：archive/compressed/2026-06-06_minecraft_agent_rounds.md
- 原稿：archive/raw/2026-06-06_minecraft_agent_raw.md
- 关键词：RCON、mcpi、GBK、QUERY
- 状态：仍在进行
```

## 实用原则

- **日常续作** → 二次压缩稿
- **细节补充** → 一次压缩稿
- **事实核验/原话还原/重新压缩** → 原稿

不要把“保存”和“喂给模型”混为一谈。保存是为了不丢证据，喂给模型是为了当前任务高效推进。文件体系先用文件夹+索引，资料量大后再考虑向量检索或知识库。

# 长对话记忆治理协议最终方案

GPT-5.5提出最终方案：建立“上下文底稿 + 长期记忆/技能 + 原稿证据库 + 按需检索”的四层记忆治理流程，命名为“长对话记忆治理协议 v1”。

## 一、四层结构

1. **当前上下文底稿**：日常喂给AI。包含稳定区（目标、约束、结论、方案、废弃项、未解决问题）、脉络脊柱（每阶段一行：做了什么→为何转向→结果，带锚点或日期）、近况区（最近N轮要点）。目标：让对话继续，不无限增长。

2. **长期记忆/语义事实**：保存以后还会用到的事实（如“RCON命令不能带斜杠”），带适用范围、来源、状态。目标：积累可复用经验。

3. **技能库**：保存可迁移的方法。技能卡格式：触发场景、操作步骤、已验证经验、适用边界、来源。目标：将经验晶化为程序记忆，实现代际传播。

4. **原稿证据库**：冷存档，不日常喂AI。作为最终证据、可追溯来源、重新压缩材料、争议核验依据。

## 二、一次压缩稿的处理

一次压缩稿不必永久作为核心资产。建议：项目活跃期保留以方便回查；阶段闭环后折叠；若原稿完整，一次压缩稿非必需。最终必须长期维护的是当前底稿、长期记忆/技能、原稿证据；一次压缩稿可选保留。

## 三、闭环内容四分流规则

| 内容类型 | 处理方式 |
|---|---|
| 通用可复用 | 升级为技能库 |
| 项目内长期有效 | 写入长期记忆 |
| 影响当前推进 | 留在当前上下文底稿 |
| 一次性过程细节 | 冷存原稿 |

核心原则：**闭环不是删除资格，而是晶化资格。**

## 四、按需召回顺序

当前上下文底稿 → 长期记忆/技能库 → 脉络脊柱锚点 → 一次压缩稿 → 原稿。仅在模糊表述、AI重复旧错、结论被挑战、需还原细节、重新压缩、升级技能、审计时回查。

## 五、最小落地结构

三个文件或一个“项目记忆包.md”包含：当前上下文底稿、长期记忆、技能库、原稿索引/脉络锚点。

# 压缩流程理解与澄清

用户概括了四个理解点：1）先有一次压缩（原则未定但可用）；2）折叠入库环节使用模板投喂大模型，但记忆包中的近况区与本轮压缩稿的关系不明确；3）人工粘贴入库（更新记忆包、技能卡、原始对话）；4）在技能库末尾记录判定日志。

Claude-Opus-4.8指出大体正确，但有三处需要纠正，特别是第2点。

## 关键纠正

**第1点**：一次压缩的产物 = 本轮压缩稿，是同一份临时桥稿，不落盘，喂完就丢。

**第2点（核心）**：两个附件需彻底分清——【现有记忆包】是`记忆包_项目名.md`全文（含稳定区/脊柱/近况区/长期记忆）；【本轮压缩稿】是新一轮对话的一次压缩，尚未进入记忆包。两者是**存量与增量**的关系，不重复。

近况区装的是**最近3轮的一次压缩**（即之前几轮已折进来的桥稿原样），不是raw全文也不是只剩一行的脊柱。一轮完整生命周期：raw原稿 → 一次压缩（本轮压缩稿）→ 进近况区当“最新一轮” → 新轮进来、旧轮超过3轮时折叠（抽出脊柱一行 + 闭环内容四分流）→ 详细压缩从近况区删除，细节去raw找。模型执行提示词时同时做三件事：纳入本轮压缩稿、折叠最旧一轮、产出技能卡草稿。

**第3点**：记忆包由模型输出全文，用户核对后**整体替换**；技能卡是草稿，需人工审核后粘入技能库；raw原稿是独立动作（每轮随手存证据），不叫“入库”。

**第4点**：判定日志并进`技能库.md`末尾一节，每次一行，试点期不单独建文件。

## 结论

用户的理解只差第2点“存量vs增量”的区分。建议在记忆包顶部贴一张单轮数据流图（raw → 一次压缩 → 近况区 → 折叠分流）作为操作备忘。

# 单轮数据流操作备忘

## 一、操作备忘核心内容

直接贴在`记忆包_项目名.md`顶部作为操作指南：

### 一轮的完整生命周期

- 原对话 → ①一次压缩（保留逻辑链/去对话体/每轮加标题，不落盘）→ 本轮压缩稿（增量）

- 模型同时做三件事：
  - A. 将本轮压缩稿纳入近况区，成为“最新一轮”
  - B. 将超出3轮的最旧一轮折叠：抽脊柱一行（做了什么→为何转向→结果，挂raw锚点），闭环内容做四分流，该轮详细压缩从近况区删除
  - C. 产出技能卡草稿
- 独立动作：原对话随手存为`raw_日期_主题.md`，按`## 轮次NNN`分节

### 四分流规则

- 通用可复用 → 技能卡草稿（人工审后粘进`技能库.md`）
- 项目内长期有效 → 写入长期记忆节（状态=已验证）
- 仍影响推进 → 留在稳定区
- 一次性细节 → 不保留（raw已存档）

- 记忆包：核对模型输出全文后整体替换
- 技能卡：人工审核草稿后粘入技能库
- 判定日志：每次一行，写进技能库末尾节
- raw文件：每轮结束随手存

- **近况区** = 最近3轮的一次压缩（详细压缩）
- **本轮压缩稿** = 本轮新产生、尚未进库的那一份

- 全局技能库全文（建议每次都给，用于判断能否晶化新卡及避免重复）

注意：`_templates/`下的规则文件不喂给模型，规则已写在提示词中。

## 输出：双产物模式

原输出节只要求记忆包全文，遗漏了技能产出。现修正为两种模式：

**产物A · 记忆包全文**（必出）
采用全文重写，因其四节交织（开放问题转入长期记忆、脊柱追加），全文+行数核对最稳。

**产物B · 技能追加片段**（仅当有方法论级内容时输出）
只输出新增技能卡，不重写全文。理由：技能库是全局资产，一旦让模型每次重输全文，老卡面临被静默改写风险。采用追加模式，老卡碰都不碰。新增卡片必须含`source:`字段（出自哪个项目）。

配套指针：在记忆包脊柱对应轮次锚点后附`→ skills: <卡名>`，以便追溯复用。

## 修订后的维护提示词关键改动

- 规则4新增“技能卡留痕”：晶化出技能卡时，在脊柱锚点后附轻指针
- 输出节拆分为“产物A·记忆包全文”和“产物B·技能追加片段”，后者无新增时写“（本轮无新增技能卡）”

技能卡的字段模板可后续定，但`source:`是必备。

# 记忆治理方案收尾总结与问题澄清

## 整体评价

方案设计已闭合，核心资产包括：有界结构（记忆包四节、全局技能库、raw证据层、脊柱锚点）、折叠指令（4份输入、双产物输出）、三条铁律（单一真源、存量逐字保留+行数核对、raw永不改名）、两周体检机制。关键安全设计包括记忆包全文重写配行数核对防静默丢失、技能库append-only防全局污染、判定日志显式化隐性判断。唯一失败风险是维护成本导致弃用，设计已将每场人工动作压至核对行数、审核技能卡、存raw三件事。

注意设计变更：早期“近况区（最近3轮）”已取消，“3轮”含义从保留窗口变为折叠节奏——每3轮触发折叠，记忆包只存状态和脊柱，细节全在raw。

## 问题1：raw文件夹存什么

**裁定：存原始对话原文**，按主题线一个文件，内部用`## turnNNN`分节。raw是证据库（摘要有争议、AI重犯旧错、需还原细节时的最终依据），不能是有损的一次压缩稿。需修改README：数据流中`─(fold)→ raw`改为`─(archive)→ raw`；4份输入清单第2条改为“本轮raw段落的一次压缩桥稿（临时产物，不落盘）；轮次短时可直接喂raw原文”。

## 问题2：是否要先做“一次压缩”

**是，但作为临时桥稿不落盘**。每次折叠完整动作：取3轮raw原文，做一次标准轮次压缩（保留逻辑链、去对话体、加标题），产出桥稿；然后将4份输入（maintenance_prompt指令、桥稿新料、记忆包全文存量、技能库全文参照）喂给模型；拿到产物A（记忆包全文，行数核对后整体替换）和产物B（新技能卡，人工审核后append）；桥稿用完即丢。若3轮很短可跳过桥稿直接喂raw原文。

## 问题3：已有存量多轮压缩（如附件）如何处理

**做一次性“首场折叠”冷启动**，不按3轮节奏回放。

步骤：
1. 将附件存为`projects/taskgraph/raw/raw_2026-06-12_compression.md`（虽非原稿，但原始对话可能无法找回，文件顶部标注“本文件为一次压缩稿，原稿未存”作为既成事实，铁律保护新内容）。
2. 做首场折叠：4份输入中“新料”即整份附件（覆盖全部历史轮次的桥稿），记忆包用空壳模板。产出第一版记忆包（建立行数基线）和第一批技能卡（如“可重入压缩”“存量逐字保留+行数核对”“单一真源”“全局资产只追加”等）。
3. 从下一场对话起进入标准节奏：每轮结束存raw，攒满3轮触发折叠。新范式无近况区，开新会话时的喂法为：记忆包 + 上次折叠后尚未折叠的raw段落（最多2轮）。记忆包保持有界，不与raw重复。
# 记忆治理的“支流与水池”架构讨论

用户提出新想法：当前的记忆治理是从时间维度沉淀对话上下文（类似于人类的日常记忆支流），但还需要领域知识积累（类似于长期学习形成的专业知识水池）。上下文工程管理应同时具备支流和水池。

Claude-Fable-5认为此直觉精准，对应记忆科学中的情景记忆（支流）与语义记忆（水池）。并与Karpathy的知识库体系对比，发现二者对偶：

- **支流（情景记忆）**：按时间组织，回答“我们进行到哪了”；必须有界（每场对话需全量喂给模型），靠滚动折叠维持。
- **水池（语义记忆）**：按概念组织，回答“关于X我知道什么”；可无界，靠索引按需检索，不依赖全量入上下文。

现有体系已包含水池胚胎：折叠操作是河口，判断哪些知识脱离原始语境后入池（技能库、长期记忆）。但目前池子仅为平面列表，长期会退化为堆积。未来需引入wiki结构：概念页、交叉引用、lint检查。

完整架构：
- 外部资料可直接入池（不经过对话）
- 对话经raw、折叠后进入支流（memory_pack）及水池（技能+wiki）

但**现在不建**，等待触发信号：某项目长期记忆超二三十行、领域事实跨项目重复落、概念查询需求出现。届时再升格为wiki。当前仅需在README末留远景钩子。

核心价值：不是池或流本身，而是“折叠作为河口”的输运机制。

# memory_pack 是否可直接作为对话上下文

用户询问：`memory_pack.md` 是最终提交给一般对话 LLM 的上下文，还是需要再次加工。

Claude-Opus-4.8 回答：`memory_pack` 是“准上下文”，大部分可直接喂，但有一节需要处理。

## 两种读者区分

- **折叠 LLM**（维护用）：需要看全四节（概览、长期记忆、脊柱、开放问题），以知道如何追加内容。
- **日常对话 LLM**（一般对话）：只需要“我现在该记住什么”。

## 逐节分析

| 节 | 是否直接喂 | 说明 |
|---|---|---|
| 概览 | ✅ 直接喂 | 项目定位 |
| 长期记忆 | ✅ 直接喂 | 事实、偏好、硬约束，核心内容 |
| 脊柱 | ❌ 需加工 | 锚点（如 `raw/...#turn003`）是给人和检索系统的，对话 LLM 无法访问 raw 文件，锚点是死字符串。应保留脉络文字，去掉锚点 |
| 开放问题 | ✅ 直接喂 | 当前活线程，帮助续接 |

## 推荐方案

**投影（轻量加工）**：从 memory_pack 生成一份对话 LLM 可用的上下文，只包含“概览 + 长期记忆 + 开放问题 + 脊柱（仅脉络句、去锚点）”。投影是只读、临时、不落盘的操作，与折叠流水线隔离。

可实现一个类似 `project_context()` 的函数，符合用户“先模板控制、再脚本实现”的工程节奏。

## 结论

`memory_pack` 不是最终投喂格式，但只需轻量投影（去锚点）即可。建议将此规则写入 README 数据流节，明确区分两种读者的使用方式。
