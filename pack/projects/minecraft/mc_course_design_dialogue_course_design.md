# 一、课程设计与教学策略

## 主题：面向青少年的Minecraft AI教学课程大纲（六个阶段，从自然语言问答到多AI分工与记忆）

### 课程定位
以青少年对手动玩MC的热情为起点，以“让游戏世界越来越听话”为主线，将AI、编程思维、系统设计概念融入游戏体验。不追求背诵知识，而是让青少年亲眼看到“AI是如何工作的”。

### 第一阶段：AI是会说话的百科全书
- 起点：孩子玩游戏时遇到问题（如“怎么建不漏雨的房子”、“苦力怕怎么躲”）。
- 做法：引导孩子尝试用网页版大模型提问。
- 核心体验：发现AI很博学，但不知道自己位置、背包状态，没有手和眼睛。这个“局限”成为后续学习的驱动力。

### 第二阶段：代码是AI的手
- 展示老师已搭建的Python脚本：老师按几个键，学生游戏里出现房子或获得钻石剑。
- 核心概念：代码就是AI的手，没有手，AI只能说不能做。
- 技术基础：局域网连接、mcpi（建造）、rcon（给物品）。
- 目标：孩子能说清楚“老师的电脑做了什么”，不要求写代码。

### 第三阶段：说一句话，游戏就行动（理解与执行分离）
- 核心系统：几十行Python，将自然语言指令通过大模型转换为`CMD:`、`QUERY:`或Python代码前缀，再路由执行。
- 精彩教材：三个真实失败案例
  1. **斜杠问题**：AI生成带`/`的命令，RCON无法识别 → 防御性代码修复。
  2. **数量问题**：用户“去掉一把宝剑”，AI清空所有 → 提示词补充“删除必须指定数量”约束。
  3. **查询问题**：用户“我有几把宝剑”，AI反而给了一把剑 → 增加`QUERY:`分支，强制AI回复无法查询。
- 核心启发：AI擅长理解意图，但对自己能力边界没有清醒认知。我们的工作是通过规则和格式协议（字符串前缀）来弥补这个缺口，实现理解与执行的分离。

### 第四阶段：给AI装上眼睛（感知两条路径）
- **路径一：mcpi结构化扫描**
  - 可用接口：`getTilePos()`, `getRotation()`, `getBlock()`, `getBlocks()`, `pollBlockHits()`。
  - 实现方案：后台线程定时扫描5x5x5区域，统计水/岩浆/树木，检测脚下方块、头顶是否洞穴，注入Prompt。
  - 优点：精确坐标、实时、可编程；缺点：看不到UI（血量、背包）。
- **路径二：豆包视觉语义感知**
  - 手机摄像头看屏幕，理解“血量低、面前悬崖、天色变暗”。
  - 优点：语义丰富、人类友好；缺点：描述模糊，无法直接执行。
- 两条路径的互补性：mcpi精确定位，豆包发现问题。

### 第五阶段：两个AI的分工合作（系统思维）
- 完整闭环：
  - mcpi结构化扫描+豆包（视觉层）→ 描述状态 → DeepSeek（决策层，拥有技能文件+记忆）→ 输出mcpi/RCON指令 → 执行
  - 或 mcpi扫描（精确层）提供坐标数据 → 合并到决策层。
- 类比：厨房里服务员（豆包）、食材清单（mcpi）、厨师（DeepSeek）、灶台（执行层）。
- 学习成果：孩子能画出或说出系统各部分的职责。

### 第六阶段：让AI变得更懂你（记忆和技能）
- **记忆**：用JSON文件存储玩家偏好、建造历史、已知事实，每次请求注入Prompt。
- **技能**：用SKILL.md形式的文本文件存储Minecraft领域知识（物品ID、建筑模式、苦力怕距离等）。
- 核心体验：对比“无记忆AI”和“有记忆AI”在同一问题上的表现，理解上下文的价值。

### 课程主线（一句话总结）
AI能说话 → AI有了手 → AI能听懂你 → AI有了眼睛 → 多个AI学会分工 → AI记住了你

每一步都由孩子在游戏中的真实问题驱动，而非抽象概念。

## 主题：课时5“AI军师的软肋”教案（纯电脑版）核心框架压缩

### 课时基本信息
- **所属阶段**：第二阶段·第2课（共2课时，当前阶段）
- **时长**：60分钟，7岁学生
- **核心目标**：让孩子自己发现并说出：“AI很聪明，但它看不见、没有手、反应慢，不能帮我动手做。”

### 教学准备
- 文件：`MC询问豆包.txt`（上节课对话记录）
- 学生电脑：Minecraft，已进入**冒险模式**，安全区域出生点
- 老师电脑：豆包网页版登录，屏幕朝向孩子
- 白板/大白纸 + 笔，两张便利贴（“脑子”“手”）

### 教学流程（缩写）

#### 1. 热身回顾（7分钟）
- 展示上节课真实对话记录，逐条指读。
- 追问：“谁去挖木头、杀蜘蛛、合成弓箭？” → 孩子回答“我自己”。
- 白板写关键词：**“豆包：动嘴”**。

#### 2. 冒险挑战：建安全屋（约35分钟）
**任务**：冒险模式第一晚活下来。孩子可问豆包，但每次只能问一个问题，老师当“打字员”。

**三阶段暴露局限性**：
1. **选址 → 暴露“看不见”**（8分钟）
   - 老师替孩子问：“选什么地方建房子最安全？”豆包答：“找平坦开阔、靠近树木的地方。”
   - 孩子执行后，老师连环问：你脚下是什么方块？豆包知道吗？左边有什么？身后有苦力怕吗？豆包知道吗？
   - 引导孩子说出：“它看不见！” → 白板写 **“豆包：看不见🙈”**。

2. **建造 → 暴露“没有手”**（12分钟）
   - 老师替问：“最简单的生存小屋怎么搭？”豆包给三步（地基、柱子、封顶）。
   - 孩子动手建造时手忙脚乱，老师喊暂停。
   - 追问：豆包帮你砍树/合成/放过方块吗？所有事谁做的？
   - 引导出：“没有！我自己做的！” → 白板写 **“豆包：没有手🦾❌”**。

3. **防御 → 暴露“反应慢”**（8分钟）
   - 僵尸出现（或老师主动提示天黑），孩子紧急提问：“僵尸追我怎么办？”
   - 老师故意慢速打字，等豆包生成后念出答案（如“用方块堵门”）。
   - 引导感受时间差：从提问到得到答案花了多久？豆包来得及救你吗？真正救你的是谁？
   - 引导出：“它反应太慢了！” → 白板写 **“豆包：反应慢🐢”**。

**环节总结**（1分钟）：老师指着白板上三条，总结“豆包三个毛病”。

#### 3. 总结提炼：大脑和手的相遇（12分钟）
- **画图 + 贴标签**：老师画大脑和打叉的手。让孩子贴“脑子”（给AI）和“手”（给代码，回忆第一阶段）。
- **引出核心悬念**：把“脑子”和“手”并排放，中间画一条线连起来，问：“如果把它们连起来，我们对游戏说‘给我造个安全屋’，它就真的自己动起来了？”停顿3秒制造悬念。
- **手势歌收尾**：手指太阳穴（AI有脑子）→双手比机器爪（代码有手）→双手蒙眼（AI看不见）。重复两遍。结束语：“下节课当连接师，把AI的大脑和代码的手接在一起！”



## 主题：第6课“奇迹发生——说一句话，房子拔地起”教案压缩版

### 课时基本信息
- **时长**：60分钟，7岁学生
- **所属阶段**：第三阶段第1课（总第6课时）
- **核心目标**：直接回应课时5悬念，让孩子第一次体验“自然语言控制游戏”的完整闭环，**纯体验课，不讲原理**。

### 教学准备
- 老师电脑：已预测试`mc_ai_agent.py`，确保5个以上指令成功（如钻石剑、改时间、3×3金块地板、一圈火把、玻璃房子），成功指令用大字写在白板上作为“今日魔法菜单”。
- 学生电脑：Minecraft已连接。
- 上节课“大脑→手连线图”的照片或留白板。

### 教学流程（60分钟）

#### 1. 回顾悬念（5分钟）
- 拿出上节课的大脑→手连线图，复习“豆包有脑子没手，代码有手”。
- 老师宣布：“今天已经把脑子和手连起来了。你说话，游戏真的会自己动。”

#### 2. 奇迹演示：第一个魔法（10分钟）
- 老师运行脚本，孩子输入`@ai 给我一把钻石剑`。
- 观察控制台（AI思考过程）和游戏画面（钻石剑出现）。
- 第一个总结：“AI脑子听懂，代码的手做了。”
- **小揭秘（3分钟）**：用卡片画出“你 → Python传话筒 → API小窗口 → DeepSeek大脑 → 返回 → 游戏执行”，解释传话筒和API的概念，不深究技术细节。

#### 3. 魔法菜单：你选，AI做（25分钟）
- 展示白板上的“今日魔法菜单”（5个成功指令+留空“你还可以自己发明魔法”）。
- 逐个体验（每个约5分钟）：孩子选指令→输入→观察控制台与游戏→感受结果→简短对话（如“谁动的手？”）。
- 推荐顺序：RCON类（物品/时间）→简单建筑→复杂建筑（玻璃房子作为高潮）。
- 若孩子发明未测试指令，成功则加入菜单，失败则记入“待学习魔法”。

#### 4. 指令失效实验（10分钟）——认知升华
- 孩子输入`@ai 我有几把宝剑`，AI返回“无法查询背包信息”。
- 对比讨论：“为什么刚才给剑可以，现在问有几把就不行？”（AI看不见背包）。
- 点题：“我们在代码里装了拦截网，让AI懂装不懂，比不懂装懂更厉害。”

#### 5. 感受收集与命名仪式（8分钟）
- 孩子说出体验：“我说话，游戏自己动了。”
- 回到大脑→手连线图，引出核心认知：**“变成了能听懂我说话、帮我做事的AI搭档”**。
- 孩子给搭档起名，写在白板上。做手势：脑子+手合在一起。

#### 6. 悬念预告（2分钟）
- “今天AI很乖，因为我们提前帮它准备好了。下节课我们要给AI考试，看它会不会犯错，然后当医生修理它。”



## 主题：课时7语音输入方案决策（阿里云STT）与课时定位调整

### 背景
- 课时6中孩子不会打字严重影响体验。Windows内置听写测试失败（识别差）。
- 用户申请了阿里云智能语音交互（ASR，“识音石V1-端到端模型”），用于语音转文字。

### 核心决策
- **采用阿里云ASR作为语音输入方案**，而非OpenClaw语音模块（架构过重，改变交互模型）或系统听写（效果差）。
- **交互方式**：孩子按住快捷键（如F2）说话 → 录音 → 阿里云ASR转文字 → 脚本自动拼接成`@ai 指令`并填入剪贴板 → 孩子在游戏聊天框Ctrl+V粘贴并回车。
- **优点**：中文识别准确率高、低延迟、国内稳定；使用剪贴板而非模拟键盘（避免反作弊拦截，容错高）。

### 对课程大纲的影响
- **语音作为输入方式升级**：在课时7开头增加“新工具介绍”（约3分钟），演示语音输入，不深入STT原理。后续指令输入由打字改为语音（快捷键+粘贴）。
- **备份方案**：语音失败时回退到老师协助打字。

### 技术路径确认
- 语音转文字（STT）是独立小工具，不修改核心`mc_ai_agent.py`，保持稳定性。
- 识别模型：阿里云“识音石V1-端到端模型”（实时语音识别）。


## 主题：第7课“AI小助手-语音输入升级

### 课时定位
- **总第7课时**，第三阶段第2课（承接第6课“AI搭档”）
- **核心目标**：引入语音输入（F2）解决打字慢问题。
- **关键词**：语音输入

### 课前准备与语音工具
- 已测试通过：Minecraft服务器、`mc_ai_agent.py`、RCON、语音脚本`voice_input.py`（阿里云ASR，F2按键说话 → 转文字 → 剪贴板生成`@ai 指令`）。
- 白板写“今日语音魔法菜单”（如钻石剑、苹果、改白天、火把圈、小玻璃房）及备用指令。

### 课堂流程（60分钟）

#### 1. 回顾上节课（5分钟）
- 复习“AI大脑 + 代码的手 = AI搭档”。白板画：我说话 → AI大脑 → 代码的手 → Minecraft动起来。
- 引入：“今天升级新本领：用嘴巴说，不用打字。”

#### 2. 新工具：F2语音魔法棒（10分钟）
- 老师演示：按住F2说“给我一把钻石剑” → 松开 → 剪贴板得到`@ai 给我一把钻石剑` → 粘贴到游戏聊天框 → 回车执行。
- 解释：`@ai`是喊AI名字，不喊它不知道指令是给它的。
- 孩子练习一次（如“给我一个苹果”），老师协助粘贴。

#### 3. 语音魔法菜单体验（15分钟）
- 孩子从菜单选指令，用F2说出，粘贴执行，白板打勾。推荐顺序：苹果（稳定）→ 改白天（改世界）→ 火把圈（建造）→ 玻璃房子（高潮）。
- 强化认知：“AI大脑听懂，代码的手执行。”



#### 6. 总结（5分钟）
- 白板总结：F2语音魔法棒
- 下节课预告：“AI还会犯更吓人的错：你说去掉一把宝剑，它可能清空所有。我们继续当AI医生。”

### 成功指标（达成任意三项）
1. 孩子能用F2完成一次语音输入。
2. 能说出`@ai`是在叫AI。

### 应急要点
- 语音识别失败 → 老师代为输入或孩子说老师打字；用“魔法棒感冒了”轻松带过。
- 剪贴板未复制 → 测试记事本粘贴，重启脚本。
- AI执行失败 → 记入“待学习魔法”，不说“坏了”。
- 孩子注意力分散 → 喊“暂停！魔法师收手！”



## 主题：第8课教案“宝剑全没了——AI忘了说几个”核心压缩

### 课时定位
- **总第8课时**，第三阶段第3课（承接第7课斜杠问题）
- **核心目标**：让孩子理解AI执行命令时必须说清“数量”（删除物品时若缺少数量参数可能导致清空全部），并引入“数量锁”规则。

### 核心概念
- AI可能听懂“去掉宝剑”但漏掉“1把”，导致生成`clear @a diamond_sword`（清除全部）而非带数量的`clear @a diamond_sword 0 1`。
- 修复方式：在提示词中明确要求删除物品时必须指定数量，并在代码层不做改变（提示词规则）。

### 课堂流程（60分钟）
1. **回顾第7课**（5分钟）：斜杠帽子与防撞护栏。
2. **故事引入**（10分钟）：小玩家的三把宝剑全没了，因为AI没说数量。
3. **玩偶剧场**（10分钟）：用三张纸片演示“拿走一把” vs “拿走全部”，孩子判断对错。
4. **真实安全实验**（15分钟）：先给多把剑（如“给我三把钻石剑”），再用语音说“去掉一把钻石剑”，验证只删除一把（系统已修好，不复现错误）。
5. **数量锁规则**（10分钟）：展示错误命令`clear @a diamond_sword`和正确命令`clear @a diamond_sword 0 1`，解释“最后的1就是数量”。告知这是AI规则书。
6. **小练习**（5分钟）：孩子判断指令是否需要数量（如“给我一个苹果”需要，“去掉所有泥土”危险）。
7. **总结与下节课悬念**（5分钟）：引出第9课“AI装作会，其实不会”。

### 课前准备
- 测试稳定指令：`@ai 给我三把钻石剑`、`@ai 去掉一把钻石剑`（仅使用已修好的系统，不真实执行危险命令）。
- 准备纸片（钻石剑道具）、白板、语音输入（可选）。

### 成功指标（达成任意三项）
- 孩子能说出“AI忘了数量”。
- 能理解“一把”和“全部”不一样。
- 能说出“删除东西要说几个”。
- 能用语音完成“给物品”和“删除一个物品”。
- 能理解规则书让AI更安全。

### 应急要点
- AI删除失败 → 用纸片演示，不现场调试。
- 语音失败 → 老师打字。
- 孩子想清空所有物品 → 说明“危险魔法只在测试世界用”。

### 核心教学语言（儿童版）
- “AI不能只说‘拿走宝剑’，必须说‘拿走1把宝剑’。”
- “删除物品时，一定要带数量，就像最后写个数字1。”

### 下节课预告
AI明明看不见背包，却假装知道 → 教AI说“我不知道”。



## 主题：第7课后反馈与第8课主题调整（舍弃斜杠/数量错误，改为产品体验与未来设计课）

### 实际问题反馈
- 第7课时间不够，斜杠问题未讲；即使讲了也不直观，孩子不感兴趣。
- 孩子核心痛点：语音输入后仍需复制粘贴、切换窗口，不能在游戏内直接说话，流程太麻烦。

### 第8课教学定位调整
- **不再讲解**：斜杠、RCON格式、clear数量锁、execute跨世界等工程细节。
- **新主题**：为什么现在这么麻烦？——让AI小助手更像真正的游戏助手。
- **核心目标**：让孩子理解当前系统由“耳朵（语音转文字）→大脑（AI决策）→手（RCON执行）”组成，但各部分未完全集成进游戏，导致需要人工搬运文字。引导孩子设计理想中的游戏内语音助手。

### 课堂流程（60分钟）概要
1. **承认问题，激发改造目标**（5分钟）：孩子指出“复制粘贴太麻烦”，明确目标“不用粘贴，直接在游戏里说话”。
2. **拆解AI小助手的身体**（10分钟）：画机器人图：耳朵、大脑、手、Minecraft，说明它们现在分散在不同窗口。
3. **画出当前麻烦流程**（10分钟）：孩子说话 → 语音窗口 → 复制 → 切到游戏 → 粘贴@ai → 执行。让孩子指出最麻烦的步骤（复制/切换）。
4. **设计未来版本**（15分钟）：列出版本1（现在，需粘贴）→版本2（按键说话自动发送）→版本3（游戏内直接语音）→版本4（AI像伙伴一样感知世界）。让孩子给版本打星。
5. **孩子当产品经理**（10分钟）：提问想去掉哪一步、AI起什么名字、危险命令是否要先确认。
6. **画“未来助手设计图”**（10分钟）：画出耳朵、大脑、手、Minecraft，并写出希望的功能（直接听、不用粘贴、帮忙建造/给物品、危险先问）。
7. **总结**（5分钟）：发现的问题不是AI不会做，而是用起来太麻烦；下一步目标让AI自己把话送进游戏。

### 课堂产出建议
- 生成“Minecraft AI助手升级清单”，如：不用复制粘贴、不用切窗口、不用打@ai、直接语音、危险命令先问、起名字。

### 后续工程伏笔
- 课后可改进：语音识别后直接调用AI决策并自动RCON执行，无需人工粘贴，第9课展示升级成果。

### 结论
第8课从“修bug”转为“产品改造课”，围绕孩子真实的体验痛点设计未来，更符合兴趣和教学效果。



## 主题：第8课后技术实现可行性分析（去除复制粘贴可行，原生语音聊天暂不承诺）

### 用户担忧
担心建议的“产品改造”（如在游戏内直接语音、无需复制粘贴）技术上无法实现，造成课堂尴尬。

### 可实现程度分层（供教学中承诺参考）
| 目标 | 是否建议承诺 | 技术可行性 |
|------|-------------|------------|
| 语音识别后不再复制粘贴，直接执行 | ✅ 可以承诺 | 高 |
| Minecraft窗口不切出，按快捷键说话自动执行 | ✅ 可尝试作为第9课升级 | 较高 |
| 在Minecraft聊天框里原生语音输入 | ⚠️ 不建议承诺 | 需Mod/插件/输入法层改造 |
| 不按键，AI一直听你说话 | ❌ 暂不建议 | 噪音、误触发、安全风险高 |
| AI像伙伴一样看见世界、记住你 | ✅ 后面阶段逐步做 | 需要感知+记忆 |

### 最稳的后续方案（第9课可承诺）
- **承诺内容**：去掉最麻烦的步骤——不用复制粘贴了。  
- **技术路径**：将语音识别结果直接传给`mc_ai_agent.py`，绕过剪贴板和手动粘贴。
- **新流程**：按快捷键说话 → 阿里云ASR转文字 → 直接交给`mc_ai_agent.py` → AI决策 → RCON/mcpi执行。  
  移除：复制、切窗口、粘贴、`@ai`、聊天框中转。

### 推荐实现方式
- 不改造`voice_input.py`独立执行命令，而是将其作为`mc_ai_agent.py`的一个输入入口。
- 共用同一个处理函数`handle_user_text(text)`，统一处理聊天文本和语音识别文本。
- 保持现有RCON/mcpi执行逻辑不变，只增加语音入口。

### 给孩子的说法（安全承诺）
> “上节课你发现AI小助手太麻烦了，要人帮它搬话。这节课我们把‘搬话’交给程序。它还不算真正住进Minecraft，但已经不用我们复制粘贴了。”

### 不要承诺的说法
- “以后你在Minecraft里随便说话，它都会听见。”（需要持续监听、唤醒词、焦点、反误触发等，复杂度高，不适合短期课堂承诺）

### 对第8课结尾的调整建议
- 第8课仍可做“产品改造课”，但结尾的下一步目标应设定为：“不用复制粘贴，按键说话后直接执行”，而不是一步到位变成游戏内原生语音助手。这样既回应痛点，又技术上可落地。



## 主题：第8课教案调整（改为“按按钮说话的AI助手”设计课）

## 主题：第9课“AI对讲机上线——按按钮说话，Minecraft自动响应”教案压缩版

### 课时定位
- **总第9课**，承接第8课设计，实现“按按钮说话，无需复制粘贴/切窗口/打@ai”的AI对讲机V1。
- **核心体验**：按住按钮说话 → 松开 → AI听懂 → Minecraft自动执行。
- **不讲**：Arduino代码、串口协议、ASR细节、RCON格式、命令版本。

### 核心目标
孩子能体验并说出：“以前是人帮AI搬话，现在是按钮帮AI搬话；不用离开Minecraft也能执行指令。”

### 课前准备
- **硬件**：教室主机（麦克风、Minecraft客户端/服务器）、Arduino按钮装置（已连接USB）、可选LED反馈灯（红/黄/绿）。
- **软件**：`mc_ai_agent.py`、`voice_bridge.py`、串口连接、阿里云ASR配置。
- **测试指令**（至少通过）：给我一个苹果、给我一把钻石剑、把时间改成白天、给我造一个小玻璃房子、我有几把宝剑（应返回无法查询）。

### 课堂流程（60分钟）
1. **回顾第8课设计**（5分钟）：复习“去掉复制粘贴、切窗口、打@ai”，板书“旧版本：人搬话 → 新版本：按钮搬话”。
2. **展示AI对讲机V1**（8分钟）：按钮是“门铃”，按下去叫醒电脑开始听，松开后理解执行。白板画：按钮 → 电脑耳朵 → AI大脑 → 代码的手 → Minecraft。
3. **第一次魔法演示**（10分钟）：老师演示“给我一个苹果”，强调没切窗口、没粘贴。提问“谁帮AI送话？”（按钮和程序）。
4. **孩子亲自使用按钮**（15分钟）：从菜单选指令（苹果、钻石剑、改白天、火把圈、小玻璃房），每次按按钮说话，观察执行，白板打勾。强化“按钮帮送话”。
5. **小揭秘**（8分钟）：按下=START开始录音，松开=STOP，电脑声音变文字，AI文字变动作。儿童版：“按下竖起耳朵，松开说去做吧”。可看控制台简单日志，不深讲。
6. **测试边界**（8分钟）：孩子说“我有几把宝剑”，预期返回“无法查询”。引导：AI有按钮耳朵，但看不到背包。保留“AI不是万能”认知。
7. **命名与版本升级**（4分钟）：孩子给助手起名，板书“AI助手V1：能听按钮说话”。预告下一步灯光反馈、说话反馈、感知位置。
8. **总结**（2分钟）：孩子补全“以前我要________，现在我只要________。”（复制粘贴/切窗口 → 按按钮说话）。

### 成功指标（达成任意三项）
- 能用按钮完成语音指令。
- 能说出“不用复制粘贴了”。
- 能说出“按钮叫醒AI”。
- 能看懂耳朵/大脑/手的分工。
- 能理解“AI听见≠什么都知道”。
- 愿意继续升级按钮助手。

### 应急方案
- 按钮失效 → 用键盘快捷键替代，说“按钮感冒了用备用按钮”。
- ASR识别失败 → 老师重复孩子话或手动输入识别文本。
- Minecraft未执行 → 记入“待学习魔法”，不长时间调试。
- 孩子说太长指令 → 引导用短句（如“给我一个苹果”）。
- 孩子说危险指令 → 回应“危险魔法要管理员确认”。

### 本课不讲的内容
串口协议、Arduino代码、阿里云ASR接口、RCON包格式、命令版本差异。

### 课程定位一句话
第9课是把第8课产品设计变成真实体验的工程体验课，孩子感受到自己设计的AI助手真的升级了。


## 第10课教案：给AI装上指点眼睛——用剑点方块，让AI知道“这里”是哪儿

### 课程定位
第9课解决了“AI能听见我说话”（通过按钮和语音输入）。第10课解决下一个自然问题：孩子说“把这里变成金块”，AI如何知道“这里”是哪儿？答案是不直接讲复杂的后台自动扫描，而是用剑点一下方块，让AI记住这个位置。这比自动扫描更适合7岁孩子，直观且聚焦于“坐标”与“感知”概念。

### 核心目标
孩子上完后能说出：
- 第9课的AI有耳朵（能听说话）；
- 第10课的AI有了第一只眼睛（我点哪里，它就知道哪里）；
- “这里”不是魔法词，必须变成坐标地址；
- AI要动手做事，需要知道目标位置。

儿童版总结：按钮让AI听见我；剑点方块让AI知道我说的是哪里；代码的手才能准确动起来。

### 不建议第10课讲的内容
Arduino代码、阿里云ASR原理、RCON协议、背包查询、末地传送、自动扫描5×5×5区域、豆包视觉。这些都可以后面讲。第10课只聚焦一个体验：我点一下 → AI记住位置 → 我说一句 → 游戏中该位置发生变化。

### 60分钟课堂流程

#### 1. 回顾第9课（5分钟）
提问：上节课我们给AI加了什么？引导回答：按钮、耳朵、按住说话松开执行。白板画“按钮耳朵 → AI大脑 → 代码的手 → Minecraft”。然后问：如果我说“把这里变成金块”，AI知道“这里”是哪儿吗？引出本课问题。

#### 2. 暴露问题：AI不懂“这里”（8分钟）
老师先不让孩子点方块，直接说“把这里变成金块”。程序最好返回“请先用剑点一下方块”。向孩子解释：AI不是故意偷懒，它只是没有看到你手指的地方。白板写：“这里”=不清楚，“坐标”=清楚地址。

#### 3. 新能力：用剑点一下（10分钟）
老师演示：孩子拿剑对准一个方块右键点击，控制台出现类似“🎯 选中方块：x=120, y=64, z=88, id=2”。解释：这就像给AI发了一张小纸条——我说的“这里”，地址是(x,y,z)。

#### 4. 第一次成功魔法：把这里变成金块（10分钟）
孩子先点一个方块，然后按按钮说“把这里变成金块”，Minecraft中被点的方块变成金块。总结：AI不是猜的，它先拿到地址，再让代码的手去改那个地址。这是本课高潮。

#### 5. 指哪儿改哪儿菜单（15分钟）
准备稳定指令（顺序推荐）：
- 把这里变成金块
- 把这里变成玻璃
- 把这里变成钻石块
- 把这里挖掉
- 在这里造一根三格高的柱子
- 以这里为中心铺一个3×3金地板

每次都重复：点一下 → 说一句 → 看变化。让孩子感受到：不是AI随便造，而是我指定位置，它准确执行。

#### 6. 小揭秘：AI怎么知道这里？（7分钟）
儿童版解释：Minecraft世界里的每个方块都有一个地址（门牌号）。用剑点一下，程序就把这个方块的门牌号记下来。然后你说“这里”，AI就去看刚才记下来的门牌号。白板写：方块=有地址的积木，坐标=方块的门牌号，点一下=告诉AI门牌号。

#### 7. 总结与下节课悬念（5分钟）
让孩子补全句子：第9课，AI能________（听我说话）；第10课，AI知道________（我点的是哪里）。预告：今天AI只能看见你点的一个方块，下一步要让AI自己看看你周围有什么（脚下是什么？旁边有没有水？附近有没有岩浆？），自然引出下一课的“自动扫描世界”。

### 课前技术准备
让系统具备能力：玩家用剑点方块 → 程序记录最后一次点击的方块坐标。建议变量命名`world_state = {"target": None, "target_block_id": None}`，控制台打印`🎯 [指点] 选中方块：x=..., y=..., z=..., block_id=...`。AI逻辑：如果用户说“这里”“这个方块”“那儿”，优先使用target坐标；若target为空，回复“请先用剑点一下方块”。注意由于运行`voice_bridge.py`会导入`mc_ai_agent.py`，感知线程不要在`main()`中启动，最好在`handle_user_text()`前确保感知线程已启动。

### 推荐课堂稳定指令（仅限这些）
- 把这里变成金块
- 把这里变成玻璃
- 把这里变成钻石块
- 把这里挖掉
- 在这里造一根三格高的柱子
- 以这里为中心铺一个金块地板

暂时不要用“把这里点燃”“在这里放水”“在这里放岩浆”“把这里传送到末地”“这里有什么怪物”“我背包里有什么”等，以免引入额外问题。

### 第10课一句话定位
第9课让AI听见你；第10课让AI知道你指的是哪里。这节课是第四阶段“给AI装上眼睛”的第一步：先做指点式眼睛，再做自动扫描眼睛和视觉AI眼睛。
