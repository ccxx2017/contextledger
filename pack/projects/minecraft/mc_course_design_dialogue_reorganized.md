# 一、课程设计与教学策略

## 主题：面向青少年的Minecraft AI教学课程大纲（六个阶段，从自然语言问答到多AI分工与记忆）

### 课程定位
以青少年对手动玩MC的热情为起点，以“让游戏世界越来越听话”为主线，将AI、编程思维、系统设计概念融入游戏体验。不追求背诵知识，而是让青少年亲眼看到“AI是如何工作的”。

### 第一阶段：AI是会说话的百科全书
- 起点：孩子玩游戏时遇到问题（如“怎么建不漏雨的房子”、“苦力怕怎么躲”）。
- 做法：引导孩子尝试用网页版大模型提问。
- 核心体验：发现AI很博学，但不知道自己位置、背包状态，没有手和眼睛。这个“局限”成为后续学习的驱动力。

### 第二阶段：代码是AI的手
- 展示老师已搭建的Python脚本：老师按几个键，学生游戏里出现房子或获得钻石剑。
- 核心概念：代码就是AI的手，没有手，AI只能说不能做。
- 技术基础：局域网连接、mcpi（建造）、rcon（给物品）。
- 目标：孩子能说清楚“老师的电脑做了什么”，不要求写代码。

### 第三阶段：说一句话，游戏就行动（理解与执行分离）
- 核心系统：几十行Python，将自然语言指令通过大模型转换为`CMD:`、`QUERY:`或Python代码前缀，再路由执行。
- 精彩教材：三个真实失败案例
  1. **斜杠问题**：AI生成带`/`的命令，RCON无法识别 → 防御性代码修复。
  2. **数量问题**：用户“去掉一把宝剑”，AI清空所有 → 提示词补充“删除必须指定数量”约束。
  3. **查询问题**：用户“我有几把宝剑”，AI反而给了一把剑 → 增加`QUERY:`分支，强制AI回复无法查询。
- 核心启发：AI擅长理解意图，但对自己能力边界没有清醒认知。我们的工作是通过规则和格式协议（字符串前缀）来弥补这个缺口，实现理解与执行的分离。

### 第四阶段：给AI装上眼睛（感知两条路径）
- **路径一：mcpi结构化扫描**
  - 可用接口：`getTilePos()`, `getRotation()`, `getBlock()`, `getBlocks()`, `pollBlockHits()`。
  - 实现方案：后台线程定时扫描5x5x5区域，统计水/岩浆/树木，检测脚下方块、头顶是否洞穴，注入Prompt。
  - 优点：精确坐标、实时、可编程；缺点：看不到UI（血量、背包）。
- **路径二：豆包视觉语义感知**
  - 手机摄像头看屏幕，理解“血量低、面前悬崖、天色变暗”。
  - 优点：语义丰富、人类友好；缺点：描述模糊，无法直接执行。
- 两条路径的互补性：mcpi精确定位，豆包发现问题。

### 第五阶段：两个AI的分工合作（系统思维）
- 完整闭环：
  - mcpi结构化扫描+豆包（视觉层）→ 描述状态 → DeepSeek（决策层，拥有技能文件+记忆）→ 输出mcpi/RCON指令 → 执行
  - 或 mcpi扫描（精确层）提供坐标数据 → 合并到决策层。
- 类比：厨房里服务员（豆包）、食材清单（mcpi）、厨师（DeepSeek）、灶台（执行层）。
- 学习成果：孩子能画出或说出系统各部分的职责。

### 第六阶段：让AI变得更懂你（记忆和技能）
- **记忆**：用JSON文件存储玩家偏好、建造历史、已知事实，每次请求注入Prompt。
- **技能**：用SKILL.md形式的文本文件存储Minecraft领域知识（物品ID、建筑模式、苦力怕距离等）。
- 核心体验：对比“无记忆AI”和“有记忆AI”在同一问题上的表现，理解上下文的价值。

### 课程主线（一句话总结）
AI能说话 → AI有了手 → AI能听懂你 → AI有了眼睛 → 多个AI学会分工 → AI记住了你

每一步都由孩子在游戏中的真实问题驱动，而非抽象概念。

## 主题：课时5“AI军师的软肋”教案（纯电脑版）核心框架压缩

### 课时基本信息
- **所属阶段**：第二阶段·第2课（共2课时，当前阶段）
- **时长**：60分钟，7岁学生
- **核心目标**：让孩子自己发现并说出：“AI很聪明，但它看不见、没有手、反应慢，不能帮我动手做。”

### 教学准备
- 文件：`MC询问豆包.txt`（上节课对话记录）
- 学生电脑：Minecraft，已进入**冒险模式**，安全区域出生点
- 老师电脑：豆包网页版登录，屏幕朝向孩子
- 白板/大白纸 + 笔，两张便利贴（“脑子”“手”）

### 教学流程（缩写）

#### 1. 热身回顾（7分钟）
- 展示上节课真实对话记录，逐条指读。
- 追问：“谁去挖木头、杀蜘蛛、合成弓箭？” → 孩子回答“我自己”。
- 白板写关键词：**“豆包：动嘴”**。

#### 2. 冒险挑战：建安全屋（约35分钟）
**任务**：冒险模式第一晚活下来。孩子可问豆包，但每次只能问一个问题，老师当“打字员”。

**三阶段暴露局限性**：
1. **选址 → 暴露“看不见”**（8分钟）
   - 老师替孩子问：“选什么地方建房子最安全？”豆包答：“找平坦开阔、靠近树木的地方。”
   - 孩子执行后，老师连环问：你脚下是什么方块？豆包知道吗？左边有什么？身后有苦力怕吗？豆包知道吗？
   - 引导孩子说出：“它看不见！” → 白板写 **“豆包：看不见🙈”**。

2. **建造 → 暴露“没有手”**（12分钟）
   - 老师替问：“最简单的生存小屋怎么搭？”豆包给三步（地基、柱子、封顶）。
   - 孩子动手建造时手忙脚乱，老师喊暂停。
   - 追问：豆包帮你砍树/合成/放过方块吗？所有事谁做的？
   - 引导出：“没有！我自己做的！” → 白板写 **“豆包：没有手🦾❌”**。

3. **防御 → 暴露“反应慢”**（8分钟）
   - 僵尸出现（或老师主动提示天黑），孩子紧急提问：“僵尸追我怎么办？”
   - 老师故意慢速打字，等豆包生成后念出答案（如“用方块堵门”）。
   - 引导感受时间差：从提问到得到答案花了多久？豆包来得及救你吗？真正救你的是谁？
   - 引导出：“它反应太慢了！” → 白板写 **“豆包：反应慢🐢”**。

**环节总结**（1分钟）：老师指着白板上三条，总结“豆包三个毛病”。

#### 3. 总结提炼：大脑和手的相遇（12分钟）
- **画图 + 贴标签**：老师画大脑和打叉的手。让孩子贴“脑子”（给AI）和“手”（给代码，回忆第一阶段）。
- **引出核心悬念**：把“脑子”和“手”并排放，中间画一条线连起来，问：“如果把它们连起来，我们对游戏说‘给我造个安全屋’，它就真的自己动起来了？”停顿3秒制造悬念。
- **手势歌收尾**：手指太阳穴（AI有脑子）→双手比机器爪（代码有手）→双手蒙眼（AI看不见）。重复两遍。结束语：“下节课当连接师，把AI的大脑和代码的手接在一起！”



## 主题：第6课“奇迹发生——说一句话，房子拔地起”教案压缩版

### 课时基本信息
- **时长**：60分钟，7岁学生
- **所属阶段**：第三阶段第1课（总第6课时）
- **核心目标**：直接回应课时5悬念，让孩子第一次体验“自然语言控制游戏”的完整闭环，**纯体验课，不讲原理**。

### 教学准备
- 老师电脑：已预测试`mc_ai_agent.py`，确保5个以上指令成功（如钻石剑、改时间、3×3金块地板、一圈火把、玻璃房子），成功指令用大字写在白板上作为“今日魔法菜单”。
- 学生电脑：Minecraft已连接。
- 上节课“大脑→手连线图”的照片或留白板。

### 教学流程（60分钟）

#### 1. 回顾悬念（5分钟）
- 拿出上节课的大脑→手连线图，复习“豆包有脑子没手，代码有手”。
- 老师宣布：“今天已经把脑子和手连起来了。你说话，游戏真的会自己动。”

#### 2. 奇迹演示：第一个魔法（10分钟）
- 老师运行脚本，孩子输入`@ai 给我一把钻石剑`。
- 观察控制台（AI思考过程）和游戏画面（钻石剑出现）。
- 第一个总结：“AI脑子听懂，代码的手做了。”
- **小揭秘（3分钟）**：用卡片画出“你 → Python传话筒 → API小窗口 → DeepSeek大脑 → 返回 → 游戏执行”，解释传话筒和API的概念，不深究技术细节。

#### 3. 魔法菜单：你选，AI做（25分钟）
- 展示白板上的“今日魔法菜单”（5个成功指令+留空“你还可以自己发明魔法”）。
- 逐个体验（每个约5分钟）：孩子选指令→输入→观察控制台与游戏→感受结果→简短对话（如“谁动的手？”）。
- 推荐顺序：RCON类（物品/时间）→简单建筑→复杂建筑（玻璃房子作为高潮）。
- 若孩子发明未测试指令，成功则加入菜单，失败则记入“待学习魔法”。

#### 4. 指令失效实验（10分钟）——认知升华
- 孩子输入`@ai 我有几把宝剑`，AI返回“无法查询背包信息”。
- 对比讨论：“为什么刚才给剑可以，现在问有几把就不行？”（AI看不见背包）。
- 点题：“我们在代码里装了拦截网，让AI懂装不懂，比不懂装懂更厉害。”

#### 5. 感受收集与命名仪式（8分钟）
- 孩子说出体验：“我说话，游戏自己动了。”
- 回到大脑→手连线图，引出核心认知：**“变成了能听懂我说话、帮我做事的AI搭档”**。
- 孩子给搭档起名，写在白板上。做手势：脑子+手合在一起。

#### 6. 悬念预告（2分钟）
- “今天AI很乖，因为我们提前帮它准备好了。下节课我们要给AI考试，看它会不会犯错，然后当医生修理它。”



## 主题：课时7语音输入方案决策（阿里云STT）与课时定位调整

### 背景
- 课时6中孩子不会打字严重影响体验。Windows内置听写测试失败（识别差）。
- 用户申请了阿里云智能语音交互（ASR，“识音石V1-端到端模型”），用于语音转文字。

### 核心决策
- **采用阿里云ASR作为语音输入方案**，而非OpenClaw语音模块（架构过重，改变交互模型）或系统听写（效果差）。
- **交互方式**：孩子按住快捷键（如F2）说话 → 录音 → 阿里云ASR转文字 → 脚本自动拼接成`@ai 指令`并填入剪贴板 → 孩子在游戏聊天框Ctrl+V粘贴并回车。
- **优点**：中文识别准确率高、低延迟、国内稳定；使用剪贴板而非模拟键盘（避免反作弊拦截，容错高）。

### 对课程大纲的影响
- **语音作为输入方式升级**：在课时7开头增加“新工具介绍”（约3分钟），演示语音输入，不深入STT原理。后续指令输入由打字改为语音（快捷键+粘贴）。
- **备份方案**：语音失败时回退到老师协助打字。

### 技术路径确认
- 语音转文字（STT）是独立小工具，不修改核心`mc_ai_agent.py`，保持稳定性。
- 识别模型：阿里云“识音石V1-端到端模型”（实时语音识别）。


## 主题：第7课“AI小助手-语音输入升级

### 课时定位
- **总第7课时**，第三阶段第2课（承接第6课“AI搭档”）
- **核心目标**：引入语音输入（F2）解决打字慢问题。
- **关键词**：语音输入

### 课前准备与语音工具
- 已测试通过：Minecraft服务器、`mc_ai_agent.py`、RCON、语音脚本`voice_input.py`（阿里云ASR，F2按键说话 → 转文字 → 剪贴板生成`@ai 指令`）。
- 白板写“今日语音魔法菜单”（如钻石剑、苹果、改白天、火把圈、小玻璃房）及备用指令。

### 课堂流程（60分钟）

## 主题：第7课备课中“传送到末地”命令失败的问题解释与处理建议

### 问题现象
- 用户指令：`@ai 将我传送到末地`
- AI生成：`CMD:execute in minecraft:the_end run tp @p 0 50 0`
- RCON反馈：`Entity 'in' cannot be found`（程序仍显示“执行成功”，因为仅表示命令已发送）
- 实际结果：传送失败。

### 技术原因
- AI生成了新版Minecraft的`execute in <维度> run ...`语法，但当前服务器/插件/版本不支持，按旧语法解析，将`in`误认为目标实体名。
- “Entity ‘in’ cannot be found”即服务器找不到名为“in”的实体。

### 教学建议（不在此节课修复）
- **给孩子解释**（儿童版）：
  > AI写了一句高级咒语，但我们的Minecraft版本听不懂，把里面的`in`当成了一个人的名字，所以说“我找不到叫in的人”。今天我们先不修这个，因为本课重点是语音和斜杠帽子。
- **列为暂不支持指令**：课堂引导使用稳定指令（给物品、改时间、简单建造），避免跨世界传送、维度切换等复杂命令。
- **可加的护栏**：在Prompt中添加规则，如“遇到跨世界传送（末地/地狱）请求时，输出`QUERY:跨世界传送暂不支持，本节课先不执行`”，避免AI生成无效命令。
- **程序判定修正**：注意RCON通信成功不等于游戏内命令成功，反馈信息需正确展示。

### 结论
本节课不解决此问题，仅作为“命令语法版本不匹配”的示例口头解释，后续课程再处理。



## 主题：第8课教案“宝剑全没了——AI忘了说几个”核心压缩

### 课时定位
- **总第8课时**，第三阶段第3课（承接第7课斜杠问题）
- **核心目标**：让孩子理解AI执行命令时必须说清“数量”（删除物品时若缺少数量参数可能导致清空全部），并引入“数量锁”规则。

### 核心概念
- AI可能听懂“去掉宝剑”但漏掉“1把”，导致生成`clear @a diamond_sword`（清除全部）而非带数量的`clear @a diamond_sword 0 1`。
- 修复方式：在提示词中明确要求删除物品时必须指定数量，并在代码层不做改变（提示词规则）。

### 课堂流程（60分钟）
1. **回顾第7课**（5分钟）：斜杠帽子与防撞护栏。
2. **故事引入**（10分钟）：小玩家的三把宝剑全没了，因为AI没说数量。
3. **玩偶剧场**（10分钟）：用三张纸片演示“拿走一把” vs “拿走全部”，孩子判断对错。
4. **真实安全实验**（15分钟）：先给多把剑（如“给我三把钻石剑”），再用语音说“去掉一把钻石剑”，验证只删除一把（系统已修好，不复现错误）。
5. **数量锁规则**（10分钟）：展示错误命令`clear @a diamond_sword`和正确命令`clear @a diamond_sword 0 1`，解释“最后的1就是数量”。告知这是AI规则书。
6. **小练习**（5分钟）：孩子判断指令是否需要数量（如“给我一个苹果”需要，“去掉所有泥土”危险）。
7. **总结与下节课悬念**（5分钟）：引出第9课“AI装作会，其实不会”。

### 课前准备
- 测试稳定指令：`@ai 给我三把钻石剑`、`@ai 去掉一把钻石剑`（仅使用已修好的系统，不真实执行危险命令）。
- 准备纸片（钻石剑道具）、白板、语音输入（可选）。

### 成功指标（达成任意三项）
- 孩子能说出“AI忘了数量”。
- 能理解“一把”和“全部”不一样。
- 能说出“删除东西要说几个”。
- 能用语音完成“给物品”和“删除一个物品”。
- 能理解规则书让AI更安全。

### 应急要点
- AI删除失败 → 用纸片演示，不现场调试。
- 语音失败 → 老师打字。
- 孩子想清空所有物品 → 说明“危险魔法只在测试世界用”。

### 核心教学语言（儿童版）
- “AI不能只说‘拿走宝剑’，必须说‘拿走1把宝剑’。”
- “删除物品时，一定要带数量，就像最后写个数字1。”

### 下节课预告
AI明明看不见背包，却假装知道 → 教AI说“我不知道”。



## 主题：第7课后反馈与第8课主题调整（舍弃斜杠/数量错误，改为产品体验与未来设计课）

### 实际问题反馈
- 第7课时间不够，斜杠问题未讲；即使讲了也不直观，孩子不感兴趣。
- 孩子核心痛点：语音输入后仍需复制粘贴、切换窗口，不能在游戏内直接说话，流程太麻烦。

### 第8课教学定位调整
- **不再讲解**：斜杠、RCON格式、clear数量锁、execute跨世界等工程细节。
- **新主题**：为什么现在这么麻烦？——让AI小助手更像真正的游戏助手。
- **核心目标**：让孩子理解当前系统由“耳朵（语音转文字）→大脑（AI决策）→手（RCON执行）”组成，但各部分未完全集成进游戏，导致需要人工搬运文字。引导孩子设计理想中的游戏内语音助手。

### 课堂流程（60分钟）概要
1. **承认问题，激发改造目标**（5分钟）：孩子指出“复制粘贴太麻烦”，明确目标“不用粘贴，直接在游戏里说话”。
2. **拆解AI小助手的身体**（10分钟）：画机器人图：耳朵、大脑、手、Minecraft，说明它们现在分散在不同窗口。
3. **画出当前麻烦流程**（10分钟）：孩子说话 → 语音窗口 → 复制 → 切到游戏 → 粘贴@ai → 执行。让孩子指出最麻烦的步骤（复制/切换）。
4. **设计未来版本**（15分钟）：列出版本1（现在，需粘贴）→版本2（按键说话自动发送）→版本3（游戏内直接语音）→版本4（AI像伙伴一样感知世界）。让孩子给版本打星。
5. **孩子当产品经理**（10分钟）：提问想去掉哪一步、AI起什么名字、危险命令是否要先确认。
6. **画“未来助手设计图”**（10分钟）：画出耳朵、大脑、手、Minecraft，并写出希望的功能（直接听、不用粘贴、帮忙建造/给物品、危险先问）。
7. **总结**（5分钟）：发现的问题不是AI不会做，而是用起来太麻烦；下一步目标让AI自己把话送进游戏。

### 课堂产出建议
- 生成“Minecraft AI助手升级清单”，如：不用复制粘贴、不用切窗口、不用打@ai、直接语音、危险命令先问、起名字。

### 后续工程伏笔
- 课后可改进：语音识别后直接调用AI决策并自动RCON执行，无需人工粘贴，第9课展示升级成果。

### 结论
第8课从“修bug”转为“产品改造课”，围绕孩子真实的体验痛点设计未来，更符合兴趣和教学效果。



## 主题：第8课后技术实现可行性分析（去除复制粘贴可行，原生语音聊天暂不承诺）

### 用户担忧
担心建议的“产品改造”（如在游戏内直接语音、无需复制粘贴）技术上无法实现，造成课堂尴尬。

### 可实现程度分层（供教学中承诺参考）
| 目标 | 是否建议承诺 | 技术可行性 |
|------|-------------|------------|
| 语音识别后不再复制粘贴，直接执行 | ✅ 可以承诺 | 高 |
| Minecraft窗口不切出，按快捷键说话自动执行 | ✅ 可尝试作为第9课升级 | 较高 |
| 在Minecraft聊天框里原生语音输入 | ⚠️ 不建议承诺 | 需Mod/插件/输入法层改造 |
| 不按键，AI一直听你说话 | ❌ 暂不建议 | 噪音、误触发、安全风险高 |
| AI像伙伴一样看见世界、记住你 | ✅ 后面阶段逐步做 | 需要感知+记忆 |

### 最稳的后续方案（第9课可承诺）
- **承诺内容**：去掉最麻烦的步骤——不用复制粘贴了。  
- **技术路径**：将语音识别结果直接传给`mc_ai_agent.py`，绕过剪贴板和手动粘贴。
- **新流程**：按快捷键说话 → 阿里云ASR转文字 → 直接交给`mc_ai_agent.py` → AI决策 → RCON/mcpi执行。  
  移除：复制、切窗口、粘贴、`@ai`、聊天框中转。

### 推荐实现方式
- 不改造`voice_input.py`独立执行命令，而是将其作为`mc_ai_agent.py`的一个输入入口。
- 共用同一个处理函数`handle_user_text(text)`，统一处理聊天文本和语音识别文本。
- 保持现有RCON/mcpi执行逻辑不变，只增加语音入口。

### 给孩子的说法（安全承诺）
> “上节课你发现AI小助手太麻烦了，要人帮它搬话。这节课我们把‘搬话’交给程序。它还不算真正住进Minecraft，但已经不用我们复制粘贴了。”

### 不要承诺的说法
- “以后你在Minecraft里随便说话，它都会听见。”（需要持续监听、唤醒词、焦点、反误触发等，复杂度高，不适合短期课堂承诺）

### 对第8课结尾的调整建议
- 第8课仍可做“产品改造课”，但结尾的下一步目标应设定为：“不用复制粘贴，按键说话后直接执行”，而不是一步到位变成游戏内原生语音助手。这样既回应痛点，又技术上可落地。

主题：外接设备实现语音控制Minecraft的可行性与架构建议（回应YouTube TTS课程）
用户核心设想
将语音识别（ASR）外接到独立设备（如树莓派），设备接收语音转文字，通过Python脚本传给Minecraft后端执行。

目标：孩子全程不离开Minecraft界面，无需切换窗口、复制粘贴，表象上语音与游戏一体。

助手核心判断
方向非常好，技术上可落地。不是把语音嵌入Minecraft客户端，而是做一个外置AI对讲机/语音遥控器。

注意区分：YouTube课程内容是TTS（文字转语音），解决AI“嘴巴”；用户需要的是ASR（语音转文字），解决AI“耳朵”。两者可结合，但当前优先ASR。

推荐架构（最稳）
外置设备（树莓派）只负责：录音、ASR、将文字通过HTTP发送给主电脑。

主电脑负责：mc_ai_agent.py、DeepSeek决策、RCON/mcpi执行、安全规则。

外置设备 = AI耳朵，主电脑 = AI大脑+手，Minecraft = 被控制世界。

最低可行版本（MVP）
外置设备按实体按钮开始录音，识别文字后发送POST请求到http://主电脑IP:8765/voice，内容如{"text": "给我一把钻石剑"}。

主电脑上mc_ai_agent.py增加接口，调用handle_user_text(text)，复用原有决策与执行逻辑。

课程应用建议
第8课可定位为“给AI小助手做一个对讲机”。

儿童解释：“以前AI耳朵在电脑窗口里，要切过去说话；现在我们给它一个小耳朵放在旁边，你不用离开Minecraft也能听见你。”

可安全承诺：下节课做AI对讲机，不用复制粘贴和切窗口。

不承诺：“AI完全住进Minecraft里了”，正确说法是“接了一个外置耳朵”。

后续可扩展：加入TTS让设备说话（如“已给你一把钻石剑”），形成完整耳朵+嘴巴+大脑+手。

结论
强烈建议此方向。它技术上可实现、教学直观、解决孩子真实痛点、不破坏现有代码架构，并能自然引出多模态AI概念。

4. **孩子亲自使用按钮**（15分钟）：从菜单选指令（苹果、钻石剑、改白天、火把圈、小玻璃房），每次按按钮说话，观察执行，白板打勾。强化“按钮帮送话”。
5. **小揭秘**（8分钟）：按下=START开始录音，松开=STOP，电脑声音变文字，AI文字变动作。儿童版：“按下竖起耳朵，松开说去做吧”。可看控制台简单日志，不深讲。
6. **测试边界**（8分钟）：孩子说“我有几把宝剑”，预期返回“无法查询”。引导：AI有按钮耳朵，但看不到背包。保留“AI不是万能”认知。
7. **命名与版本升级**（4分钟）：孩子给助手起名，板书“AI助手V1：能听按钮说话”。预告下一步灯光反馈、说话反馈、感知位置。
8. **总结**（2分钟）：孩子补全“以前我要________，现在我只要________。”（复制粘贴/切窗口 → 按按钮说话）。

### 成功指标（达成任意三项）
- 能用按钮完成语音指令。
- 能说出“不用复制粘贴了”。
- 能说出“按钮叫醒AI”。
- 能看懂耳朵/大脑/手的分工。
- 能理解“AI听见≠什么都知道”。
- 愿意继续升级按钮助手。

### 应急方案
- 按钮失效 → 用键盘快捷键替代，说“按钮感冒了用备用按钮”。
- ASR识别失败 → 老师重复孩子话或手动输入识别文本。
- Minecraft未执行 → 记入“待学习魔法”，不长时间调试。
- 孩子说太长指令 → 引导用短句（如“给我一个苹果”）。
- 孩子说危险指令 → 回应“危险魔法要管理员确认”。

### 本课不讲的内容
串口协议、Arduino代码、阿里云ASR接口、RCON包格式、命令版本差异。

### 课程定位一句话
第9课是把第8课产品设计变成真实体验的工程体验课，孩子感受到自己设计的AI助手真的升级了。


## 第10课教案：给AI装上指点眼睛——用剑点方块，让AI知道“这里”是哪儿

### 课程定位
第9课解决了“AI能听见我说话”（通过按钮和语音输入）。第10课解决下一个自然问题：孩子说“把这里变成金块”，AI如何知道“这里”是哪儿？答案是不直接讲复杂的后台自动扫描，而是用剑点一下方块，让AI记住这个位置。这比自动扫描更适合7岁孩子，直观且聚焦于“坐标”与“感知”概念。

### 核心目标
孩子上完后能说出：
- 第9课的AI有耳朵（能听说话）；
- 第10课的AI有了第一只眼睛（我点哪里，它就知道哪里）；
- “这里”不是魔法词，必须变成坐标地址；
- AI要动手做事，需要知道目标位置。

儿童版总结：按钮让AI听见我；剑点方块让AI知道我说的是哪里；代码的手才能准确动起来。

### 不建议第10课讲的内容
Arduino代码、阿里云ASR原理、RCON协议、背包查询、末地传送、自动扫描5×5×5区域、豆包视觉。这些都可以后面讲。第10课只聚焦一个体验：我点一下 → AI记住位置 → 我说一句 → 游戏中该位置发生变化。

### 60分钟课堂流程

#### 1. 回顾第9课（5分钟）
提问：上节课我们给AI加了什么？引导回答：按钮、耳朵、按住说话松开执行。白板画“按钮耳朵 → AI大脑 → 代码的手 → Minecraft”。然后问：如果我说“把这里变成金块”，AI知道“这里”是哪儿吗？引出本课问题。

#### 2. 暴露问题：AI不懂“这里”（8分钟）
老师先不让孩子点方块，直接说“把这里变成金块”。程序最好返回“请先用剑点一下方块”。向孩子解释：AI不是故意偷懒，它只是没有看到你手指的地方。白板写：“这里”=不清楚，“坐标”=清楚地址。

#### 3. 新能力：用剑点一下（10分钟）
老师演示：孩子拿剑对准一个方块右键点击，控制台出现类似“🎯 选中方块：x=120, y=64, z=88, id=2”。解释：这就像给AI发了一张小纸条——我说的“这里”，地址是(x,y,z)。

#### 4. 第一次成功魔法：把这里变成金块（10分钟）
孩子先点一个方块，然后按按钮说“把这里变成金块”，Minecraft中被点的方块变成金块。总结：AI不是猜的，它先拿到地址，再让代码的手去改那个地址。这是本课高潮。

#### 5. 指哪儿改哪儿菜单（15分钟）
准备稳定指令（顺序推荐）：
- 把这里变成金块
- 把这里变成玻璃
- 把这里变成钻石块
- 把这里挖掉
- 在这里造一根三格高的柱子
- 以这里为中心铺一个3×3金地板

每次都重复：点一下 → 说一句 → 看变化。让孩子感受到：不是AI随便造，而是我指定位置，它准确执行。

#### 6. 小揭秘：AI怎么知道这里？（7分钟）
儿童版解释：Minecraft世界里的每个方块都有一个地址（门牌号）。用剑点一下，程序就把这个方块的门牌号记下来。然后你说“这里”，AI就去看刚才记下来的门牌号。白板写：方块=有地址的积木，坐标=方块的门牌号，点一下=告诉AI门牌号。

#### 7. 总结与下节课悬念（5分钟）
让孩子补全句子：第9课，AI能________（听我说话）；第10课，AI知道________（我点的是哪里）。预告：今天AI只能看见你点的一个方块，下一步要让AI自己看看你周围有什么（脚下是什么？旁边有没有水？附近有没有岩浆？），自然引出下一课的“自动扫描世界”。

### 课前技术准备
让系统具备能力：玩家用剑点方块 → 程序记录最后一次点击的方块坐标。建议变量命名`world_state = {"target": None, "target_block_id": None}`，控制台打印`🎯 [指点] 选中方块：x=..., y=..., z=..., block_id=...`。AI逻辑：如果用户说“这里”“这个方块”“那儿”，优先使用target坐标；若target为空，回复“请先用剑点一下方块”。注意由于运行`voice_bridge.py`会导入`mc_ai_agent.py`，感知线程不要在`main()`中启动，最好在`handle_user_text()`前确保感知线程已启动。

### 推荐课堂稳定指令（仅限这些）
- 把这里变成金块
- 把这里变成玻璃
- 把这里变成钻石块
- 把这里挖掉
- 在这里造一根三格高的柱子
- 以这里为中心铺一个金块地板

暂时不要用“把这里点燃”“在这里放水”“在这里放岩浆”“把这里传送到末地”“这里有什么怪物”“我背包里有什么”等，以免引入额外问题。

### 第10课一句话定位
第9课让AI听见你；第10课让AI知道你指的是哪里。这节课是第四阶段“给AI装上眼睛”的第一步：先做指点式眼睛，再做自动扫描眼睛和视觉AI眼睛。

### 3. 视觉方案替代（豆包视觉收费后）
- 豆包视觉本质是**多模态大模型（VLM）语义理解**，能看懂游戏场景（血量、悬崖等）。用户截图中的阿里云视觉智能平台是**专用CV接口**（人脸、OCR、目标检测），无法替代。
- **正确替代**：阿里云百炼（DashScope）的**通义千问VL（Qwen-VL）**，提供多模态大模型API。建议**程序截取Minecraft窗口画面直接传给Qwen-VL**，取代“手机拍屏幕”方案，更清晰、无缝集成Python架构。成本极低（课堂演示用量可忽略）。

### 4. 第10课教案：给AI装上指点眼睛（用剑点方块）

  4. 魔法菜单：点→说“变成玻璃/钻石块/挖掉/造柱子/铺地板”等稳定指令。

  6. 总结与预告：下节课让AI自动看你周围（脚下、水、岩浆等）。

## 主题：课程顺序的核心分歧与优化建议（Claude观点）

### 总体判断
用户提出的顺序（STT→大模型→手→坐标→视觉→TTS→记忆→规则→多智能体）作为“解释AI Agent组成”的叙事主线几乎最优；但作为实际上课顺序存在关键问题。

### 与GPT-5.5的主要分歧点

1. **不要用STT开场**  
   第一课只看到“说话→出文字”不是奇迹，会失去吸引力。最大杀手锏“自然语言→游戏真的动了”应放在第一课（如打字命令造房子），语音可作为第二课的升级奇迹。原则：**第一课必须在Minecraft里看得见东西在动**。

2. **坐标就是第一只眼睛**  
   第10课的“剑点方块”本质上是一种精确但单点的感知。建议将坐标与视觉合并为一条“感知弧线”：坐标（精确一点）→ 扫描/VLM（模糊但有语义），形成“三种眼睛”对比，主线更紧。

3. **失败案例必须绑在“手”上，不能拖到第八课**  
   斜杠问题、清空背包、AI装懂等最值钱的教学素材天然发生在“手”刚出现的时刻。规则应分两层：手出现时立即用翻车现场埋钩子，后期再抽象为护栏体系。不要憋到第八课。

4. **TTS是点缀，不是核心器官**  
   保留它作为“嘴巴”对身体隐喻的完整性有价值，孩子也爱听，但不值得投入太多课时。

5. **尾部（记忆→规则→多智能体）很好**  
   形成“情感钩子→认知成熟→系统思维”的上升曲线，作为课程终章有张力。

### 最终建议
- **对外叙事**：保留用户顺序（耳朵→大脑→手→坐标→眼睛→嘴巴→记忆→规则→多智能体），用于向家长/孩子解释成品构成。
- **实际教学节奏**：将奇迹前移（第一课直接用文字命令或完整链路震撼），语音作为升级；失败案例锚定在“手”后立即出现；坐标与视觉合并为感知弧线。核心原则：骨架对外讲，教学按钩子排。

# 二、AI感知能力与硬件交互

## 主题：评估现有Minecraft AI脚本的迭代上限与主动感知方案

### 现有脚本的根本局限
当前脚本本质是**无记忆的单次问答循环**：每次对话独立，AI不知道历史、不会积累技能、无法主动观察游戏世界。能力确实较低。

### 记忆与技能系统（可行，低投入高回报）
- **记忆**：通过一个JSON文件存储玩家偏好、建造历史、已知事实，在每次请求前注入Prompt，执行后更新——代码量不超过100行，能让AI从“傻瓜”变“懂行助手”。
- **技能**：将特定知识（物品ID、建筑模式、世界管理等）写成文本文件（SKILL.md风格），启动时读入Prompt，教会AI Minecraft专属知识。

### 主动感知的现状与可行性
用户询问“第二步加主动感知”是否可行。核心限制：mcpi提供的读取接口非常有限，现有脚本完全没有“眼睛”。但可以通过以下方式部分实现（可行方案）：

- **后台线程轮询**：每隔一定时间（如2秒）使用mcpi读取：玩家位置（`mc.player.getTilePos()`）、玩家面对方向（`mc.player.getRotation()`）、头顶/脚下方块类型等。
- **注入感知上下文**：将读取到的信息格式化后加入每次调用AI的system prompt，例如：
  ```
  Current world state: Player at (x,y,z), Yaw=90°, Looking at stone. Time=day. Nearby blocks: ...
  ```
- **能力边界**：mcpi无法直接读取其他玩家血量、生物位置、背包内容。但可以间接通过RCON命令（如`data get entity`）在1.13+获取，但用户版本1.11.2不支持。所以感知范围有限但仍有实用价值（自动识别可能摔落、危险、建造位置等）。

### 最终建议与演进路线
1. **立即做**：记忆+技能文件（低投入，高效果）。
2. **可选做**：后台线程轮询获取位置/方向/周围方块，实现“感知上下文”。
3. **根本性限制**：无法绕过mcpi API的限制，主动感知无法做到像计算机视觉那样的完整环境理解。

用户明确表示将采纳记忆和技能建议，并对主动感知方案感兴趣，后续可进一步讨论具体实现。



## 主题：Minecraft AI主动感知的三个可行方案（基于mcpi与RCON）

### 现有可用的读取接口
- `mc.player.getTilePos()`：玩家坐标
- `mc.player.getRotation()` / `getPitch()`：朝向与仰角
- `mc.getBlock(x,y,z)` / `getBlocks(x1,y1,z1,x2,y2,z2)`：单个或区域方块
- `mc.events.pollBlockHits()`：玩家用剑右键点击方块的事件
- `mc.events.pollChatPosts()`：聊天事件（已在使用）

### 方案一：环境自动扫描（后台线程）
- 每隔数秒扫描玩家周围5x5x5区域，统计方块类型（水、岩浆、树木等），检测脚下方块、头顶方块（判断洞穴），将结果存入全局字典`world_state`。
- 在Prompt中注入：站立地面类型、是否在洞穴、附近有无水/岩浆/树木等。
- **价值**：完全自动，持续提供环境上下文。

### 方案二：用剑“指点”交互
- 主循环中调用`mc.events.pollBlockHits()`，获取玩家用剑右键点击的方块坐标和类型。
- 更新`world_state["player_looking_at"]`，玩家先点方块再说指令（如“把这里改成玻璃”），AI就能知道具体位置。
- **价值**：代码量最少，交互体验直观有趣。

### 方案三：RCON查询输出
- 利用`use_mcrcon.run()`执行`time query daytime`等命令，获取游戏时间、天气等状态。
- **限制**：用户Minecraft版本1.11.2下RCON可查询的状态非常少。

### 优先级建议
- **先做方案二**（用剑指点）：代码最少，立刻增强交互维度。
- **再做方案一**（环境扫描）：作为持续的背景感知层，让AI更“懂”周围世界。
- 方案三作为补充有限，暂且搁置。



## 主题：两条游戏感知路径的对比与融合建议

### 两条路径的本质差异
- **mcpi扫描（结构化感知）**：精确、实时、可编程，可获得坐标、方块ID、事件队列等客观数据。但无法理解语义（如“快满的宝箱”），且无法感知UI层信息（背包、血量、饥饿值）。
- **豆包视觉（语义感知）**：能理解人类语言描述（血量低、面前有悬崖、天色变暗），但描述模糊（如“大概10格远”），无法直接转化为精确执行坐标。

### 单独使用的致命缺陷
- 只用mcpi：AI面对一堆方块ID数组，需大量翻译工作才能理解场景，且永远感知不到背包/血量。
- 只用豆包：描述模糊，执行层无法精确定位，无法生成mcpi坐标或RCON命令。

### 建议方案：两者都要，明确分工
- **豆包负责语义层**：发现问题，输出自然语言状态描述（血量、背包、环境语义）。
- **mcpi负责精确层**：定位问题，提供确切坐标、周围方块分布。
- **DeepSeek决策层**：结合两路输入，利用技能文件和记忆，生成可执行的mcpi代码或RCON命令。

### 即时可执行的一步
为豆包设计**固定的输出格式**（如统一包含血量/饥饿/背包/面前环境四个字段），使描述结构化，便于DeepSeek可靠解析。用户表示愿意先定格式。



## 主题：用Arduino作为实体按钮实现Minecraft语音控制（技术可行性评估与实现路径）

### 核心方案调整
- 原设想：Arduino作为录音设备 → **修正为**：Arduino仅作为**实体按钮**（按下/松开），通过USB串口发送`START/STOP`信号；主机电脑负责录音、ASR、AI决策和Minecraft执行。
- 原因：普通Arduino（Uno/Nano/Mega）不擅长高质量录音和中文语音识别，但作为按钮+状态灯完全胜任。

### 整体架构（最稳）
```
[孩子] 按住Arduino按钮说话
   ↓ USB串口 (START)
[主机Python] 开始录音
   ↓ 松开按钮 (STOP)
[主机Python] 停止录音 → 阿里云ASR → 文字
   ↓ handle_user_text(text)
[mc_ai_agent.py] DeepSeek决策 → RCON/mcpi执行
   ↓
[Minecraft] 出现效果
```

### 复杂度评估
- Arduino端（读取按钮、串口打印）：**低**
- Python端（监听串口、触发录音/ASR）：**中低**
- 接入现有`mc_ai_agent.py`（抽出`handle_user_text()`）：**中低**
- 整体适合课堂项目，不复杂。

### 实现步骤（渐进式）
1. **V0（通信测试）**：Arduino发送START/STOP，Python收到后打印“开始录音/停止录音”。
2. **V1（模拟指令）**：松开按钮后，Python假装识别出固定文字（如“给我一个苹果”）并执行。
3. **V2（真ASR）**：接入阿里云语音识别，实际转文字并调用`handle_user_text()`。

### 推荐文件结构

- 每次循环开始时检查暂停标志，若暂停则`sleep(0.1)`后跳过

- 从`mc_ai_agent`导入`handle_user_text`、`pause_auto_perception`、`resume_auto_perception`

## 主题：确认mcpq事件API——推荐回调模式及@ai链路上线方案

### Claude-Opus-4.8：
纠正此前“mcpq只能注册回调”的说法——mcpq同时支持轮询和回调两种事件接收方式，但回调模式更简洁，推荐使用。

**聊天事件API**：
- `mc.events.chat.register(on_chat)`注册回调，回调接收`ChatEvent`对象
- `event.player`：玩家对象（含`.name`、`.pos`）
- `event.message`：玩家发送的聊天内容字符串
- 回调在mcpq后台线程执行，注册后主线程挂起即可，无需轮询循环

**方块点击事件API**：
- `mc.events.block_hit.register(on_block_hit)`注册回调
- `event.pos`：被点击方块的`Vec3`坐标
- `event.player`：点击者
- 其他字段（`right_hand`、`held_item`、`face`）需实跑`print(vars(event))`确认

**最小可运行版本（@ai指令响应链路）**：
- 注册聊天回调，检测消息是否以`@ai`开头
- 提取指令文本，调用现有`handle_user_text(user_text)`
- 主线程通过无限循环挂起（`while True: time.sleep(1)`）
- 上线时发送`mc.postToChat("§a[AI] 已上线")`

**两个注意事项**：
1. **回调串行阻塞**：同一事件流中上一个回调未完成时后续事件不会进入，若`handle_user_text`耗时较长会卡住指令响应，后续可考虑在回调中仅接收、将实际处理交由工作线程；
2. **锁语义变更**：原`_mcpi_lock`的保护对象与mcpq的gRPC连接不同，应用层共享状态（如自动巡检线程与事件回调并发访问）仍需锁保护，但语义已变化，待核心链路跑通后再整理。

## 环扫感知的默认关闭与雷达触发设计

环扫感知不再设计为程序启动后自动定时扫描和输出，而是默认关闭，仅在玩家主动触发时工作。触发方式有两种：一是通过语音指令进行一次性扫描（如“环扫一下”），二是放置特定方块（推荐信标）作为环扫雷达设备，敲击方块开启/关闭持续扫描，距离设备过远自动暂停，挖掉则自动关闭。开启后环扫线程只做三件事：定时扫描、更新结构化缓存 `world_state["surroundings"]`、检测危险状态是否发生变化，但不再每次扫描都打印日志或主动提醒。

危险提醒改为状态变化式提示：仅当危险从无到有或风险升级时才提醒一次，相同危险不重复输出，危险解除时可选择提示一次。感知结果保存为结构化 JSON，包含玩家位置、脚下/头顶方块、附近水/岩浆/树木、前方落差、实体列表和风险评估等字段，后续按需转换为控制台摘要、LLM 注入文本或玩家自然语言回复。

LLM 只接收压缩后的感知摘要（如玩家坐标、朝向、关键方块、实体和危险描述），不注入完整数据；当雷达关闭时，注入上下文会明确告知 AI 环扫未开启，禁止编造周围信息。

模块边界上，环扫逻辑独立为 `perception_manager.py`，负责一次性扫描、雷达开关控制、上下文生成和环境问答解析，`mc_ai_agent.py` 仅调用其 `get_prompt_context()` 获取注入文本。事件处理使用 mcpq 的回调接收方块点击等事件，但回调中只执行状态更新（如切换雷达开关），不直接调用 LLM，避免阻塞。整体设计目标是使环扫感知成为一个为 AI 服务的“传感器”，不抢占输出、不干扰语音交互，同时为后续视觉感知等扩展留出清晰接口。
## 对GPT-5.5方案的评价与按需环扫优先建议

Claude-Opus-4.8 基本认可方案的三个改进点：危险提醒改为状态变化触发而非冷却计时、LLM只接收压缩感知摘要且雷达关闭时明确禁止AI编造、回调中不调用LLM。但核心保留意见在于该方案将两套触发机制（语音一次性“环扫一下”和信标常驻雷达）同时纳入，导致刚被消除的后台循环、开关状态管理、暂停/恢复等复杂度重新引入。实际上，一次性环扫根本不需要后台线程，问了才扫、扫完即答，能一次性解决之前的刷屏、抢锁等问题；而信标常驻模式需要后台监控，且mcpq的`block_hit`事件无法检测挖掉信标，“挖掉即停”仍需额外定时轮询该坐标，反而又制造了后台循环。

## 按需环扫的意义重估：从环境问答到执行链路的坐标校验

用户质疑按需环扫的实际价值。Claude-Sonnet-5指出，若环扫仅用于回答“我周围有什么”，意义薄弱——玩家本身就能看见周围的羊、岩浆、悬崖，AI只是在复述玩家已知信息，感知并未产生真正的决策价值。真正有意义的场景是感知服务于动作执行，而非独立问答：当玩家发出“帮我把羊圈起来”“看看这里能不能盖房子”“清理周围怪物”等指令时，AI需要精确坐标才能生成建造指令或执行攻击，此时扫描将玩家看得见但说不清的位置信息转化为AI可执行的数字，或补充玩家未主动核对的安全隐患。因此环扫应定位为“动作执行前的坐标/安全校验步骤”，而非独立的环境问答功能。

如此调整后，扫描只在真正需要执行动作时触发一次，天然按需，无需额外设计雷达设备来管理触发时机；课堂叙事也更为诚实，重点从“AI能看见了”转向“AI能理解需要坐标的指令”，与课程主线“AI能力边界”形成更紧密的呼应。若保留“看看周围”问答，应明确其目的是教学可视化而非产品价值，真正决定功能意义的在于后续是否接入了需要坐标才能完成的动作。

| 旧mcpi写法 | 新mcpq写法 |
|---|---|
| `mc.events.pollChatPosts()` | `mc.events.chat.poll()` |
| `mc.events.pollBlockHits()` | `mc.events.block_hit.poll()` |
| 聊天内容 | `event.message` |
| 发送者 | `event.player` |
| 被点击方块位置 | `event.pos` |
| 被点击面的前方位置 | `event.pos_front` |
| 点击的面 | `event.face` |
| 手中物品 | `event.held_item` |

```python
from mcpq import Minecraft
mc = Minecraft("192.168.1.71")

# 首次poll启动事件捕获
mc.events.chat.poll()
mc.events.block_hit.poll()

while True:
    for event in mc.events.chat.poll(None):
        if event.message.startswith("@ai"):
            # 调用原有聊天处理逻辑
    for event in mc.events.block_hit.poll(None):
        # 调用原有方块点击处理逻辑（event.pos为方块位置，event.pos_front为面前方位置）
    time.sleep(0.05)
```

### 根本原因
- AI未被告知`mc`对象已预先连接并可用，导致其生成重复建立连接的代码。

   ```python
   if not clean_content.startswith("CMD:") and not clean_content.startswith("QUERY:"):
       lines = clean_content.split('\n')
       safe_lines = [line for line in lines if ".create" not in line and "import mcpi" not in line]

3. **同时保留其他有效约束**（建造位置、版本语法、查询拦截等）。

### 结论与废弃方案
- 仅靠Prompt约束不够，AI仍可能出错；仅靠过滤可能误伤或漏掉，必须两者结合。
- 此修复已集成到后续的`mc_ai_agent.py`中，成为标准安全措施。

                          ("world.getEntities", [-1])):

                if raw is not None:
                    break

    animals, monsters, others = [], [], []

        fields = chunk.split(",")
        if len(fields) < 3:
            continue
        name = fields[2].strip() or fields[1].strip()
        kind, cn = _classify_entity_name(name)
        if kind == "animal":
            animals.append(cn)
        elif kind == "monster":
            monsters.append(cn)
        else:
            others.append(cn)
    return {"supported": True, "animals": animals, "monsters": monsters, "others": others}
```
