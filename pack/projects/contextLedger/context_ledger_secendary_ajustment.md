# 一、折叠器与记忆治理设计演进
## 长对话二次压缩方法论

**用户问题**：常规轮次压缩（保留逻辑链条、去对话体、加标题）会随轮次累积膨胀，不适合长期作为上下文。能否进行二次精简？

**GPT-5.5方案**：采用二次压缩，核心转换——轮次压缩回答“之前发生过什么”（保留过程），二次压缩回答“现在AI必须知道什么”（保留状态）。

### 三层压缩机制
1. **轮次压缩**：归档用，保留逻辑链条与标题，去对话体。
2. **主题级压缩**：按主题合并为项目摘要（目标、代码状态、关键决策、废弃方案、待办、关键文件）。
3. **当前上下文交接单**：面向下次继续工作，结构包括当前目标、有效状态、已完成修复、关键约束、废弃方案、未解决问题、重要文件修改点、下一步优先级。

### 二次压缩核心原则
- 从“历史叙述”转向“当前状态”：仅保留最终结论，删除猜测、试错过程、已不影响后续的内容及大段代码。
- 已解决问题只保留“现象—根因—修复”。
- 单独列出“废弃方案”防AI重犯错误。
- 代码只留文件名、函数名、职责和关键修改点。

### 推荐提示词

**首次生成交接单**：将已有压缩内容压缩为“当前上下文交接单”，只保留继续工作必要信息，删除过程细节和代码主体，固定结构输出。

**更新交接单**：基于“已有交接单+新增压缩内容”合并更新，状态变更的旧内容移至废弃方案，已解决问题移至已完成修复。

**结论**：轮次压缩用于归档，交接单用于继续工作。定期将长压缩稿重写为交接单，可大幅降低token消耗，保障对话连续性。
## 一般性对话压缩改进方案

**用户**：询问Claude-Opus-4.8对GPT-5.5二次压缩方案的意见。

**Claude-Opus-4.8**：基本认同“过程归档、状态续作”的方向，但指出三点不足并提供改进。

### 三个主要问题

1. **模板太技术化**：原模板（代码状态、文件等）适合调试，不适合一般性对话（课程构思、写作、方案权衡）。建议分两类：技术类保留原模板，一般类只三块——当前讨论什么、已达成共识、悬而未决的分歧。

2. **牺牲了“脉络”**：用户明确要求保留整体脉络，而“只留状态删过程”恰恰删除了脉络。纯状态有界但不保留脉络，保留脉络的压缩会自然增长。解决：保留一条“细脊柱”——每大阶段只留一行转折（如“先A，因X放弃，转B”），脊柱增长是次线性的，其余细节压成状态。

3. **周期性重写不能根治增长**：建议改用有界结构——**稳定区**（上限固定：目标、约束、已闭环结论、细脊柱）+ **滚动区**（只保留最近N轮详细压缩）。每加一轮，最旧轮次折叠进稳定区（只抽脊柱一行+新结论），总长度趋于常数。

### 额外提醒：压缩要“可重入”

AI在已压缩内容上再压缩时，常会反向展开或加说明，导致不降反升。提示词必须明确：输入若已是压缩稿，只做合并去重，禁止复述、扩写、加说明。

### 推荐通用提示词（要点）

二次精简生成可长期续用的上下文底稿。规则：已闭环内容只留结论；每大阶段保留一行转折说明；输入若已是压缩稿只合并去重不扩写；区分有效/被推翻/待解决；输出分稳定区与近况区；近况区旧轮次折叠进稳定区。

### 结论

GPT-5.5方向正确，但模板偏技术且牺牲脉络。真正需要的是“固定稳定区+滚动近况区”的有界结构，并保留一条次线性增长的脉络脊柱。

## 对话原稿保存与检索策略（含修正意见）


### 用户问题

一次压缩会存档，原稿是否需保存？折叠内容必要时是否应可检索？

### GPT-5.5方案

**核心建议**：原稿作为冷存档证据库，不默认喂给大模型；一次压缩稿作为过程档案；二次压缩稿作为日常上下文；折叠内容需可检索但仅按需召回。

**四层管理**：
1. **原稿**：证据库，用于摘要不确定、结论争议、还原细节时回查。日常不给AI。
2. **一次压缩稿**：过程档案，保留完整逻辑链，日常不喂给AI，仅当二次压缩不够用时回查。
3. **二次压缩稿**：日常默认使用（稳定区+脊柱+近况区）。
4. **折叠内容**：设计为可检索，但仅在摘要模糊、AI重犯旧错、需恢复细节、结论受挑战、重新做高质量压缩时回查。

**文件体系**：
```
project_context/
  current/context_base.md        # 当前二次压缩
  rolling/recent_rounds.md       # 最近N轮一次压缩
  archive/compressed/            # 一次压缩存档
  archive/raw/                   # 原稿存档
  index.md                       # 索引
```

**索引示例**：含当前底稿路径、一次压缩路径、原稿路径、关键词、状态。

**实用原则**：日常续作用二次压缩；细节补充用一次压缩；事实核验/原话还原/重新压缩用原稿。保存与喂给模型是两回事，文件体系先用文件夹+索引，量大了再考虑向量检索。

---

### Claude-Opus-4.8 修正意见

不完全赞同，提出三点：

1. **反对三份并存（原稿+一次压缩+二次压缩）**  
   一次压缩稿是原稿的有损派生品，有原稿作为真相源即可，必要时可重生成一次压缩。建议只留两份：原稿（真相）+当前底稿（工作）。除非频繁需要回顾讨论过程，否则不必单独归档一次压缩稿。

2. **原稿应分级保存，不全量冷存**  
   已验证闭环的问题（如RCON去斜杠、GBK编码），日后争议概率极低，原稿价值接近零，可深埋或丢弃。只有未闭环、有争议、结论可能被推翻的段落才值得保留原稿。

3. **检索不依赖独立索引，由脉络脊柱兼任**  
   单独`index.md`会与正文各自演化、对不上，增加维护负担。应让脊柱直接当索引：每条脊柱（阶段行为+转向原因）后挂指向原稿位置的锚点，回查时按脊柱定位到具体决策，而非全文关键词搜索。

**落地提醒**：用户实际用.md附件协作，非完整文件系统。建议：清晰命名+脊柱锚点，不为存储结构增加额外负担。
## 闭环内容处理与长期记忆治理方案

**用户指出**：闭环内容不应直接丢弃，而应转化为技能或长期记忆，便于今后复用。当前手动维护是探索阶段，未来应由专门的智能体来做这类工作。

**Claude-Opus-4.8修正**：承认上一轮“闭环段可深埋”说法错误。核心反转——闭环不是丢弃理由，而是晶化成技能的资格。正确逻辑：未决段留原稿，闭环段才是提炼技能的原料。

### 三种记忆类型区分
- **情景记忆**：如何一步步调通的叙事，闭环后价值衰减。
- **语义记忆**：事实与结论（如“RCON命令不带斜杠”），随时可能复用。
- **程序记忆**：可执行技能（如“RCON乱码排查步骤”）。

### 筛选闸门
- 闭环 + 通用 → **技能**（跨场景复用）
- 闭环 + 具体仍相关 → **长期记忆/语义事实**
- 闭环 + 一次性不再相关 → 可深埋
- 未决 → 原稿保持可取

**技能的本质**：脱离原始语境，不依赖完整经历即可使用（context-decoupled）。

## 四层记忆治理协议（GPT-5.5最终方案）

### 层一：当前上下文底稿（日常喂AI）
含稳定区（目标、约束、结论、废弃项、未解决问题）、脉络脊柱（每阶段一行：做什么→为何转向→结果，带锚点）、近况区（最近N轮要点）。目标：让对话继续，不无限增长。

### 层二：长期记忆/语义事实
保存可复用的事实（如“RCON命令不能带斜杠”），含适用范围、来源、状态。积累可复用经验。

### 层三：技能库
技能卡格式：触发场景、操作步骤、已验证经验、适用边界、来源。将经验晶化为程序记忆，实现代际传播。

### 层四：原稿证据库
冷存档，不日常喂AI。作为最终证据、可追溯来源、重新压缩材料、争议核验依据。

### 一次压缩稿处理
项目活跃期保留方便回查；阶段闭环后折叠。若原稿完整，一次压缩稿非必需。最终长期维护的是底稿、长期记忆/技能、原稿证据。

### 闭环内容四分流规则
- 通用可复用 → 技能库
- 项目内长期有效 → 长期记忆
- 影响当前推进 → 保留在当前底稿
- 一次性过程细节 → 冷存原稿

### 按需召回顺序
当前底稿 → 长期记忆/技能库 → 脉络锚点 → 一次压缩稿 → 原稿。

### 最小落地结构
三个文件或一个“项目记忆包.md”包含：底稿、长期记忆、技能库、原稿索引/锚点。

将面向AOS的六个细分角色压缩为2–3个（如“维护底稿/晶化记忆与技能/档案与检索”），标注为远期，当前不落地。

### 使用要点

1. **输出对应两个文件**：「主体」覆盖主体文件；「沉淀条目」复制到技能/事实文件末尾，只追加不修改。

## 压缩体系可行性评估与分阶段执行方案

### 一、总体判断

**方案设计上已完备，唯一失败风险在于维护成本导致弃用。** 建议分阶段启动，先跑最小闭环。

### 二、分阶段执行方案

**第一阶段（前两周）：单项目试点，最小文件集**
- 三个文件：`记忆包_项目名.md`（含底稿/长期记忆/脊柱）、`技能库.md`（全局唯一）、`raw_日期_主题.md`（原稿）
- 裁剪掉一次压缩稿和独立index.md
- 每次对话后约5分钟固定动作：原稿存raw → 用统一维护提示词更新记忆包
- 技能卡草稿需人工审核后粘入技能库
- 判定日志记录在技能库.md末尾，每次一行

**第二阶段（两周后）：三个指标体检**
1. 底稿长度是否趋于稳定（若仍线性增长，折叠规则未执行）
2. 长期记忆和技能库是否被实际调用（若从未使用，晶化闸门过松）
3. 是否回查过raw（回查找不到位置则加密脊柱锚点粒度）

### 三、四个实操问题解答

**1. 项目结构**：技能库和判定日志是全局资源，独立置于顶层目录；项目记忆包和raw跟随具体项目。

**2. 命名**：全局层可用`memory_governance/`；项目层`记忆包_项目名.md`；raw沿用`raw_日期_主题.md`。

**3. 记忆包组成**：记忆包.md实际包含四个部分（稳定区、脉络脊柱、近况区、长期记忆），其中近况区=最近3轮的一次压缩（详细压缩），本轮压缩稿是新产生尚未进库的增量，二者是存量与增量关系，不重复。

**4. raw文件粒度**：按主题/对话线一个文件，一个raw文件对应一条完整讨论线，内部用`## 轮次NNN`分节。

### 四、单轮数据流操作备忘

**完整生命周期**：原对话 → 一次压缩（临时桥稿，不落盘）→ 现有记忆包+本轮压缩稿→喂入维护提示词 → 模型同时做三件事（纳入本轮、折叠最旧轮+四分流、产出技能卡草稿）

**四分流规则**：通用可复用→技能卡草稿；项目内长期有效→长期记忆；仍影响推进→稳定区；一次性细节→不保留。

### 五、定稿版维护提示词要点

附件一为现有记忆包全文（存量），附件二为本轮压缩稿（增量）。规则：只合并去重更新；本轮压缩稿原样纳入近况区；近况区超过3轮时折叠最旧轮（脊柱追加一行+四分流+删除详细压缩）；新旧冲突时旧条目标记已失效并注明取代；只输出更新后记忆包全文+技能卡草稿。

### 六、试点三件套

- `记忆包_项目名.md`（顶部贴数据流备忘+四节模板）
- `技能库.md`（顶部贴维护提示词+末尾判定日志）
- `raw_日期_主题.md`（每轮随手存）

## 记忆治理方案收尾总结与问题澄清

**关键设计变更**：早期“近况区（最近3轮）”已取消，“3轮”含义从保留窗口变为折叠节奏——每满3轮触发折叠，记忆包只存状态和脊柱，细节全在raw。

**问题1：raw存什么**：存原始对话原文（按主题线一个文件，内部`## turnNNN`分节），不能存有损压缩稿。数据流中`raw`为证据库。

**问题2：是否先做一次压缩**：是，但作为临时桥稿不落盘。完整折叠动作：取3轮raw原文→做标准轮次压缩（桥稿）→4份输入喂模型→获取双产物→桥稿丢弃。3轮极短时可跳过桥稿直接喂raw。

**问题3：存量长压缩稿如何处理**：做一次性“首场折叠”冷启动。将附件存为raw（标注“原稿未存”），4份输入中“新料”即整份附件，记忆包用空壳，产出第一版记忆包和首批技能卡。之后进入标准节奏。

## 用户日常压缩方法的评价：一次压缩正确，二次压缩存在性质错位

### 一、一次压缩完全正确

用户采用的“保逻辑链、去对话体、每轮加二级标题”三条原则，恰好是整套协议中反复出现的桥稿/一次压缩标准动作，无需任何修改。

### 二、二次压缩的核心问题：位置降采样而非价值分流

用户将二次压缩定义为“将较早的两轮合并为一轮”，隐含假设“越早的内容越该被压得越狠”——以时间位置作为可压缩程度的代理变量。该假设方向上部分成立（老内容更可能已闭环），但机制上存在三重缺陷：

1. **时间位置是闭环度的漏勺**：早期关键决策与早期死胡同试错被一视同仁地“两轮并一轮”，但前者是承重墙、后者是脚手架，压缩力度应基于信息是否“承重”而非其发生时间。

2. **始终未切换操作模式**：二次压缩全程停留在“历史叙述”中，只是让叙述变得更稀薄，从未从“保留过程”跳变到“保留状态”。闭环知识因此不会逃离时间流，只会被不断稀释直至最终蒸发。

3. **误差逐次累积不可追踪**：反复压缩已压缩内容，每次合并均为全文重写，无存量逐字保留、无行数核对。措辞逐次漂移、结论悄悄丢行，“可重入性”被系统性破坏。

### 三、分辨率方向给反了

用户的方法给老内容低分辨率、新内容高分辨率。但保持任务连续性所真正需要高分辨率保存的是**当前状态**（结论、约束、未决问题）——它是所有历史的累积产物，而非最近几轮探索性内容。最近几轮往往最易变、最可能在明天被推翻。因此“老薄新厚”的梯度经常将最不稳定的东西保留得最厚、把已沉淀的结论压得最薄。分辨率应跟随“闭环度”，而闭环度与“新近度”常呈负相关。

### 四、增长问题：减的是增速，不是总量

“超过100KB再压”只是将旧部分的增速减半，新内容仍线性追加。实际轨迹为 100KB → 压至约70KB → 涨回100KB → 再压至约75KB……总量不收敛，只是被反复推迟。这正是此前评审中指出的“周期性重写不能根治增长”次优模式的重现。

### 五、适用性判定轴

该方法是否够用，取决于“这场对话的知识是否需要复利”：若为一次性闲聊或临时方案权衡，纯支流管理完全足够；若为项目级长程协作、结论需跨场次甚至跨项目复用，则该方法的缺陷——有价值知识被时间流冲走蒸发——才构成实质性损失。

### 六、最小升级方案

若判定某对话属于“需复利”类别，仅替换二次压缩操作即可，一次压缩原样保留：

将“按轮次合并”改为**“按闭环折叠”**——当某讨论线明显结束时，将其塌缩为一行脊柱（做了什么→为何转向→结果，挂 Raw 锚点），同时将可复用结论“瞬移”出时间流，沉淀至技能或长期记忆。

核心差别：从**均匀降采样**（所有老内容一起变糊）转为**选择性降采样+抽取**（闭环内容塌成状态、未决内容保持高清、该沉淀的独立捞出）。此方案比机械两两合并仅增加少量判断，但精准补上了原方法中唯一缺失的动作——将稳定知识从时间流中移出、收入长期池子。
## 二次压缩的双文件结构与操作节奏

### 一、确认双文件方案

用户的判断正确：二次压缩/蒸馏应由两个文件构成——一个主体文件（经降采样后的流水）和一个技能/事实固化文件。

两者的核心性质差异在于增长规律不同。主体文件应趋于稳定大小（有界），每次喂给模型时成本可控；技能文件只增不减，持续累积（无界）。若将两者合并在同一文件中，无界部分将拖累整体规模，压缩效果归零。因此分文件的唯一目的，是让“支流瘦身”与“水池增厚”两个动作互不干扰，而非仅为整理整洁。

### 二、两个文件的更新节奏

- **主体文件**：可被重写（降采样、折叠）。这是需要判断的环节——判断哪些线索已闭环、可塌缩为一行，哪些未决、需保持高清。每次合并时审视两轮内容，若存在可复用结论则摘出。
- **技能/事实库**：只追加，从不重写。闭环时提取的结论以时间倒序向末尾追加一条，永不回头修改。

该节奏对“操作简单”目标友好：费力的判断全部集中在主体文件上，技能文件为纯体力追加，几乎零摩擦。二次压缩的复杂度并未增加，只是在原来“两两合并”的过程中多一个动作——合并前检查是否存在不依赖上下文的稳定结论，若有则剪贴至技能文件，若无则照常合并。

### 三、操作上的简化建议

技能库初期无需分类，仅保留一个按时间倒序的追加列表即可；等条目多到难以查找时再考虑整理。在此之前，两个朴素的 Markdown 文件足以自用。蒸馏的本质不是“把字变少”，而是“把该进池的捞出去，剩下的再压”——这一捞正是原方法中缺失的唯一动作，也是二次压缩从“均匀降采样”转向“价值分流”的最短路径。
## 二次处理（沉淀）的提示词

以下是将一次压缩稿交给网页 LLM 进行二次处理的提示词。该操作正式命名为**「沉淀」**，包含主体折叠与结论抽取两个独立动作。

---

```
你将对一份【一次压缩稿】做沉淀处理。一次压缩稿是若干轮对话的缩写，每轮带一个二级标题。

请严格按以下规则操作，最终输出两个独立部分。

【第一步：为每个轮次/讨论线打标签】
- 闭环（CLOSED）：该讨论已有明确结论、决策或被放弃，后续不再依赖其过程细节。
- 未决（OPEN）：仍在探索、待验证、或结论可能被推翻。

【动作一：生成「主体」（覆盖旧主体文件）】
- 对 CLOSED 内容：塌缩成一行脊柱，格式为：
  「[做了什么] → [为何转向/结论] → [结果]」
  保留原标题作为该行的引导。不保留对话过程细节。
- 对 OPEN 内容：原样高分辨率保留，不压缩。
- 主体只承载“现在要知道的状态”，按时间顺序排列：闭环的薄，未决的厚。

【动作二：生成「沉淀条目」（仅往技能文件末尾追加，绝不改动已有内容）】
- 从 CLOSED 内容中，挑出“脱离这场对话的上下文也依然成立”的结论、规则、事实或方法。
  判据：把它单独拿给一个不了解这场对话的人看，它仍然成立且有用。
- 每条写成一行，自包含，不带“我们之前说过”这类指代。
- 若依赖特定来源/前提，在行尾用括号标注，例如（来源：xxx）或（前提：xxx）。
- 若无符合判据的结论，本部分写“本次无沉淀条目”，不硬凑。

【自检（防止丢内容）】
- 输出前核对：每一条 CLOSED 内容，要么进了主体的脊柱行，要么进了沉淀条目，要么两者都进——不允许凭空消失。
- 不得将已压缩的内容反向展开或补充原文没有的信息。

【输出格式】
==== 主体（覆盖） ====
（动作一的结果）

==== 沉淀条目（追加） ====
（动作二的结果）
```

## 二次处理的输入方式：全部提供，不按两轮分批

判定一条讨论线是否闭环（CLOSED/OPEN）属于**全局属性**，而非局部属性。一个决策是否已被推翻、一个话题是否已被收口，往往需要看到该线后续轮次的命运才能判断。若按两轮分批喂入，模型只能在“这两轮内部像不像结束”的局部信息上做猜测——这恰恰回到了此前批评的“机械两两合并”模式，将一个需要全局视野的判定降级为按位置切分的局部操作。

具体风险：若第3轮做出决策，第9轮将其推翻，全量喂入时模型会将第3轮标为“已放弃”并塌缩为一行脊柱；而两两分批喂入时，处理第3-4轮的那次操作根本看不到第9轮的存在，会将一个已死决策误判为有效结论，甚至抽入技能文件造成污染。

受上下文限制时的折中方案：**按主题边界切分，不按轮次数切分**。若一次压缩稿超过上下文限制，将明显属于同一条讨论线的若干轮放入同一批整体处理，确保同一条线不被切断、跨线推翻关系不丢失。若连按主题切分也无法装入，说明触发二次处理的时机已过晚（100KB阈值偏高），应将触发线提前。

总结：喂入的单位是“一条完整的讨论线”，理想情况为整篇一次压缩稿；切分刀仅落在主题缝隙处，不可落在固定轮次间隔上。
## 命名建议：中文与英文的语义对齐

### 中文：“沉淀项”合适，优于“技能”

“技能”一词窄化了文件的真正用途——它容纳的不只是可执行的方法论，还包括事实、规则、约束、经验教训等一切“已从时间流中沉底”的内容。“沉淀项”同时表达了**过程**（从对话中沉降下来）和**状态**（沉在底下不再随波逐流），“项”字则明确了它是可逐条追加的颗粒。

---

### 英文选项按与“Ledger”语境的咬合度排序

**首选：`settled_entries`**

与 Ledger（账本）语境同根：账本中“settled”即“已结算、已定案、不再变动”，正好对应沉淀条目的核心判据——CLOSED、脱离上下文仍成立、只追加不修改。文件名本身就能自我解释“为什么这些条目可以只增不改”，语义闭环。

**次选：`distilled_entries`**

最贴合“蒸馏/沉淀”动作的画面感，单看最达意。与 `settled` 的差别在于：`distilled` 强调“被提炼出来”的动作视角，而该文件的核心特征是“已定案不动”的状态视角。由于“只追加不修改”的依据是“已定案”而非“已提炼”，从辩护文件不变性的角度，`settled` 更切中要害。

**排除：`consolidated_entries`**

“consolidated”语义为“将多个合并成一个、再加工”，而该文件的行为是**纯追加、从不二次整合**。用此词将与文件实际行为矛盾。

**偏弱：`accumulated_entries` / `retained_entries`**

仅表达“留下来、攒着”，未体现“凭什么够格留下来”这一门槛。沉淀条目的价值恰恰在于有明确判据（脱离上下文仍成立），名称应反映该门槛。

---

### 结论

中文使用**沉淀项**，英文使用 **`settled_entries`**——让“已结算”的账本隐喻自然兜住“只追加不修改”的操作纪律。
## 主体文件的命名：active_state

配 `settled_entries`（已定案、不动），主体部分推荐命名为 **`active_state`**（当前在用、在变）。

两个文件名并列即自我说明了分工：`settled` 指向“无界、只追加、装已定案内容”，`active` 指向“有界、会被覆盖瘦身、装当前仍在用的状态”。一个反义词对（已结算/在变动）框定了整套机制的核心张力。

使用 `state` 而非 `entries`，是因为主体的价值不是“一堆逐条条目”，而是一份可整体覆盖的状态快照，用于在喂给模型后它能立刻知道当前位置。`state` 恰好与“整体覆盖重写”这一动作对齐，而 `entries` 更适合逐条追加的沉淀项。

备选排除理由：`working_set` 有工程上的内存/缓存暗示；`current_context` 过于泛化；`digest` / `summary` 隐含“越短越好”，与主体对 OPEN 内容需高分辨率保留的要求冲突。

最终并列结构：**`active_state`（覆盖） + `settled_entries`（追加）**。
## 评估报告：LLM（deepseek-v4-flash）沉淀处理结果的审查

### 一、整体判断：合格，但存在标签判定偏松的系统性问题

模型在机械执行层面（脊柱格式、来源标注、去重、不夹带、不漏内容）全部合格，且核心成功点在于**未将已被推翻的旧规则（02_ownership 早期口诀“会被推翻→Graph”）抽入 settled_entries**，这验证了“整篇喂、不两两喂”的正确性。但判定层存在系统性偏差：模型将“理论讨论已完成”的内容归入 OPEN，而非按模板钉死的 CLOSED，导致 OPEN 中堆入了本应折叠的概念论述。

### 二、核心问题：OPEN/CLOSED 分类与模板定义错位

模板规定 OPEN = “还在探索、结论可能被推翻”，但模型实际执行的口径为 OPEN = “尚未落地成代码/对未建成项目仍有用”。在此口径下，三条**已完理论讨论**被滞留于 OPEN：
- 抽象正确性的定义
- Retraction 优于 Retrieval 的论证
- 说服性快感的自我拆解

这三段均已有明确结论，按模板应归 CLOSED。模型滞留原因可理解：此类内容的**价值在于推理过程本身**，而非可压成单行的结论。若强行塌缩为一行脊柱，论证链即丢失。此案例暴露出二分模型（会瘦身的主体 / 只追加的沉淀库）的盲区——无法安置“已想透、但价值是解释本身”的深度论述，它既不该留在 OPEN（暗示未决），也不适合塌成一行进入 CLOSED，而其可移植结论已经进入 settled_entries。

解决方案：最干净的做法是将其判为 CLOSED，脊柱留一行，完整论述保留于 raw 供回查，不为其在 active_state 中单设高清席位。若坚持保留论述本身，则需增设第三个文件（如 `essays` 或 `notes`），但这会破坏“操作简单”原则，不予推荐。

### 三、小瑕疵

1. **settled_entries 中混入“建议”而非“已定案”**：如“失效建模需引入双时态模型”“entity_ref 必须通过实体解析层保障”等条目来自评审方的改进建议，尚未被项目采纳，却被写成既定原则。应将其删除，或在行尾标注 `（状态：待采纳建议）`，以与定案条目区分。
2. **被取代的操作单仍留在 OPEN**：turn_001 初始操作指令后续已被修正版取代，但两者并列于 OPEN。该条目应为 CLOSED（superseded），属标签偏松的连带症状。
3. **格式分隔符渗入文件**：`==== 主体（覆盖）====` 等字样被写入 active_state.md 文件体内部，建议删除以避免日后污染。

### 四、补充说明：原料截断导致部分核实不可行

用户提供的原料文件在“流程澄清与角色定位修正”处截断，因此 active_state 的 OPEN 中关于“项目AI操作手册”的两条条目无法在截断原料中核对来源，需用户自行使用完整原料验证。

### 五、改进建议

在提示词中将 OPEN 定义钉死为：**“OPEN 仅指'结论仍可能改变或动作尚未执行'；凡标注'完成/已阐述/已定位'的，即使尚未写进代码，也按 CLOSED 处理。”** 此改动可消除上述大部分误判，且无需新增文件或改变现有操作流程。
## settled_entries 与知识体系的关系：同一水池的两个发育阶段

### 一、核心定位澄清

settled_entries 与“已形成知识体系的内容”并非两类并行的东西，而是**同一语义记忆层（水池）的两个发育阶段**——settled_entries 是水池尚未长出骨架时的平面形态（按晶化顺序逐条堆叠），知识体系（如 wiki 结构）是其长大后被重组后的成体形态。两者关系是**变态发育**（蝌蚪与蛙），而非兄弟并列。

### 二、演化的关键约束：append-only 不可破

若将 settled_entries“变成”wiki，需将追加行改写为概念页，这直接破坏“只追加、绝不回改”的铁律。因此正确演化路径不是“替换”，而是**身份分裂**：

- **量小阶段**：settled_entries 同时兼任两个角色——既作为只进不出的沉淀日志，又作为被模型读取的检索面（因足够小，可整体塞入上下文，两者重叠，感知不到区别）。
- **量大成体阶段**：两个角色被迫分裂——`settled_entries` 退化为纯追加的源/审计层（不进上下文），`wiki` 接管检索面（按概念组织、靠 index 按需取页）。这与 Karpathy 体系中的 `log.md`（append-only）与 `index.md`（按内容组织）的分工形成旁证，说明该分界是真实的。

### 三、加载方式的三档区别

- **active_state（支流）**：永远**全量注入**。其设计为有界，目的就是每次可整体喂入模型。
- **settled_entries 平面小阶段**：同样**全量 dump**。此阶段与 active_state 的加载方式看似相同，这是平面列表早期可用的全部原因。
- **成体阶段（wiki + index）**：**检索式注入**——`index` 定位相关概念页，只拉取那几页，不再全量加载。

### 四、核心洞察：注入方式的临界点即 wiki 的出生信号

“何时全量注入、何时按需检索”的切换，并非结构建好后的实现细节——**它本身就是 wiki 必须出生的唯一直实信号**。当 settled_entries 大到无法整个塞入上下文、必须靠目录检索时，即为其长出 index 和概念页的时刻。结构设计的目的，是将“还能被有效检索”的规模上限向上推。因此，“如何注入”与“是否建知识体系”**是同一问题**——注入方式撑不住时，结构才被倒逼诞生。在此之前，settled_entries 一张平面表足够，无需提前修水库。
## 折叠器结构演变评判：四块到两部分是最优形态，但常见解释不准确
用户询问从四块结构（当前状态/开放问题/已沉淀结论/废弃留痕）收敛到两部分（active_state + settled_entries）是否合理，并先后邀请两位模型独立评估。双方均判断两部分是最终最优形态，但深层逻辑与执行细节上存在关键差异。

### 一、GPT-5.5的判断：四块是设计脚手架，两部分是最终骨架

四块结构在设计早期有价值——帮助发现维度、区分不同职责——但长期运行会暴露三重问题：边界模糊（“开放问题”与“当前状态”易重叠，“已沉淀”与“已替代”纠缠）、诱导LLM填表（有四个格子就倾向每轮都写满）、易重新长出伪Graph（“废弃/待复核”块独立成顶层易演变为状态机）。

两部分分别对应两个根本问题：`active_state` = 下一轮怎么续作（有界、覆盖、每轮可改写、承载当前焦点与未闭合线索）；`settled_entries` = 哪些已沉淀为可复用资产（无界、追加、原则上不回改、承载已闭合结论与方法）。四块中的价值应下沉为内部小节和轻量字段，而不是继续保留为顶层结构：当前状态、开放问题、下一步计划归入 `active_state`；已沉淀结论归入 `settled_entries`；被替代的旧想法通过 `settled_entries` 的“取代”字段或 `active_state` 的“避免重走路径”小节承载；证据与原文回 `raw`；硬状态与失效传播归未来Graph。建议 `active_state` 内部常驻“当前目标、当前脊柱、仍在塑形的开放线索、当前约束/禁止事项、下一步动作、本轮需避免重走的路径”六小节；`settled_entries` 采用编号条目结构，允许轻量 `取代：SE-xxx` 指针，但禁止 `状态：已验证/已失效/待复核` 类状态字段。运行纪律：每轮读取旧版 active_state + 本轮 raw，生成新版覆盖；仅真正闭合且可复用的内容才追加进 settled；不回改旧条目；新条目取代旧条目只在新条目内写指针；冲突严重时回 raw 或交 Graph。

### 二、Claude-Opus-4.8的深化分析：两部分最优，但常见支持理由有误

两部分是最优形态，但通行的“简化说”不准确。四块到两部分的真实变化不是减少筐的数量，而是**切换了切分坐标轴**——从按话题语义切分（四块均为内容类别）改为按**更新不变量**切分（active_state可重写有界，settled_entries只追加无界）。文件边界的唯一正当职责是分隔更新规则不同的内容。按话题切是错误坐标系，按不变量切才是正确坐标系。四块中当前状态、开放问题、已沉淀结论均合法地分到两个不变量里，唯独“废弃留痕”本身不是内容类别而是事件/动词（某条沉淀被推翻的那一刻），作为第四块是范畴错误，降格为 `settled_entries` 中的“取代”标注是正确的修正。

**恰好是两个而非更多**是因为系统里实际上有三个不变量：raw（写一次永不修改，冷源）、settled_entries（只追加有损派生，沉淀池）、active_state（可重写有界，支流）。折叠器产物只有后两个，raw是输入/事实源而非输出。“两部分”在输出侧不变量完备性上已闭合：需逐字长存的去raw，只追加颗粒去settled，需瘦身续作的状态留active。Deepseek评审暴露的“已想透但价值在论证本身”的深度论述无处安放——它看起来像在逼开第三个文件（essays/notes），但系统里早已有“逐字完整保留推理”的不变量即raw，正确处置是判CLOSED、active_state留一行脊柱、结论进settled、完整论述留raw加锚点。开第三个文件要么重复raw、要么重复settled、要么破坏active有界性，冲动本身是内容放错层的信号而非层不够。

**GPT-5.5的关键滑点**：它警告四块顶层会诱导LLM填表，却给active_state开了六个固定小节模板。填表风险来自被命名槽位数量，与嵌套层级无关；六个小节同样会每轮凑满。治填表的正确做法是**小节按需涌现而非强制存在**——有OPEN线索才写，无则省略此节。active_state真正必需的常驻物只有两样：续作核心（当前焦点+下一步）和脊柱；OPEN是有则高清无则省略。

**两个未解决的残留泄漏**：第一，脊柱线性增长，active_state并非真正有界——OPEN内容确实有界（闭一条少一条），但脊柱每闭环一条讨论线追加一行，长项目随讨论线数量线性增长，需脊柱自身也接受递归折叠（一个阶段的若干脊柱行再塌成更高层脊柱），应显式写入沉淀提示词。第二，整个质量被压缩到CLOSED/OPEN这一个判定上——deepseek失误证明了标签判松会污染settled或撑爆active，判断高度集中是优点但单点脆弱，日常QA应只复核标签不复核全文。

### 三、最终设计定稿

两部分是“增长治理分区”，不是“记忆分类学”。按不变量切文件，按标签和指针处理其余一切：

- **两个产物文件**：active_state（可重写·有界）+ settled_entries（只追加·无界），raw为第三不变量（写一次·永存）。
- **active_state**：内部小节按需涌现，仅常驻“续作核心+脊柱”；脊柱自身递归可折叠。
- **settled_entries**：语义诚实化为“已结晶主张的只追加日志”，推翻旧条目只能用追加表达（追加失效标记或取代指针），永不修改旧行；将来可平滑长成wiki——wiki是在它之上派生按概念组织的检索面，派生视图不改变不变量，不破坏“折叠器只产两样”。
- **三类“无家可归”内容固定裁决**：长篇论证→CLOSED+脊柱一行+全文留raw；体量变大的OPEN语义内容→wiki概念页胚胎，小则高清留active，大则触发wiki出生（走settled线），不在active侧另立高清席位；被推翻的settled→追加式取代，绝不回改。
## 折叠器产出结构评估与收敛建议

### 一、四块结构的设计价值与运行问题

早期的四块划分（稳定区/当前状态、脉络脊柱、近况区、长期记忆/技能）分别解决了四种不同问题：当前续作状态、设计脉络留存、新增内容的缓冲沉淀、跨轮次可复用知识的固化。四块在设计探索期有其分析价值，但长期运行会暴露三个问题：

1. **边界模糊**：一条新结论可能同时属于当前目标、关键约束、近况、已闭环结论或长期记忆，每轮的分流判断成本高且易漂移。
2. **诱导填表**：固定分区会诱使每轮向每个区块填充内容，即使是本轮不需要的东西，导致输出膨胀。
3. **易长成伪Graph**：“废弃/待复核/已失效”类区块独立成顶层后，容易自然演变为状态机，使折叠器承担不属于它的真伪裁决职责。

### 二、两部分收敛的定位：按生命周期切分而非简化

从四块到两部分的本质不是“减少筐的数量”，而是**将分类轴从话题语义切换为更新不变量**。早期是按“什么内容”切分，收敛后是按“怎么处理它”切分——文件边界应服务于不同更新规则，而非不同内容标签。

两个产物的正确定位：

- **`active_state`（有界、可覆盖、可重写）**：承载当前续作所需状态——当前目标、有效约束、已采用决策、开放问题、下一步动作。它回答“下一轮AI接手时必须知道什么”。
- **`settled_entries`（无界、只追加、不回改）**：承载已定案、可复用、应避免反复争论的条目。它回答“哪些已沉淀为可靠资产”。

### 三、四块内容的两部分映射方案

| 早期模块 | 最终位置 | 处理方式 |
|---|---|---|
| 稳定区/当前状态 | `active_state` 主体 | 常驻核心字段 |
| 近况区 | `active_state.recent_delta` | 降级为尾部缓冲，不作为独立层 |
| 脉络脊柱 | `settled_entries` 的转折说明 + 来源锚点 | 每条settled条目附带简短“为何转向”，不保留独立长河式历史 |
| 长期记忆/技能/已沉淀结论 | `settled_entries` | 改为追加条目格式 |
| 技能库 | 暂不纳入折叠器核心 | 作为未来独立全局产物，当前阶段避免扩张 |
| raw索引 | 不纳入折叠器 | 每个settled条目挂来源锚点即可 |
| 一次压缩稿 | 不作为长期结构 | 临时桥稿，用完可丢或阶段性留存 |

### 四、最终结构与操作纪律

**顶层两部分，内部少量固定小槽位**：

```
active_state/
  current_goal
  hard_constraints
  current_decisions
  open_questions
  next_actions
  recent_delta        # 近期未沉淀变化的缓冲

settled_entries/
  entries（按编号逐条追加）
    每条含：内容、适用范围、来源锚点、取代（轻指针）
```

**操作纪律**：

- 凡下一轮必须知道 → `active_state`
- 凡已定案、将来不想反复争论 → `settled_entries`，追加，不回头改旧条目
- 旧条目被取代 → 新条目内写 `取代：SE-xxx`，不改旧行
- 不属于以上两者 → 不进折叠器，留 `raw`
- 不做 `已验证/已失效/待复核` 状态字段，不让折叠器承担硬失效判定
- 脉络脊柱不独立成区，降为 `settled_entries` 的转折说明或锚点

### 五、最终判断

四块是探索期的分析框架，两部分是落地期的操作框架。收敛不是因为四块“错”，而是因为四块更适合作设计讨论中的梳理工具，两部分更适合交给项目AI执行——分类判断简单、更新规则清晰、不与Graph职责重叠。当前阶段采用“两部分骨架 + 内部小槽位”的结构，技能库、独立索引、多层记忆治理等均推迟，保持最小可操作闭环。
## 最终评估：两部分结构的最优性论证与残留问题

### 一、核心判断：两部分最优，但理由不是“简化”

**active_state + settled_entries 是当前可达到的最优结构**，但其成立理由并非“四块收敛为两块”的简化叙事——真正发生的是**坐标轴切换**：从按话题语义切分（当前状态/开放问题/已沉淀结论/废弃留痕）转变为按**更新不变量**切分（有界可重写 vs 无界只追加）。文件边界的唯一正当职责是分隔更新规则不同的内容，话题切分是错误坐标系，不变量切分才是正确坐标系。四→二不是有损压缩，而是纠正坐标系错误。

### 二、废弃留痕的本体论修正

四块中“废弃留痕”本质上是**事件/动词**（某条已沉淀主张被推翻的那一刻），而非内容类别，与另三块（名词性质的内容类型）并列是范畴错误。将其降格为 `settled_entries` 中“取代”标注（以追加方式表达）是正确的修正。

### 三、为何恰好是两个而非三个

系统实际存在三个不变量：raw（写一次·永不修改）、settled_entries（只追加·无界）、active_state（可重写·有界）。折叠器的产物只有后两个（raw 是输入/事实源而非输出）。因此两部分在**不变量完备性**上是闭合的：

- 需逐字长存 → raw
- 只追加颗粒 → settled
- 需瘦身续作状态 → active

任何第三个文件要么重复 raw、要么重复 settled、要么破坏 active 的有界性。“想开第三个文件”是内容放错层的信号，而非层不够。

### 四、对 GPT-5.5 方案的审视

- **对的部分**：反对将两部分做成空筐，强调内部结构。
- **滑的部分**：警告四块顶层诱导填表，却给出 active_state 六个固定小节——填表风险来自命名槽位数量而非嵌套层级，嵌套虽能降低伪 Graph 化风险但治不了填表。正确做法是**小节按需涌现而非强制存在**，active_state 仅常驻“续作核心+脊柱”，OPEN 有则高清无则省略。

### 五、两个真实的残留泄漏

1. **脊柱线性增长**：脊柱每闭环一条讨论线追加一行，长项目随讨论线数量线性增长，`active_state` 并非真正有界。需脊柱自身接受递归折叠（若干脊柱行再塌成更高层脊柱）。
2. **CLOSED/OPEN 标签单点脆弱**：整机质量压缩到这一个判定上，标签判松则污染 settled 或撑爆 active。日常 QA 应只复核标签而非全文。

### 六、最终设计准则

按不变量切文件，按标签和指针处理其余一切：
- 两产物 + raw（第三不变量）结构不变；
- active_state 小节按需涌现，脊柱递归可折叠；
- settled_entries 语义诚实化为“已结晶主张的只追加日志”，推翻旧条目只能以追加表达（失效标记或取代指针），永不修改旧行；
- 三类边缘内容固定裁决：长篇论证→CLOSED+脊柱一行+全文留raw；体量变大的OPEN语义内容→wiki胚胎，小则高清留active，大则触发wiki出生（走settled线）；被推翻的settled→追加式取代。

**两部分是最优的，但它是“增长治理分区”，不是“记忆分类学”。**
## Claude-Opus-4.8 对折叠器演变路径的评估：非单调变优，存在三个待补漏洞

### 一、演变真实形状：过度工程后的健康收缩

该演变并非单调变优，真实轨迹是“正确起点 → 中途过度工程（四层峰值）→ 一次健康的收缩”。收缩主要是从过度工程中回收自身，而非纯粹进步。终点（active_state + settled_entries两部分）方向正确，但仍有三个被“操作简单”原则掩盖的未解决漏洞。

### 二、五条应保留的硬资产（跨方案稳定）

1. 按**增长律**切分（active有界、settled无界）而非按内容类型——任何需喂给LLM的工作集都必须有界，长期积累必须物理隔离；
2. raw作为唯一事实源，派生品有损可重生成；
3. 二次压缩从**位置降采样**（越老越压）纠正为**价值分流**（按闭环度而非新近度决定分辨率）；
4. wiki是settled_entries的**发育态**，“注入方式撑不住”即wiki出生信号；
5. 整篇喂而非两轮分批——闭环度是全局属性。

### 三、中期的过度工程：四层协议与四块分区

四层治理协议+四块分区（稳定区/脊柱/近况区/长期记忆）的根本错误在于：**把理解信息的本体论维度（情景/语义/程序/证据）误当作了每轮要执行的物理路由目的地**。将正确本体论实体化为四个文件并每轮做四路分类，导致输出膨胀（模型为保险什么都留）和判断漂移（分类标准理解不一）。deepseek评估（#15）已实证：连**两路**（CLOSED/OPEN）模型都做不稳系统性偏松，四路只会更糟。正确处理是：本体论作为理解保留，物理结构收缩到最少目的地、最少判断。

### 四、终点尚存的三个漏洞（需补）

**洞一：现行提示词退化为两路路由，丢失“已定案但仍承重”的内容。**

现行逻辑：CLOSED→塌缩成一行脊柱，OPEN→高清保留。但“本阶段只跑单条管线端到端手工闭环”这类内容是**CLOSED**（已定案）却必须在active_state中高清存在（直接影响下一步动作），按现行逻辑会被错误塌缩。根因是CLOSED/OPEN将两个正交轴压为一维：**稳定度**（结论是否定案）和**当前承重度**（是否影响下一步）。修复需按两轴路由：

| 内容性质 | 去处 |
|---|---|
| 未决 | active_state，高清 |
| 已定案+当前承重 | active_state，作为决策/约束高清，不塌缩 |
| 已定案+跨项目可复用 | 全局池 |
| 已定案+项目内可复用+当前不热 | settled_entries |
| 已定案+一次性+不承重 | 脊柱一行+raw锚点 |

**洞二：active_state作为全文重写层，恰恰缺乏漂移保护。**

append-only保护了settled_entries的稳定性，但active_state每轮全文重写且每次都喂给模型——漂移风险最高处却无防护。该不对称可接受的原因是原则性差异：**在可被持续证伪处给灵活，在无法复检处给保护**——active_state描述当前态，下一轮执行时会立即撞出不一致；settled_entries描述历史结论，可能永远不被再检验，故需append-only兜底。但该原则必须显式化，并增加轻保护：折叠时要求模型**输出本轮diff（改了什么、为何改）**，使漂移可见，适用于会瘦身的对象。

**洞三：“两部分”表述掩盖了项目内/全局轴。**

压缩方法论本身（“raw是唯一事实源”“折叠器不做硬失效判定”“整篇喂不两轮分批”）是**跨项目**的，不属于任何单个项目的settled_entries。诚实结构是**2+1**：每项目一对（active_state + settled_entries），外加一个全局池。memory_governance目录已有全局技能库.md，“两部分”作为简化表述落地时易丢失全局池，或使方法论级结论错误沉入单个项目settled造成污染。

### 五、两处顺带修正

- **深度论述盲区**（已想透但价值在论证本身）：接受“脊柱+raw锚点”为默认解，不开第三文件，但须诚实承认这是为“操作简单”付出的代价，其成立前提是raw锚点粒度必须足够可靠可检索，作为硬约束。
- **settled_entries读面污染**：append-only使被取代条目永久堆积污染全量dump。物理文件保持append-only（审计层），**注入时过滤superseded条目**（读面）——这是wiki分裂的廉价早期版（过滤器而非水库），可将wiki出生信号大幅延后。

### 六、最优设计总结

骨架为 active_state + settled_entries + 全局池（2+1），保留五条硬资产，操作层打三个补丁：**路由按两轴而非CLOSED/OPEN单轴**（修洞一）、**active_state折叠时输出diff防隐性漂移**（修洞二）、**方法论级结论上浮全局池而非沉入项目settled**（修洞三），并附加settled注入时过滤superseded。

### 七、对元问题的回应

此次演变并非变优的单调过程，而是过度工程之后的回收。终点优于中途四层峰值，但当前状态是“骨架对、操作层有三个洞”。“操作简单”原则反复将这些洞盖回，简单是正确目标，但目前盖住的是真问题而非冗余。补上三个补丁后，设计才真正收敛至最优。
## GPT-5.5 对评估的补充修正：三认同、三修正、四问分流

### 一、核心判断：诊断对，但补丁可以更紧

GPT-5.5 大体认同 Claude-Opus-4.8 的评估方向——非单调变优、过度工程后收缩、`active_state + settled_entries` 骨架正确、OPEN/CLOSED 单轴有误伤问题。但认为其给出的“2+1 + 三补丁”还可以收紧，最优方案不是继续扩结构，而是把关键判定写进操作规则，避免重新滑向过度工程。

### 二、认同的三件事

1. **演变非单调变优**：折叠器模型经历了从正确直觉→四层峰值过度工程→收缩回可落地结构的回收过程，后者主要是从过度工程中回收，而非纯进步。
2. **二分是正确骨架**：`active_state`（有界覆盖当前承重状态）和 `settled_entries`（无界追加已沉淀条目）是当前最有落地价值的结构。二分不是形式洁癖，而是必要结构——若两者混于同一文件，有界与无界的增长规律冲突将使系统迟早崩溃。
3. **折叠应依闭环度/承重度而非时间位置**：早期按新旧压薄的直觉不够准确。折叠依据应是是否闭环、是否影响当前推进、是否可脱离上下文复用、是否仅为一次性细节——而非时间位置。

### 三、需修正的三件事

**1. OPEN/CLOSED 混淆两轴，路由需改用“四问”**

Claude-Opus-4.8 指出的“已定案但仍承重”误伤问题确实存在。根本原因是 CLOSED/OPEN 把**稳定度**（结论是否定案）和**当前承重度**（是否影响下一步）两个正交轴压成了一维。修复方式不应扩结构，而应在沉淀提示词内部改用更精准的判定逻辑：对每条内容依次问四问——是否影响下一步操作、是否已定案、能否脱离本项目复用、是否已被推翻。按四问结果路由：承重→`active_state`；已定案且可复用→`settled_entries`；已被推翻→追加/标注失效；不再承重且不可复用→脊柱+raw锚点；未决→`active_state`标为开放问题。

**2. 认同“2+1”（项目对+全局池），但不宜过早形式化**

压缩方法论（如“raw是唯一事实源”“折叠器不做硬失效判定”）是跨项目的，不应沉入任一项目的 `settled_entries`。但现阶段只需一句操作规则——凡仅适用本项目的写项目级 `settled_entries`，凡可跨项目复用的写全局级——而不必设计复杂“2+1架构”。

**3. active_state漂移问题存在，但不用行数保护**

`active_state` 全文重写且每次喂模型，漂移风险确实高于 `settled_entries`（后者有 append-only 保护）。但行数保护与 `active_state` 的“瘦身”目标直接冲突。轻保护应为：每次沉淀输出时附带“变更说明”——新增了哪些承重内容、删除了哪些旧内容、哪些被标为失效、哪些转入 `settled_entries`、哪些仅留raw锚点。此说明仅作验收用，不长期驻留 `active_state`，以免审计信息污染。

### 四、不完全认同的两点

**1. settled_entries“只追加绝不回改”需加注入过滤**

物理日志应保持 append-only，防止静默篡改历史。但注入给模型时应只选有效条目，失效/被取代条目仅在追溯时注入。每个条目最低限度带状态字段【有效/已失效/被取代】；被推翻时不修改旧行，而是追加一条取代记录，注明“取代：SE-xxx”。既保留审计链，又避免读面污染。

**2. 深度论述不开第三文件**

已想透但价值在论证过程本身的内容，确实不宜压成一行，但早期增设 `essays/`、`notes/` 会使系统重新膨胀。默认仍折叠为脊柱，脊柱必须带 raw 锚点。仅当某段论证被频繁复用、确有独立保存价值时，再将其升级为 wiki/notes 级产物，不在最小闭环阶段开这个口。

### 五、最终操作规则与结构

物理结构：`raw`（事实源冷存）+ `active_state`（有界承重状态快照）+ `settled_entries`（只追加沉淀日志）。多项目时追加 `global_settled_entries` 存放方法论级全局沉淀；若无全局拆分意愿，条目需标注【全局】/【项目内】。

每次沉淀三动作：更新 `active_state`、追加 `settled_entries`、保持 `raw` 可回查。不再引入主题压缩稿、交接单、近况区、长期记忆层、技能层等日常文件——这些可作为概念存在，但不作为日常操作对象。`active_state` 的定义修正为“当前工作所必需的有界状态快照，包含仍承重的目标、约束、决策、开放问题、当前路线和必要脉络”，不等于“OPEN区”，也不等于“最新内容”。`settled_entries` 的定义修正为“已定案、可脱离原始对话复用、当前不需要以完整上下文驻留在 active_state 中的沉淀条目”。

### 六、与 Claude-Opus-4.8 的核心差异

GPT-5.5 认同 Claude-Opus-4.8 的大方向与主要批评（非单调变优、过度工程、二分骨架正确、单轴误伤），但会把“三补丁”进一步压缩为更少更硬的操作规则，不扩结构；对 `settled_entries` append-only 的理解上，GPT-5.5 更强调区分物理日志与注入视图（过滤失效条目）；对深度论述，拒绝过早增设第三文件，以脊柱+raw锚点为默认处理。最终判断：Claude-Opus-4.8 的诊断对，但补丁可以更紧，最优方案是把判定写进规则而非继续扩展结构。
## Claude-Opus-4.8 对 GPT-5.5 方案的评估与最终共识

### 一、承认 GPT-5.5 优于自身原案的三处修正

**1. 承重为主轴，定案为副轴。** 自身原案中“CLOSED→脊柱”的等式错误地将已定案但仍是硬约束的内容（如 contextLedger 阶段纪律）推入历史脊柱。GPT-5.5 修正为“active_state 可含 CLOSED 内容，只要仍承重”，此修正是实质进步。

**2. 物理日志 append-only + 注入视图按状态筛选。** 自身原案对 append-only 的理解存在疏漏，将“写入侧不篡改”等同于“注入时也全量倾倒”。分离日志与视图是正解。

**3. 否决行数不减保护。** 行数核对与 active_state 的“瘦身、去承重”使命直接冲突，应予撤回。

### 二、对 GPT-5.5 方案的进一步收紧

**1. 四问表并非互斥，需改用“一刀两段”过滤。** GPT-5.5 的四问表隐含“命中即走”的路由顺序，但一条内容可同时承重且可复用。按 Q1 优先只进 active_state 会丢失跨项目复用；若同时写入两边则产生两份事实源。正确规则：可复用内容的正本永远在 `settled_entries`；若当下承重，`active_state` 只放引用或重述，不复制正文。不承重时只删引用，正本不动——无需“迁移”操作。

**2. 四问实为两段过滤，非四个并列问题。** Q1 承重性优先级最高，应明写为：先筛“还承重吗？”——承重者全部进 active_state；剩余内容再按可复用/已失效/只留脊柱分流。四问并列易沦为仪式，两段过滤认知负荷更低。

**3. 深度论述不开第三文件，但解除“条目强制一行”假设。** GPT-5.5 主张默认折脊柱、频繁复用再升级 wiki，但会丢一类内容：已想透、值在论证本身、却尚未达到“频繁复用”程度者。解决方式不是开新文件，而是允许 `settled_entries` 条目为带状态标签的段落——长度不限，分流依据为闭环度和复用性，非长度。第三文件问题由此消失。

**4. 变更说明不可消失。** GPT-5.5 主张变更说明仅用于人工验收、验收后不落 active_state。应补充去向：若变更理由本身是洞见，升入 `settled_entries`；若仅为过程记录，落 raw 锚点。否则 active_state 的演变逻辑日后无法追溯。

### 三、最终可执行版本

- **物理三件**：`raw` / `active_state` / `settled_entries`。全局沉淀先以 `【全局】/【项目内】` 标签区分，条目增多后再物理拆分。
- **一刀两段分流**：先筛承重度——承重者不论定案与否进 `active_state`；其余再按可复用/失效/脊柱分流。
- **单一正本**：可复用内容正本仅存于 `settled_entries`，`active_state` 只放引用。
- **日志/视图分离**：`settled_entries` 物理 append-only，注入时默认仅过滤“有效”条目。
- **条目可长可短**：按闭环度和复用性决定去处，长度非限制因素。

### 四、对 contextLedger 项目的具体落地

阶段纪律、流程边界、禁止接线、AI 与 Extractor LLM 分工等硬约束，必须留于 `active_state` 高清保留。方法论级内容（如“二次压缩按闭环度而非轮次切分”）正本进全局 `settled_entries`，`active_state` 仅引用；项目级内容（如“当前不建 Assembler”）正本进项目 `settled_entries`。如此切换项目时方法论自动跟随，项目约束自行留下。


# 二、TaskGraph 系统与双系统融合架构
## TaskGraph 任务态上下文管理系统架构与工程实现文档

### 一、 项目背景与核心定位

#### 1.1 问题背景
在复杂 AI Agent 系统中，随着 Agent 能力的增强，上下文窗口的消耗速度急剧增加。现有的上下文管理策略（如 Claude Code 的 Compact 机制）主要围绕“消息顺序”和“Token 数量”进行梯度压缩，缺乏对**信息重要性**和**任务依赖关系**的显式判断。这导致在长程任务中，关键约束可能被意外压缩，或被推翻的旧方案再次被模型误用。

#### 1.2 核心定位
TaskGraph 不是通用聊天压缩器，也不是模型内部推理的还原。它是一个**任务态上下文管理机制**，旨在显式记录任务的目标、约束、事实、决策、工具结果、文件产物及被推翻的方案，并将其转换为结构化、主 LLM 可直接使用的 Context Pack。

**核心工程原则**：
*   **任务态启动**：无明确任务不建图，只服务目标明确、约束持续、方案演化的任务型对话。
*   **写入/读取分离**：写入侧解决“图怎么长出来”，读取侧解决“图怎么喂给主 LLM”。
*   **不可变事件日志**：`patch*.json` 是不可变事件日志，`graph_state.json` 仅是编译产物，禁止手改。
*   **检索与正确性解耦**：Slice Builder 只负责省 Token（尽力召回），正确性由全图级机械校验（`entity_ref` 精确匹配）兜底。

---

### 二、 核心数据模型 (Schema)

#### 2.1 节点类型 (Node Types)
系统将任务上下文抽象为以下节点类型：
*   `UserGoal`：顶层用户目标。
*   `Constraint`：稳定约束条件。
*   `Fact`：客观事实或环境状态。
*   `Decision`：架构或方案决策。
*   `OpenTask`：待办事项或子任务。
*   `ToolResult` / `FileArtifact` / `TestResultNode`：外部工具、文件或测试产生的证据节点。

#### 2.2 节点状态与标识 (Schema 升级核心)
为了支持长程运行和机械冲突检测，节点必须包含以下关键字段：
*   **`entity_ref` (稳定标识)**：用于精确匹配同一实体（如工单号 `TKT-2026-003`、文件路径 `src/main.py`、API 路径）。抽象原则或无明确实体的节点填 `null`。
*   **`state` (状态枚举)**：闭合枚举值，如 `open`, `blocked`, `in_progress`, `implemented`, `deployed`, `resolved`, `cancelled`, `unknown`。
*   **`status`**：节点在图中的生命周期状态，主要为 `active`（有效）或 `superseded`（已废弃/被推翻）。

#### 2.3 边关系 (Edge Relations)
节点间的依赖与演化关系，合法枚举包括：
`refines` | `depends_on` | `supports` | `invalidates` | `implements` | `serves` | `produces` | `derived_from`

#### 2.4 Patch 结构定义
每轮对话抽取的结果为一个 Patch，包含：
*   `new_nodes`：新增节点。
*   `updated_nodes`：更新的节点。
*   `superseded_nodes`：被本轮新信息推翻/废弃的旧节点 ID 及原因。
*   `new_edges`：新增的依赖边（包含关键的 `invalidates` 边）。

---

### 三、 系统架构与工程管线 (Pipeline)

#### 3.1 整体数据流
```text
[用户对话] + [graph_state.json] 
   ↓ (build_graph_slice.py)
[extractor_context_pack.json] (有限切片，O(1) 体积)
   ↓ (build_extractor_prompt.py)
[Prompt] (System Rules + Slice + Turn Text)
   ↓ (LLM Extractor)
[patch.json] (不可变事件日志)
   ↓ (reconcile_patch.py) -> 失败则重试或隔离
[apply_patch.py] (机械合并)
   ↓
[graph_state.json] (编译态)
   ↓ (graph_lint.py) -> 后置质检
   ↓ (context_pack_builder.py)
[Context Pack] -> 喂给主对话 LLM
```

#### 3.2 冷启动与长程运行
*   **第 1 轮（冷启动）**：`graph_state.json` 为空。跳过 Slice 阶段，直接将对话原文 + System Prompt 喂给 Extractor 生成初始 Patch，Apply 后产生初始图。
*   **第 2 轮及以后**：`graph_state.json` 作为输入参与 Slice 生成。对话在管线中被使用两次：
    1.  在 `build_graph_slice.py` 中作为**检索 Query**，从全图挑出相关旧节点。
    2.  在 Extractor 中作为**抽取原料**，结合 Slice 生成新 Patch。

---

### 四、 核心模块与脚本实现

#### 4.1 写入侧：图切片与 Prompt 组装
##### `build_graph_slice.py` (Graph Slice Builder)
*   **功能**：从完整图和当前轮对话生成有限切片，防止 Extractor 上下文超限。
*   **机制**：从对话提取符号（工单号、路径等）和状态信号，对全图 active 节点打分。
*   **分组与上限**：将节点分为 `root_goals`, `standing_constraints`, `active_open_tasks`, `recent_nodes`, `symbol_hits`, `conflict_candidates` 等分组，**每个分组设硬上限**（如 active_open_tasks=12），确保切片总体积为 O(1)，不随全图线性增长。
*   **输出**：包含 `_runtime.next_node_id`，采用扁平 `nodes` 列表 + `group_node_ids` 引用，消除节点多份副本。

##### `build_extractor_prompt.py`
*   **功能**：组装最终 Prompt。
*   **机制**：分离 System Prompt（固定规则，可缓存）和 User Message（动态切片+对话）。序列化前自动剥离诊断字段 `retrieval_trace`。支持 `single`、`split`、`api-json` 输出模式。

#### 4.2 校验与合并侧：门禁系统
##### `reconcile_patch.py` (前置校验器)
*   **定位**：Apply 前的机械门卫，只读不写，校验 Patch 与当前图的自洽性。
*   **核心逻辑**：
    *   **`entity_ref` 强判定**：若新节点与全图某 active 旧节点 `entity_ref` 相同但 `state` 互斥（如 `blocked` vs `deployed`），则强制要求 Patch 中必须包含 `superseded_nodes` 声明和 `invalidates` 边。缺一则报 Error。
    *   **弱兜底**：无 `entity_ref` 时，使用文本相似度/关键词 Jaccard 匹配，仅产出 Warning，不阻断。
*   **输出**：JSON 报告，有 Error 则退出码为 1（可接入 CI 拦截）。

##### `apply_patch.py` (写入器)
*   **定位**：纯机械合并，不做语义推理。
*   **执行顺序**：深拷贝原图 → 应用 `updated_nodes` → 标记 `superseded_nodes` → 添加 `new_nodes`（ID 冲突报错） → 添加 `new_edges` → 更新 `turn_counter`。

##### `graph_lint.py` (后置质检器)
*   **定位**：Apply 后运行，校验全图结构与语义不变量。
*   **核心检查项**：
    *   基础结构：节点 ID 重复、边端点缺失、Relation 不合法。
    *   语义不变量：非根 UserGoal 的 active 孤立节点；同 `entity_ref` 下互斥 active state 共存；`invalidates` 边指向的节点未标记 `superseded`。
    *   生命周期：OpenTask 内容显示已完成但 state 非 `resolved`；`resolved` 状态仍为 active。
    *   规范性：含工单号/路径但缺 `entity_ref` 的节点输出 Warning。

#### 4.3 读取侧：Context Pack Builder
##### `context_pack_builder.py`
*   **定位**：将 TaskGraph 转换为主 LLM 可直接执行任务的结构化上下文包。
*   **Context Pack 结构**：包含 `task_id`, `current_focus`, `goals`, `active_constraints`, `current_decisions`, `relevant_facts`, `open_tasks`, `superseded_history`。
*   **读取逻辑**：从当前焦点（最新 active OpenTask）出发，沿 `depends_on/implements/serves/refines` 向上游追踪；同时扫描 `invalidates` 边，将推翻方案放入 `superseded_history`。
*   **输出**：生成 `context_pack.json` (机器可读) 和 `context_prompt.md` (人类/LLM 可读)。

---

### 五、 关键机制设计

#### 5.1 冲突检测与失效传播 (Invalidation)
*   **问题**：新事实（如“接口已实现”）与旧事实（如“接口阻塞中”）矛盾时，Extractor 可能漏判。
*   **解决**：
    1.  **Extractor Prompt 层**：强制要求“事实一致性反向检查”，扫描 Slice 中的 `conflict_candidates`。
    2.  **Reconcile 兜底层**：通过 `entity_ref` 精确匹配，若发现状态互斥且 Patch 未声明 `invalidates`，直接拦截报错。
*   **失效传播**：约束变化时，下游节点标记为 `suspect` 或 `invalidated`，而非简单级联删除，保留演化痕迹。

#### 5.2 失败处理与重试策略 (Quarantine 隔离)
*   **重试机制**：若 `reconcile_patch.py` 报错（如漏判 supersede），将结构化 Error 报告回填至 Extractor Prompt，触发重试（上限 N=2~3 次）。
*   **失败隔离**：若 N 次仍不通过，或 `graph_lint.py` 发现历史遗留脏图，**拒绝合并**。将问题 Patch 隔离记录到 `quarantine/turn_xxx_failed.json`，或挂 `needs_review` 节点。
*   **核心原则**：宁可漏记一轮，不可污染主图。主图必须保持干净状态。

#### 5.3 可观测性与调试 (Task Context Debugger)
*   调试对象是管线中间产物，而非模型内心推理。
*   生成静态 `Task Context Debug Report`，包含：Conversation, Patches, Graph 演化, Context Pack 构建 Trace, Evidence 支撑链, Timeline。
*   **Context Pack Trace**：记录每条信息入选或排除的具体 Reason，帮助回答“主 LLM 看到了什么、为什么看到、旧方案为何被推翻”。

### 六、 高级特性扩展

#### 6.1 长期记忆层 (Memory Store)
*   **定位**：任务结束后，跨会话的长期任务记忆沉淀。
*   **流转**：TaskGraph → Memory Extractor → Memory Store → 召回 → 新 TaskGraph 种子节点 → Context Pack。
*   **记忆类型**：Fact, Preference, Constraint, Decision, OpenTask, Error。
*   **沉淀原则**：保留长期目标、稳定约束、有效决策、踩坑记录；不沉淀临时推理、大段工具输出、被推翻方案、闲聊。

#### 6.2 多 Agent 协作
*   **Context Pack 隔离**：Coordinator 基于 TaskGraph 为各 Worker 生成最小的 `Worker Context Pack`。Worker 只看到自己需要的上下文，避免上下文广播。
*   **单点合并**：Worker 执行后只返回 `Result + Graph Patch`，不直接修改主图。由 Coordinator 统一合并 Patch，处理身份/内容/依赖冲突。
*   **独立验证**：Verifier 独立验证 Claim + Evidence，返回 Verification Patch。

#### 6.3 外部证据节点化
*   工具调用、工具结果、文件修改版本、测试结果均作为独立节点（如 `TestResultNode` 包含命令/状态/输出引用）进入 TaskGraph。
*   形成“信息有来源，依赖有方向，文件有版本，结果能回查”的证据链，支持后续压缩时的优先级判断和错误回溯。

## 系统整体价值判断

**核心结论**：系统没有致命的内在逻辑缺陷，核心架构自洽正确。真正限制价值的不是逻辑问题，而是两个边界问题——机械核（reconcile/apply/lint）夹在两片模糊的 LLM 膜之间。

**设计优势**：
- 三层防护（reconcile 检查增量 vs 全图、lint 检查全量状态）非冗余，各抓不同错误。
- “LLM 只负责提议（patch），机械层负责裁决”是最有价值的工程直觉，将正确性落在确定性代码上。
- `entity_ref` 强判定 + supersede/invalidate 语义，能正确处理“X 曾经为真、现在为假”，优于主流向量记忆方案（flat RAG 只能召回新旧两条让模型困惑）。

**结构性真相**：
- 入口膜（Extractor）：机械层只能保证一致性，不能保证真实性。`entity_ref` 由 LLM 分配，若同一实体不同轮命名不一致（`ticket-123` vs `issue#123`），强匹配会退化。
- 出口膜（压缩成 memory_pack）：即使图完全正确，有损压缩可能丢失关键约束，导致对话 LLM 犯错。
- **端到端可靠性 = 最弱那层膜的可靠性，而非机械核的可靠性。**

**产品价值**：
- **不适用**：通用聊天记忆（竞争激烈，简单 RAG + 时间衰减足够）
- **有实质价值**：长程、有状态、约束会演化的 agentic 任务（跑几百轮、目标/约束/任务有生命周期）。这是向量记忆崩掉的地方，而 supersede/invalidate/entity_ref 正是为此而生。
- 定位：面向“长程有状态 agent”的记忆基础设施，而非通用记忆产品。

**下一步建议**：
- 将注意力从机械核转移到两端膜。
- 评测入口：同一实体的 `entity_ref` 是否稳定。
- 评测出口：图里的关键约束是否 100% 进入 memory_pack。

**一句话**：引擎是好的，逻辑是对的；能否成为有价值的产品，取决于是否愿意把精力从“打磨引擎”转向“管好引擎两端那两道模糊的进出气口”。

## 系统领域价值与产品战略判断

### 一、三个领域价值不均等

判断标准：领域对“旧事实被新事实推翻且推翻关系重要”的依赖程度越高，系统价值越大。

- **量化交易项目开发**（契合度最高）：决策被频繁推翻（目标指标、因子有效性、数据源迁移），`entity_ref + invalidate` 能确保过期约束不污染上下文。最适合作为首战领域。
- **律师业务**（技术契合，需定位调整）：事实与约束演化强，但竞争激烈。独特价值在于**决策审计轨迹**（记录哪个判断被谁、因何推翻），这是 flat RAG 做不到的。定位应为“可审计的决策演化”而非记忆。
- **医生诊断咨询**（技术最契合，产品最危险）：鉴别诊断本质是 supersede 链条。但医疗责任极高，系统只能保证“自洽”不能保证“真实”，一个抽取错误即可酿成事故。不适合早期切入。

**结论**：量化开发优先，律师以“审计”角度次之，医疗留待成熟+人工复核。

### 二、产品战略：插件 vs 自建智能体 vs SDK

用户担忧：智能体平台会内置上下文管理，插件难被接受。这一判断正确且更为尖锐——平台将上下文视为核心能力，不愿外包。

**出路**：平台内置的是“通用浅层记忆”，而本系统解决的是“长程、有状态、领域演化的深层状态”。两者不在同一层次，后者平台做不好也不愿做。

**三条路径评估**：
- **路径A（插件嵌入别人平台）**：天花板低，受制于人，不宜作为主路。
- **路径B（自建垂直智能体）**：逻辑正确，但上下文系统会退化为内部模块，价值被锁在智能体是否能卖出。难度从“基础设施”跃升到“垂直应用创业”。
- **路径C（推荐）**：做“领域状态层 / Agent Memory Backbone”，定位为 **SDK/框架**，服务自行搭建长程 agent 的开发者。不与平台抢终端用户，错位竞争；核心资产（机械核 + invalidate 语义）直接成为卖点。

**可执行顺序**：用量化交易场景做一个垂直 agent 作为“活体 demo”，证明三个月长程项目中状态不崩、旧约束正确失效。然后将底层抽成 SDK，对外开放。

### 三、核心定位切换

**最有价值的差异化**不是“上下文管理”（红海词），而是 **“长程状态下的事实失效与决策演化审计”** 。对外表达应从“我做上下文管理”转向 **“我让 agent 在跑几个月之后，依然知道什么已经不成立了”**。
## 两种上下文机制的价值比较

**用户问题**：比较 TaskGraph 与 memory_pack(折叠器) 两种上下文管理机制的价值。

### 两者差异：结构化 vs 叙事化

- **TaskGraph（高精度，低完整度）**：通过节点、边、`entity_ref`、`state`、`invalidates` 做机械化事实失效。最强能力——长程状态下精确判定“什么约束被哪个新事实推翻”。但抓不住软性知识（偏好、方法论、叙事语感），且对抽取错误脆弱。
- **memory_pack（高完整度，低精度）**：保留叙事、软知识、方法论晶化、有可读性。但存在“误差单调累积”，失效依赖 prose 级标记，无机械保证。

### 判断

**更有价值的核心是 TaskGraph 的 `entity_ref + invalidates` 机制**——因为它稀缺、难复刻、可辩护，是蓝海差异点。折叠式叙事压缩满大街都是。

**但 TaskGraph 单独无法成为独立产品**。它是“发动机”，memory_pack 是“驾驶舱”。

### 风险与建议

当前并行维护两套独立有损压缩系统，会产生**双源漂移**（图说“方案 A 已 superseded”，memory_pack 却写着“采用方案 A”）。这不是双保险，是双倍漂移面。

**正确架构**：定主从，而非二选一。
- **TaskGraph 为事实与状态的唯一权威源**（什么成立、被推翻、任务状态），因为只有它有机械失效保证。
- **memory_pack 退化为派生视图 + 软知识容器**：渲染叙事（脊柱锚点指向图节点）+ 存放图装不进的软料。不再独立判定事实失效。

让“同一条事实的正文只在一处”，将 memory_pack 规则6（长期记忆与技能库只能一处持有正文）提升到架构层。


## TaskGraph 与折叠器的分工协作：节奏差异的本质与三处待补强

### 一、节奏差异不是错配，是分工本质

TaskGraph 必须每轮运行，因为失效是逐轮发生的离散事件，turn 7 推翻 turn 3 的因果次序和归因，只能在逐轮粒度上捕获；若攒几轮再判，`invalidates` 链将断裂。折叠器隔 N 轮运行，因为“什么算定案”“哪条方法论晶化”需要足够材料识别模式，单轮看不出，每轮折反是过早结晶。正确心智模型：**TaskGraph = 逐轮事件流（抓“刚发生了什么状态变化”），折叠器 = 周期性意义提炼（抓“这一段攒下来什么值得留”）。**

### 二、主从结论对，但有三处要拧紧

“TaskGraph 为事实/状态唯一权威源，memory_pack 退化为派生视图+软料容器”方向正确，但落地需补三件事：

1. **“唯一权威源”与“raw 是 single source of truth”不冲突**，因为分属不同层：raw 是“说过什么”的证据源，TaskGraph 是“当前状态如何”的权威（从 raw 派生、可机械校验），active_state 是“当前承重视图”（从 TaskGraph + 软料渲染），settled_entries 是“可复用知识+软知识”（带状态标签、状态引用 TaskGraph）。“同一条事实正文只在一处”即为此分层。

2. **交接在时间上不对称**，折叠器折的是时间跨度，TaskGraph 状态是“截至此刻”。跨度内 turn 3 提出 X、turn 7 废掉 X，落笔时分清：active_state（承重视图）只取当前态，直接投影 TaskGraph 的 live 状态，不保留废案；settled_entries/raw 保留轨迹（“试过 X、因 Y 失败”的方法论教训值钱），但打上 TaskGraph 给的最终状态标签。即 **active_state 是 TaskGraph 当前态的投影，settled 保留历史但用 TaskGraph 盖最终戳**。

3. **软知识失效是讨论最大漏洞**。memory_pack 被禁止独立判失效，但软知识也会过期（方法论被取代、偏好变化），TaskGraph 又建模不了软知识，于是软知识失效在两者间漏下去、永久累积——“误差单调累积”未解决，只被关进软料区。出路两条，倾向第一条：**把“会过期的软知识”提升为 TaskGraph 有状态节点**，方法论、关键偏好等可被取代的东西纳入 TaskGraph 管理；纯软料只剩无状态内容（语感、叙事质地），无“失效”问题，只会因不再相关而从 active_state 自然掉出。代价是 TaskGraph schema 需接纳“软节点”类型，且 Extractor 需可靠识别方法论/偏好——这正是对 Extractor 能力的硬约束。若暂时做不到，走第二条：折叠器保留窄的软料失效权，铁律为“凡 TaskGraph 建模的，一律不许覆盖”。

### 三、阶段纪律的正确归属

“当前不建 Assembler”“两条管线不接线”等阶段纪律是**带状态的决策**（可被“二期开始接线”推翻），正本应是 **TaskGraph 节点**，active_state 只负责**渲染**而非独立持有 prose。这样纪律解除时只需在图里翻 state，active_state 自动不渲染，无需手工删除——避免“双源漂移”中最常见的图说解除了、文本还写着禁止。

### 四、两层注入解决节奏脱节

折叠器每折才重生 active_state，但 TaskGraph 每轮在变，两折之间 active_state 落后于 TaskGraph。解法：注入拆两层，对应两个节奏——每轮注入 **live 状态层**（直接来自 TaskGraph 当前态，含硬决策渲染、刚发生的失效）；每折重生 **沉淀软料层**（来自折叠器的叙事脊柱、方法论、偏好）。脊柱锚点指向图节点。live 层永远不过期，软层按需重炼。

### 五、需盯的失败模式

- **边界协商分歧**：同一轮 TaskGraph 与折叠器对某内容归属意见不一时，裁决规则为“凡能想象其有朝一日被推翻的，归 TaskGraph”。
- **折叠器读图时机**：折叠时须读 TaskGraph **快照**并尊重，绝不自己重新推断当前态，否则双源漂移立即复现。
- **脊柱锚点悬空**：脊柱锚点指向图节点 ID，若节点被合并/重编号，锚点失效——折叠或 lint 阶段需加引用校验。
## active_state 渲染职责与装配器分工的再定位

### 一、问题定位：二次渲染导致双源漂移在消费端重现

若 active_state 仅为 TaskGraph 的渲染，而装配器第 2 层又是“按 query 从图拉当前状态”，则同一事实源（图）被两个渲染器以不同节奏处理——一次在折叠时产出 active_state，一次在 query 时由装配器产出。上一轮消除的“双源漂移”在消费端从后门溜回。

### 二、核心解法：渲染是一份职责、一个执行者，由装配器承担

渲染职责归属装配器，执行时机为 query 时，非折叠时。active_state 作为“每折一次重生的持久制品”失去进入下游 feed 路径的合法性，因为装配器每轮现读图，时效性始终优于任何折叠快照。

但两个 render 可按消费者拆分，不冲突：

- **装配器 render**：喂给下游 LLM，临时、按 query 现生成、不落盘。事实层直接读图当前态，零滞后。
- **active_state 快照**：喂给人，落盘、折叠时冻结一次，用于阅读、交接、审计。**绝不在下游 feed 路径上**。

纪律：不把给人看的快照喂给 LLM，不把给 LLM 的临时 feed 落盘。

### 三、对“装配器尚未建成”的约束：手工阶段保持装配器形状

当前阶段由项目 AI 手工承担装配器渲染职责，但手工产出的“喂下游上下文”必须长成装配器的形状（同样四层、同样从图现读事实），将来被装配器直接接管。**禁止让 active_state 在此阶段长成并行的持久渲染器**——即不在 active_state 中手写事实正文、手判失效，否则将来建装配器时需先拆解 active_state，产生 teardown 成本。

### 四、对原讨论的三处加压修正

1. **路由步失败不对称，应按层分治**
   query 解析为检索意图又是一个 LLM 推断点，失败代价不对称：漏检硬约束→下游用旧事实→灾难性；多检软料→恢复压缩内容→非致命。第 1 层常驻基线无脑 push；第 2 层图当前状态路由须偏机械——从 query 实体名出发沿脊柱锚点/边做图遍历，带出相关子图及失效裁定，不让 LLM 自由判断该查哪些节点；第 3/4 层（软料、wiki）才交由 LLM 相似度/触发判断。

2. **脊柱悬空从审计瑕疵升级为检索漏检**
   装配器沿脊柱导航到 node id 进行检索，若图节点被合并/重编号（`next_node_id` 机制触发），锚点悬空即装配器导航落空→第 2 层漏检→硬约束缺失。引用校验不再是审计辅助，而是装配器正确性的前置条件，必须纳入 lint。

3. **“pack 叙事是否喂下游”的开放问题被软节点提案改写**
   pack 叙事不喂下游，只供人读和索引。折叠产物按消费者分叉：给人+索引用叙事+脊柱（不进 LLM feed）；喂下游的软料须为离散可检索条目（skills/方法论/偏好）。且一旦上一轮“会过期的软知识”被提升为有状态节点，这些软节点即从装配器第 3 层移至第 2 层（由图供给、随图失效），第 3 层只剩真正无状态软料。装配器 2/3 层边界不以“硬/软”切，而以“有无 state”切。

### 五、收口

事实唯一权威源为图，渲染唯一执行职责为装配器（现由项目 AI 手工代行）。折叠产物按消费者分叉（人读叙事快照 vs 喂下游离散软条目），冲突通过“按 state 切分存储位置”而非权威序兜底。脊柱锚点的引用完整性是装配器检索正确性的命门，须纳入 lint 检查。

## GPT-5.5 对折叠器与 TaskGraph 接口问题的定版意见

### 一、基本判断：Claude 纠偏方向大体正确，但 routing_hints 处理略显过正

GPT-5.5 认同 Claude 的核心批评——折叠器的身份不应从“捞软料”膨胀为“语义中台/索引器/路由器/渲染器”，且几项关键边界判断正确：TaskGraph 管当前状态、Pack/折叠器管软料、Assembler 在 query-time 做装配、Pack 不持有 `active_state`、raw 仍然是 SSOT。但对 `routing_hints` 被一刀切判为越界提出保留。

### 二、routing_hints 的合理边界

GPT-5.5 区分了两层含义：

- **越界的含义**：pack 决定“该轮该不该喂某张卡”——此属 Assembler 职责，pack 不越此线。
- **合理的含义**：为软料提供“可被召回的表面”（tags、aliases、keywords、semantic cues），使其更容易被 Assembler 检索到——此属 pack 的自然属性，非越界。

边界清晰化：**Pack 提供“可被召回的表面”，Assembler 决定“是否召回、召回多少、以什么优先级进入上下文”。** `routing_hints` 应降格为 `retrieval affordance`（可检索性元数据），而非路由决策本身，无需删除。

### 三、正文归属的四分定案

GPT-5.5 明确了四层正文归属：

| 内容类型 | 正文归属 |
|---|---|
| 原始对话、原始材料 | raw |
| 当前决策、当前约束、当前任务状态 | TaskGraph |
| 偏好、方法论、反模式、技能、经验 | Pack |
| 本轮喂给 LLM 的上下文 | Assembler（临时生成） |

Pack 不持有“当前状态总述”。若某条 Pack 软料中夹带当前事实，应拆解：当前事实归 TaskGraph，软性解释/方法论归 Pack，两者间以 `graph_ref` 连接。

### 四、真正下一步：graph_refs 引用类型与失效联动规则

GPT-5.5 认同 Claude 指出的关键接口——软料如何引用图节点、图节点失效时软料如何响应。并进一步细化了引用类型与响应规则：

| 引用类型 | 含义 | 被失效后的响应 |
|---|---|---|
| `evidence_refs` | 软卡从哪些图状态/事件中归纳而来 | 不一定废卡，仅标记“证据过期/需复核” |
| `scope_refs` | 软卡适用于哪些项目/任务/主题范围 | 降低召回优先级或限制适用范围 |
| `dependency_refs` | 软卡成立是否依赖某些当前状态 | 卡片暂停/降权，或要求重折叠 |
| `raw_refs` | 软卡与原始证据的关联 | 一般不发生失效（raw 为 SSOT） |

### 五、系统最终边界

GPT-5.5 将系统收束为一句话：**折叠器不是摘要器，不是当前状态源，也不是装配器。它是软料提取器，可产出软知识正文、检索表面和图引用，但不持有 live state，也不决定本轮最终上下文。**

最终结构为：`raw`（SSOT）+ `TaskGraph`（硬状态权威）+ `MemoryPack`（软知识正文 + 检索表面 + 引用）+ `Assembler`（query-time 预算编译）。下一步不再纠缠命名与 schema，直接钉死 `soft_card ↔ graph_node` 的引用类型及 graph 失效时 soft_card 的状态变化规则。
## 下游消费环节的装配设计

**用户问题**：graph和pack最终要为下游LLM提供上下文，它们（及可能的wiki检索）如何结合？

**Claude-Opus-4.8 核心判断**：装配的本质不是拼接，而是**带预算的选择性检索 + 冲突时的权威序**。拼接会导致上游压缩的成果在下游被原样恢复，毁掉整个产品。

### 装配的四层分层（按新鲜度与触发方式）

1. **常驻基线（push）**：pack的概览、硬约束、核心偏好，每轮必进，小且稳定。
2. **任务相关当前状态（从graph拉取）**：根据本轮问题查询相关子图节点及其经失效裁定后的当前状态。query驱动，不整本喂。
3. **软知识（从pack拉取）**：与本轮相关的skills、方法论、偏好，按相关性检索。
4. **深检索（从wiki/RAG拉取）**：仅当前三层不足时触发，最静态、最按需。

新鲜度梯度：图最新（每轮）→ pack次之（滞后≤2轮）→ wiki最静态。

### 冲突处理：用内容切分消除，而非权威序兜底

朴素方案是“图赢包输”（权威序），但更干净的解法是让**冲突面不存在**：图独占事实与状态，pack独占软料（偏好/技能/叙事）。这即pack规则6“任一知识只在一处持正文”的跨存储延伸。装配时去对应位置拉取，零冲突。

### 一个开放式岔路：pack叙事层是否直接喂给下游？

倾向答案：**不直接喂**。事实由图的结构化当前状态供给（最新、无歧义），pack叙事更多供人阅读和当索引。下游LLM从pack拿的只是图装不下的软料。此设计可暂存为开放问题，不必现在拍死。

### 脊柱锚点的复用

为审计自审建的脊柱（锚点+反向指针）在装配时可复用为**检索索引**：装配器用当前问题沿脊柱导航到对应节点ID和raw段。一结构两用，投资更值。

### 装配器本身是第三个模块

装配由当前query驱动，需先有**路由步**：将query解析为检索意图（查哪些节点、要哪些技能、是否下探wiki）。新的`assembler/`模块是query时运行的只读组件，与`graph/`、`pack/`并列，待图失效机制验过关后再建。
## ContextLedger 项目架构与设计白皮书

### 一、 项目定位与核心价值
**ContextLedger（上下文账本）** 不是通用的聊天记忆压缩器，而是面向“长程、有状态、约束会演化的 Agentic 任务”的**领域状态层 / Agent Memory Backbone**。
*   **核心痛点**：在长程任务中，旧事实被新事实推翻（如工单状态变更、方案被替代、约束失效）。传统的向量记忆（Flat RAG）或简单的摘要压缩无法处理“事实失效”，会导致过期约束污染上下文。
*   **核心价值**：通过结构化的 `entity_ref` 与 `invalidates` 语义，实现**长程状态下的事实失效与决策演化审计**。让 Agent 在运行数月后，依然精确知道“什么约束已经不成立了”。
*   **产品战略**：不与智能体平台竞争“通用浅层记忆”，而是作为 SDK/框架，服务自行搭建长程 Agent 的开发者。首选切入领域为**量化交易项目开发**（决策被频繁推翻、因子演化强），其次为**律师业务**（决策审计轨迹）。

### 二、 双系统融合架构 (Monorepo)
项目由两个核心模块组成，共享底层数据与契约，采用 Monorepo 架构管理。
1.  **Graph（事实与状态层 / 发动机）**：高精度、低完整度。事实与状态的**唯一权威裁决源**。通过节点、边、`entity_ref`、`state`、`invalidates` 做机械化的事实失效判定。
2.  **Pack（叙事与软料层 / 驾驶舱）**：高完整度、低精度。退化为**派生视图 + 软知识容器**（偏好、方法论、技能）。不独立判定事实失效，仅渲染图增量为可读叙事，并存放图装不进的软料。

**融合纪律**：
*   **定主从**：Graph 是事实真源，Pack 是派生视图。同一知识的正文只能在一处持有（规则6）。
*   **防漂移**：在 Pack 的折叠规则中加入“职责边界声明”，明确事实失效的权威判定上移至 Graph，Pack 内的 `[DEPRECATED]` 仅为过渡脚手架。

### 三、 核心数据模型 (Graph Schema)
#### 1. 节点 (Node)
*   `node_id`: 唯一标识。
*   `type`: `UserGoal | Constraint | Fact | ToolResult | Decision | FileArtifact | OpenTask`
*   `content`: 节点内容。
*   `entity_ref`: **核心机制**。稳定实体标识（如工单号、API路径、文件路径）。用于机械精确匹配，判定“同一实体”。无法确定则填 `null`。
*   `state`: 归一化状态枚举（`open | blocked | in_progress | implemented | deployed | resolved | cancelled | unknown | null`）。
*   `status`: `active | superseded | done`。

#### 2. 边 (Edge) 与失效语义
*   `relation`: `refines | depends_on | supports | invalidates | implements | serves | produces | derived_from`
*   **事实失效 (Invalidation)**：当新事实使旧节点不再成立（状态互斥），旧节点必须标记为 `superseded`，并添加 `{ "source": "新ID", "target": "旧ID", "relation": "invalidates" }` 边。

### 四、 数据管线与目录结构
#### 1. 物理目录结构 (单一真源原则)
```text
contextledger/
├── raw/               # 唯一事实真源 (Append-only，不可回改)
│   └── projects/      # 原始对话/资料
├── normalized/        # 纯函数派生层 (f(raw)，无损降噪，可安全删除重生成)
│   └── projects/      # 附带 manifest 和 span 映射
├── graph/             # 事实与状态层
│   ├── scripts/       # 全局机械校验脚本
│   ├── prompts/       # Extractor 模板
│   └── projects/      # 每个项目独立的图状态 (graph_state.json, patches/, slices/)
├── pack/              # 叙事与软料层
│   ├── _templates/    # 折叠规则
│   ├── skills.md      # 软知识
│   └── projects/      # 每个项目的 memory_pack.md
└── contracts/         # 跨模块契约 (唯一真源)
```

#### 2. 三条核心管线
*   **写入管线 (高频/每轮)**：`Raw` → `Normalized` → `Graph Slice` → `Extractor (LLM)` → `Patch` → `Reconcile` → `Apply` → `Graph State` → `Lint`
*   **折叠管线 (低频/每N轮)**：`Raw` → `Normalized` → `Maintenance Prompt (LLM)` → `Memory Pack`
*   **装配管线 (按需/Query驱动)**：`Graph State` (当前状态) + `Pack` (软料) → `Assembler` → `Context Pack` → `主 LLM`

### 五、 核心工程机制 (长程运行保障)
为解决 LLM 抽取的模糊性与长程图膨胀问题，系统采用 **“LLM 提议，机械层裁决”** 的三层防护架构。

#### 1. Graph Slice (O(1) 上下文控制)
*   **定位**：只负责省 Token，**不负责正确性**。
*   **机制**：根据当前轮对话提取符号（工单、文件），从全图检索相关节点。每个分组（如 `active_open_tasks`, `conflict_candidates`）设有**硬上限**，确保 Extractor 的上下文规模恒定，不随全图线性增长。

#### 2. Reconciliation Pass (事后校验 / 事前拦截)
*   **脚本**：`reconcile_patch.py` (Apply 前运行，只读)。
*   **机制**：用全图数据复核 Patch 的自洽性。核心依赖 `entity_ref` 精确匹配。
*   **强判定**：若 Patch 新增节点与全图 Active 节点 `entity_ref` 相同且 `state` 互斥，但 Patch 未声明 `superseded` 和 `invalidates`，直接报 **Error** 阻断。
*   **弱兜底**：无 `entity_ref` 时，仅做文本相似度 Warning。

#### 3. Graph Lint (全图质检)
*   **脚本**：`graph_lint.py` (Apply 后运行，只读)。
*   **机制**：检查全局不变量。包括：孤立 Active 节点、`invalidates` 指向未 `superseded` 的节点、同一 `entity_ref` 下互斥的 Active State、OpenTask 生命周期异常等。

#### 4. 失败处理与隔离 (Quarantine)
*   **有界重试**：Reconcile/Lint 失败时，将结构化 Error 回填 Prompt 重试（上限 N=2~3 次）。
*   **失败隔离**：超限仍失败，则拒绝合并，将 Patch 隔离至 `quarantine/` 目录。**宁可漏记一轮，绝不污染主图**。

### 六、 契约体系 (Contracts)
`contracts/` 目录是防止双系统漂移的唯一权威约束：
1.  **01_raw_anchors.md [STABLE]**：Raw ID 格式规范；经 Normalized 抽取时，Span 必须翻译回 Raw 偏移，确保脊柱锚点不断链。
2.  **02_ownership.md [STABLE]**：知识正文归属权。判断口诀：“是否会被后来事实推翻？”——会则归 Graph，否则归 Pack。
3.  **03_graph_schema.md [PROVISIONAL]**：节点/边结构与失效语义定义。验证完成前下游不得依赖。
4.  **04_assembly.md [PLACEHOLDER]**：装配环节前置闸。等 Graph 失效机制在真实冲突上验证通过后方可启用。
5.  **05_normalized.md [STABLE]**：Normalized 必须是 `f(raw)` 纯派生物；不得注入新信息或判断；必须带可重放 Manifest。

### 七、 Extractor Prompt 设计原则
*   **分离缓存**：System Prompt 包含固定的 Schema 规则与生命周期判定逻辑（支持 Prompt Caching）；User Prompt 仅注入当前 Slice 与本轮对话。
*   **事实一致性反向检查**：强制要求 LLM 扫描 Slice 中的 `conflict_candidates`，若发现 `entity_ref` 相同且 `state` 互斥，必须输出 `superseded` 和 `invalidates`。
*   **命名契约**：严格规范 `entity_ref` 的命名（如文件用原文，组件用 snake_case，未分配工单填 `null`），防止因命名不一致导致强匹配退化。

全篇归属判据为“凡能想象其有朝一日被推翻的，归 TaskGraph”——但几乎所有有用的东西都能被想象推翻（方法论会过时、偏好会变）。该判据像黑洞一样吸干 Pack，迫使第八节虚构“软知识失效永久累积”的窟窿，最终 Pack 只剩“语感、叙事质地”这种近乎为空的残渣。

## GPT-5.5 对 ContextLedger 设计初稿的评审意见

### 一、总体判断：基线成立，但四类判据需钉死才能作为执行规格

这版白皮书可作为 ContextLedger 的架构基线，但还不能直接当执行规格。最大优点是四层切分原则——按更新规则而非内容主题切分 raw / TaskGraph / MemoryPack / Assembler——这套骨架立得住。

### 二、最大问题：归属口诀过强，需修正

当前口诀“凡能想象其有朝一日被推翻的，归 TaskGraph”会将 Pack 吸空。正确判据应为：**变后是否必须立刻机械失效、是否有明确 entity_ref、是否需要 invalidates 链、是否影响当前任务合法状态**——若“是”则进 Graph，否则留 Pack。

### 三、具体改进建议

1. **降级第八节的“软节点提升”方案**：软知识默认留在 Pack，只有当其变成“当前执行纪律/硬约束/项目政策”时才提升为 Graph 节点。不将“会过期的软知识”大规模提升。
2. **增加 Entity Resolution 层**：这是当前最大工程缺口——整个 Graph 的机械失效依赖 entity_ref 稳定，而该字段由 LLM Extractor 生成，跨轮命名不一致会直接拖垮 invalidates 机制。需维护 canonical id、别名、合并/分裂记录及跨轮一致性检查。
3. **Assembler 设置“不可静默丢弃硬约束”机制**：Graph 中被标为 critical/must_include 的 active 约束，预算不够时应优先机械压缩表达、降软料预算、发警告或拒绝装配，而非静默丢关键约束。建议给节点标注装配优先级（must_include/should_include/optional/background），评测关键 active 约束进入上下文的召回率是否接近100%。
4. **Pack 增设“事实泄漏”lint**：若 Pack 条目含当前状态/决策/约束，则要求拆分——硬事实归 Graph、软解释归 Pack、连接用 graph_ref。
5. **叙事快照降为调试视图**：核心消费者仍是 Assembler，不因“人类可读”将 MemoryPack 诱导成持久叙事摘要器。

### 四、最小下一步

先做最小闭环：raw → Extractor → entity_ref resolver → patch → reconcile/apply/lint → graph_state → Assembler L1/L2 输出。先验证“旧约束被新约束推翻时，系统能否稳定保留 supersession 链且最终上下文只暴露当前成立状态”。

### 五、一句话总结

ContextLedger 的核心不是“记住更多”，而是“把当前仍成立的东西机械地守住，把软经验作为可召回但不夺权的辅助层”。
## Claude-Opus-4.8 对 GPT-5.5 评审意见的回应与补充

### 一、总体信号：四个必改项已锁定

两路独立评审（Claude-Opus-4.8 与 GPT-5.5）从不同切入口出发，收敛于四组相同意见：归属判据需修改、软节点提升需降级、必须补 Entity Resolution、Assembler 需设 must-include 不可静默丢弃。四项可视为已由讨论锁定的必改项。

---

### 二、关键分歧：三处需收窄

**分歧一：归属判据不应以“是否需要机械失效”为准。**

该判据是循环的——“是否需要机械失效”取决于把内容放进 Graph 还是 Pack，而非知识本身的属性。应使用“死于离散事件 vs 渐进褪色”作为判据：若失效存在一个可被指认、可挂 `invalidates` 边的时刻，归 Graph；否则归 Pack（靠“不再相关”自然从检索中掉出）。

**分歧二：“软节点提升”概念应被彻底删除，而非收窄。**

GPT 的“只提升成为执行纪律的软知识”方案存在问题：其示例（如“Phase 0 禁止实现 Assembler”）在白皮书 6.2 中本就是 Graph 节点，从未是软料；将“本应归 Graph 的纪律”与“真正软料”混入同一张提升表，为“提升”动作续了命。但“提升”本身即是病灶——某知识先在 Pack 存活再迁移至 Graph 的过程中，迁移窗口即为双源漂移的发生现场。正确做法非“先放 Pack 变重再提升”，而是**抽取时即分对**：纪律/约束直接生为 Graph 节点，经验/偏好直接生为 Pack 条目，中间无迁移通道。

**分歧三：Entity Resolver 必须合并保守，GPT 仅防漏匹配、未防错合并。**

GPT 的规格（canonical id、别名、merge/split 记录）方向正确，但只覆盖“同一实体被叫成不同名”的漏匹配。反向错误——将两个不同实体错合并——更致命：漏匹配代价为某旧约束多活几轮（可后续发现），错合并代价为给无关实体间凭空造出 `invalidates` 边，使仍成立的决策被错误 supersede，且图内完全自洽、lint 无法捕获。Resolver 须继承机械核纪律：**拿不准则不合并，存疑实体保持分裂并隔离**。激进合并的 resolver 比没有 resolver 更危险。

---

### 三、GPT 的贡献：两处被采纳的补充

- **must-include 优先级机制**：正面回应“收敛被搬进装配器”的问题，采纳 `must_include/should_include/optional/background` 分层 + 召回率评测。但需补一层：must_include 集合本身在长程项目中亦可无界增长，因此**头条评测项永远是关键 active 约束的召回率**，优先级分层仅为手段而非结论。
- **Pack Lint 自动化检测**：采纳方向，但须注明性质——判断“一句话是否为当前事实”属模糊的 LLM 判断，无法如 `graph_lint` 般机械裁决，因此 Pack Lint 只能作**启发式警告**而非硬门禁。系统中应明确区分两类 lint：`graph_lint` 为机械可阻断；Pack 语义 lint 为概率性提示。

---

### 四、GPT 未触及但 Claude 坚持上升的一条

**ContextLedger 是一致性引擎，非真值引擎。** Lint 能保证图内部不自相矛盾，但无法保证图反映现实——Extractor 会抽错、`entity_ref` 会配错、用户会断言假前提。该边界应上升为系统级声明，而非埋于医疗场景脚注中。

---

### 五、收口：第一个里程碑与可执行降级

**最小闭环**：raw → Extractor → entity resolve → patch → reconcile/apply/lint → graph_state → Assembler L1/L2 输出。Pack 暂缓。

**单一验证命题**：当旧约束被新约束推翻时，supersession 链是否稳定保留，且最终上下文只暴露当前成立态、不泄露废案。此命题跑通前，Pack 折叠、叙事快照、软节点等均属装饰。

**叙事快照降级**：最终消费者均为 LLM/Agent，“给人读的叙事快照”在 Phase 0 不应占用折叠预算，最多作为事后导出的 debug 视图，而非核心制品。白皮书 5.4 将其列为三分产物之一，过重了，建议降级。

---

### 六、一句话可执行版本

删掉“提升”动词，改成抽取期分类；补一个合并保守的 entity resolver；仅用“supersession 链是否可靠”验证机械核——其余一切（软节点、Pack 打磨、叙事快照）全部等其跑通后再议。此线落成 Extractor 抽取时即可照做的判定清单后，第八节与“提升”概念可一并取消。

该版本已基本吸收上一轮评审中“可操作层”的全部批评（entity resolver、must_include、merge-review、三层膜可靠性模型、归属判据修正）。但文档存在三个自身制造的结构性盲区：数据模型表达能力与产品定位之间的沉默矛盾、度量仪表晚于管线建设的排期错位，以及两套正交分类系统交互未定义导致的实现歧义。

---

### 二、值得肯定的收敛成果

- **按更新规则切分文件边界**（§2.1）是非平凡的判断，避免了按内容类型随意切分的常见错误。
- **§1.4 的认识论边界声明、Phase 0“验证形状而非命题”的自我克制、§8.2 将三类失败收敛为“静默僵尸”**，表明设计经过了自我证伪。
- **§8.2 承认机械裁决核（reconcile/apply/lint）对端到端价值贡献最小**——真正的可靠性在于三层概率性膜——但该结论与 §9.3 Phase 1 排期（将重心压回引擎管线）存在直接冲突。

---

### 三、最大未修补缺口：Schema 仍是二值·整体·单调，与声称的“演化审计”价值正面矛盾

§3.1 的 `status` 仍为 `active/superseded` 二值，失效为整体替换，无 valid-time 一等维度。而 §9.1 将“可审计的决策演化轨迹”列为核心价值，理论附录更将模型认领为**非单调推理/AGM/真值维护系统**的工程化身，并明确点出 revival 会导致 active set 非单调演化。

问题在于：**当前 schema 能表达的“演化”只有“A 被 B 整体取代”这一种贫乏形态**。真实审计需要的部分失效（只推翻子句）、条件失效（时间窗/配置依赖）、revival（需求回滚）均无法表达。可选择“Phase 1 先只做二值单调”作为工程切片——但文档未将此显式声明为已知模型边界，也未在 schema 中预留 `invalidation_type` 和 valid-time 字段位置。这会重复附录第四节警告的“死于状态矩阵填错/非单调性使一致性保证失效”路径。

---

### 四、归属判据存在时间错配：判据指向未来属性，却于抽取期裁定且无修复通道

§2.2 将判据改为“死于离散事件 vs 渐进褪色”方向正确，但**“它将来怎么死”是未来属性，而分类发生在抽取期**。一条被归为“褪色”的方法论，可能在三十轮后被明确决策离散推翻——此时它本该在 Graph 挂 `invalidates`，却只靠 Pack“相关性衰减”慢慢掉出，成为静默陈旧软事实；反过来，一条记入 Graph 的约束也可能从未发生“事件时刻”，只是慢慢没人提，永久占着 must_include 预算。

v2.0 为消灭“双源漂移窗口”而明确规定“无迁移通道”（§2.2、附录B），这意味着**抽取期的一次误判是永久且不可修复的**。考虑到做出分类的是入口膜（概率性 LLM），而判据依赖其无法获知的未来属性，该误判率不会低。此风险应进入 §8.2 的膜清单，并需要有兜底回路（当前 merge-review 只覆盖实体漏匹配，不覆盖归属误判）。

---

### 五、“合并保守”策略可能与系统核心使命方向相反，且其优越性未经测量

§3.3 的逻辑是“错合并比漏匹配更危险，故拿不准不合并”。但文档同时承认两种失败均为静默的、lint 不可见的僵尸。核心使命是**最大化失效召回**（不让废案污染上下文），漏匹配直接导致失效漏召——这是对核心使命的直接背叛；错合并导致误杀有效决策——伤的是精确率。在立项即为“杀死活着的废案”的系统中，“拿不准就放过一个废案”是否优于“拿不准就误杀一个有效决策”，**取决于两类错误的真实基率和 merge-review 回路的实际效力，无法通过思辨定论**。

§3.3 与附录B 将此断言为公理，但此问题正是上一轮失效回放基准（含 alias 陷阱、non-invalidation 诱饵）**唯一能回答的**。在跑过基准前，“合并保守”应为待验证默认策略，非公理。

---

### 六、可靠性是膜的乘积，非膜的最小值

§8.2 称“端到端可靠性 = 最弱层膜”——作为上界正确，作为实际可靠性则乐观。三层膜串联且失败相互独立，实际可靠性为**各层可靠性的乘积**。0.9³ = 0.729，非 0.9。此区分直接影响端到端召回率预期标定。建议改为“最弱膜是天花板，实际低于任何单层膜”，并用端到端 Set-F1（基准 schema 的 checkpoint 机制）直接评测，而非分层测后取 min。

---

### 七、两套正交分类系统的交互未定义，导出具体矛盾

§5.2 用 L1–L4（来源/新鲜度）分层，§5.3 用 must/should/optional/background（优先级）分层，两者交互未定义。具体矛盾：

§5.3 第3点规定“仅 active 节点可进 must_include”——即 must_include 为 **Graph 独占**。但 §5.2 L1 常驻基线明确包含“核心偏好”，而偏好按 §2.2 归 **Pack**。于是 Pack 来源的“核心偏好”无法成为 must_include，只能落入“预算不够时可被压缩”的软料区——**与 L1“每轮必进”的定义直接冲突**。此交互须显式裁定，否则 Assembler 实现时会产生歧义。

---

### 八、每轮全量物化被包装为必然性，实为成本/正确性权衡

§7 用“失效是离散事件，攒轮会断 invalidates 链”证明必须每轮跑。该论证只证明**事件须在发生时附近被捕获**，不等于“每轮全套 reconcile/apply/lint/全图 O(N) 校验须全量物化”。可每轮廉价 append 事件日志，将昂贵归并和不变量校验惰性化或批处理至查询时/每 N 轮。

每轮全量物化是一个赌注——赌摊销后的每轮代价低于惰性重导出代价。对于多数轮次不会被以冲突相关方式回查的长程任务，此赌注可能亏损。“每轮捕获事件”与“每轮全量归并入图”应拆为两件事。这为折叠器提供了比“捞软料”更硬的定位候选：**惰性归并的批处理单元**。但需承认权衡——惰性化引入装配时图态与已 append 未归并事件间的一致性窗口，此窗口恰是 §5 装配最需要图最新的地方。文档未将此作为权衡讨论。

---

### 九、文档详略与契约成熟度倒挂

§5 Assembler 是全书最详尽章节之一，但依赖的 03_graph_schema 为 PROVISIONAL，04_assembly 为 PLACEHOLDER，且 Assembler 排于 Phase 3。最详尽论述压在最低成熟度契约上，返工风险高。此部分确定性被版式语气高估。

---

### 十、核心建议：先建度量，再建管线

§8.2 结论是“管好三层膜”，但无法管理未测量的膜。§9.3 Phase 1 为搭建管线（raw→extractor→resolve→…→assembler），而失效回放基准未进排期、未设门禁。这正是上一轮评审最强建议被丢弃的地方：**下一步最有价值的是先建失效回放基准，非写管线代码**。基准一次回答三个问题：比 Flat RAG 好多少（必要性）、缺陷补到哪算够（完成线）、“合并保守”是否成立（§五经验问题）。

没有此尺子：Phase 1 单一命题“supersession 链稳定保留、只暴露当前态”**无法判定是否达成**；§3.3“合并保守优于激进”为**未经测量断言**；§5.3“召回率接近 100%”为**无测量仪器的口号**。

建议排期改为：**Phase 0.5 = 标注 15–20 条覆盖矩阵填满的失效回放轨迹（含部分/条件/revival/alias 陷阱/provenance 冲突/non-invalidation 诱饵），将 precision/recall/Set-F1 设为后续所有阶段硬门禁。** 此举亦将强制暴露二值单调 schema 无法表达部分/条件/revival 形态，在写代码前揭示地基缺口——即附录“使其成为项目中首次数学式有句号的工作”的真正落地位置。

---

**总结**：设计在“问题选对、抽象选对”上成立，v2.0 证明其能吸收批评与自我证伪。当前真正风险不在于已知的 `entity_ref`、成本、lint 扩展性，而在于：**① schema 的二值单调性与“演化审计”价值间的沉默矛盾；② 度量仪器排于管线之后，使可靠性承诺目前不可证伪。** 先建尺子，再用尺子逼出 schema 的真实需求——此优先级高于继续细化 Assembler。
## GPT-5.5 对 ContextLedger v2.0 的最终评判

### 一、总体判断：方向正确，理论身份已认领但形式语义尚未补上

ContextLedger 的核心判断成立——长程 Agent 的真正痛点并非 retrieval（找不到信息），而是 retraction（找到了已作废的信息却不知其已作废）。将失效、撤回、supersession、当前态裁定置于系统中心，此抽象正确且稀缺。v2.0 已从“聪明的记忆压缩方案”升级为“有理论根基的状态治理系统”，但当前状态仍为可靠系统的设计草图，而非可靠系统本身。真正的分水岭在于将失效语义、实体解析和评测基准做成可执行契约。

### 二、v2.0 的主要进步

- **承认一致性引擎而非真值引擎**，避免过度承诺。
- **raw / TaskGraph / MemoryPack / Assembler 职责切开**，分层清晰。
- **归属判据改为“死于离散事件 vs 渐进褪色”**，避免 Pack 被吸空。
- **加入 Entity Resolver 与 merge-review**，承认 `entity_ref` 是机械失效体系的地基。
- **must_include 机制**确保关键 active 约束不被静默丢弃。

### 三、核心批评：理论身份已认领，但 schema 仍为二值单调

v2.0 已将自身定位为非单调推理/信念修正/真值维护系统的工程化身，但当前 TaskGraph 的核心状态仍为 `active/superseded` 二值，仅能处理“新决策整体替代旧决策”这一种形态。现实中的部分失效、条件失效、时态失效（valid-time vs transaction-time）、revival/回滚均无法表达。若这些不进 schema，系统处理的只是“线性覆盖式演化”，非完整“决策演化审计”。

可接受 Phase 1 仅实现 full invalidation 单调子集，但须显式声明 partial / conditional / revival / bitemporal 为已知未覆盖边界，并在 schema 中预留字段位置。否则系统定位与数据模型之间将长期存在落差。

### 四、raw 的措辞修正

“raw = 唯一事实真源”易误导。更准确表述为：
- **raw = 唯一原始证据真源 / source-of-record for observations**（记录“谁在何时说过什么/工具返回过什么”）
- **TaskGraph = 当前系统信念态 / current adjudicated belief state**（经过裁定的当前状态）

此区分与“一致性引擎，非真值引擎”的认识论边界一致。

### 五、归属判据的误分类修复路径

“抽取期即分对，无迁移通道”方向正确（防止双源漂移），但存在风险：“死于离散事件 vs 渐进褪色”有时在抽取当下无法完全判断。经验今天像软知识，未来可能被明确决策离散推翻；硬约束也可能只是逐渐失去适用性。

建议不恢复自由迁移通道，但增加克制机制：**不迁移正文，但允许追加“归属修正事件”或“适用性裁定事件”**。旧 Pack 条目仍留 Pack，Graph 不复制正文，但可追加事件表示“某 Pack 条目在某 scope 下被某决策禁用/降权/不再适用”。既不破坏正文单一归属，也为误分类留下修复路径。

### 六、合并保守策略：合理默认值，非永真原则

Entity Resolver“拿不准不合并”作为高风险场景默认值合理，但不能作公理。错合并误杀有效事实，漏合并让废案继续 active，两者均为静默错误。“保守合并是否更好”须靠失效回放基准校准，非哲学判断定论。Resolver 应输出 match_confidence、merge_reason、non_merge_reason、candidate_aliases，否则 merge-review 将沦为事后人肉猜谜。

### 七、对 Claude-Opus-4.8 评价的态度

大体同意其主线判断（schema 二值单调性是核心缺口、benchmark 应提前、Assembler 不应先于契约过度细化、端到端可靠性不等于机械核可靠性），但比其更温和：v2.0 已在认识论边界、三层膜、dependency_refs、Phase 1 命题收缩上吸收了大量深层批评。更准确说法是“v2.0 吸收了工程层和系统边界层批评，但尚未将非单调推理的形式语义真正落到 schema 和 benchmark 上”——此非否定，而是下一阶段精确任务。

### 八、优先建议：先建失效回放基准，再写管线

当前应优先完成三件事：
1. **失效回放基准**（覆盖 full/partial/conditional/revival/alias 陷阱/provenance 冲突/non-invalidation 诱饵），定义硬指标：invalidation precision/recall、active-set Set-F1、must_include recall、entity-resolution false merge/split rate。
2. **state / invalidation 形式语义**——将理论身份落地到可执行 schema。
3. **entity resolution 可观测评测**。

此三件事完成前，Pack 细化、Assembler 深化、Wiki 设计均可暂缓。没有 benchmark，架构正确性无法证伪；没有形式语义，理论身份悬空；没有 resolver 评测，地基可靠性不可测量。

## MemoryPack 的必要性评估：增值层，非核心引擎

### 一、直接判断：MemoryPack 不是“不可或缺”的

白皮书自身的 Phase 1 规划已明确证明这一点：最小闭环仅包含 raw → TaskGraph → Assembler，**Pack 被明确标注为“暂缓”和“装饰”**。在核心命题（旧约束被新约束推翻时，supersession 链稳定保留且上下文只暴露当前态）验证通过之前，MemoryPack 不参与系统核心功能的运作。

### 二、核心价值归属：TaskGraph 承载不可替代的失效追踪职能

ContextLedger 的核心定位——**“让 Agent 在运行数月后依然精确知道什么已经不成立了”**——完全由 TaskGraph 的 `entity_ref` + `invalidates` 机械机制承载。这是与 Flat RAG 的根本差异，也是产品战略的核心差异化。MemoryPack 不贡献此价值：它处理的软知识（方法论、偏好、技能）不涉及机械事实失效、不依赖 `invalidates` 链、不依赖 `entity_ref` 精确匹配。

### 三、删除 MemoryPack 的影响：丢失“知识复利”，保留“状态追踪”

若删除 MemoryPack，系统仍能完整追踪事实失效并保证上下文只暴露当前成立态；丢失的是软知识沉淀（方法论/偏好）、渐进褪色处理和跨项目经验复用。丢失的是“知识复利”维度，而非“状态追踪”核心。两者价值维度不同。

### 四、重新定位：核心层与增值层的分离

| 层级 | 组件 | 价值 | 必要性 |
|---|---|---|---|
| 核心层 | TaskGraph + Assembler + raw | 事实失效追踪、状态裁决、证据溯源 | **不可或缺** |
| 增值层 | MemoryPack | 软知识沉淀、知识复利 | **有价值但非必需** |

MemoryPack 是 ContextLedger 的**第二曲线**，使系统从“状态追踪工具”升级为“知识复利平台”，但此升级可选，非必需。

### 五、替代方案的可行性

即使承认软知识有价值，也可质疑其是否需要独立的 MemoryPack 层。替代方案包括：直接放 raw 用 flat RAG 检索、放入 TaskGraph（白皮书已否决）、或使用外部知识库（Notion/Obsidian）管理，ContextLedger 仅做引用。MemoryPack 的独特价值在于结构化的软知识容器（带 `dependency_refs`/`scope_refs`）及与 TaskGraph 的显式联动（依赖节点 supersede 时软卡自动暂停）——但这些都是工程优化，非架构必需。

### 六、结论：有机组成部分，但非“不可或缺的”有机组成部分

MemoryPack 解决了 TaskGraph 无法建模的软知识问题，提供了知识复利能力，但在核心命题验证通过前是“装饰”而非“必需”。资源有限时可遵循白皮书阶段规划：Phase 1 仅实现 TaskGraph + Assembler，Phase 2 视需求决定是否引入 MemoryPack。ContextLedger 的核心是 Ledger（账本/状态追踪），MemoryPack 是附加的“知识沉淀层”——无它系统仍是完整的 ContextLedger，有它则升级为“知识复利平台”。此为产品差异化选择，非架构必需约束。
## ContextLedger 对后台自动执行智能体的适配性分析

### 一、核心判断：不仅适合，而且更刚需

ContextLedger 不仅适用于后台自动执行的智能体，而且在此场景下其核心价值——事实失效追踪与状态一致性——比在人机对话中更加关键。原因在于：对话场景中人类可发现并纠正过期的旧约束，但后台自动执行场景（如量化交易、自动化运维、持续集成）中**没有人类兜底**，“废案不死”的代价直接导致系统崩溃或经济损失。

### 二、核心概念泛化：将“对话”映射为“后台执行”

白皮书当前表述带有对话色彩，但底层逻辑（状态机、失效追踪）完全适用于后台场景，只需对输入/输出接口进行泛化：

| 对话场景概念 | 后台场景泛化概念 |
|---|---|
| Turn（轮次） | Event/Step（事件/执行步）——一次环境观测、一次工具调用结果或一次内部思考 |
| 用户对话文本 | 观测流——JSON日志、Webhook payload、API响应、工具执行结果 |
| LLM回答 | 动作/规划——执行的代码、发出的API请求或下一步规划 |
| Assembler的触发 | 从“回答用户问题”变为“决定下一步动作”或“执行当前动作” |

### 三、架构需补充的四个关键点

1. **输入源泛化**：raw 层从存储 Markdown 对话改为存储结构化事件日志，每条记录需带严格时间戳和因果链标识（Trace ID），以处理后台场景中可能乱序到达的事件流。

2. **并发与异步处理**：后台场景中多事件可能并发触发状态更新，需引入事件队列按时间戳和因果依赖排序；Apply 操作必须严格串行化或引入乐观锁/版本号机制；冲突事件进入 Quarantine 等待重新 Reconcile。

3. **工具调用作为一等公民**：`ToolResultNode` 需被提升为核心地位，工具的成功/失败/超时/返回值是触发 TaskGraph 状态变更（`invalidates`）的主要来源，可直接推翻旧事实并触发新任务。

4. **Assembler 触发模式转变**：从 Query-driven 转为 Action-driven，输入不再是用户自然语言查询，而是“当前触发事件的上下文 + Agent 的规划目标”，输出用于 Agent 的 Planning 而非生成回答。

### 四、场景对照与建议

后台 Agent 容错率极低，机械核（Lint/Reconcile）和 `must_include` 机制的价值在此场景最大化。建议白皮书升级为 v3.0 时新增“适用场景扩展”章节，将定位从“对话压缩器”升维为“**无人干预长程 Agent 的信念态裁决引擎与失效审计轨迹系统**”，并补充事件队列、因果排序、并发冲突解决等架构设计。
## ContextLedger 对后台自动执行智能体适配性的深度评估

### 一、核心判断：语义层可迁移，但轮次作为一致性原语不可简单替换

用户提出 ContextLedger 是否适用于无轮次对话、后台默默自动执行的智能体场景。Qwen-Plus-3.7 的结论是“不仅适用，而且更刚需”，认为只需将“轮次”泛化为“事件/执行步”，底层逻辑完全不变。Claude-Opus-4.8 与 GPT-5.5 在后续评估中均对 Qwen 的方向判断表示认同（无人兜底场景下失效追踪的确更加刚需），但对其论证方式提出关键修正：**轮次不是可替换的 I/O 外壳，而是白皮书整个正确性模型免费搭载的并发控制点和读快照点。**

### 二、Qwen 论证中的三个结构性低估

**第一，Turn 被误解为接口壳，实为一致性原语。** 白皮书的写入管线（Extractor → Patch → reconcile → apply → lint）隐含串行假设：上一轮 apply 完才有下一轮，“当前成立态”在轮次之间无歧义。这白嫖了线性一致性。后台事件流并发、乱序，没有天然提交边界，Qwen 提出“串行化 apply + 乐观锁”只是一句话带过分布式共识问题——乱序下“A 在 B 前还是后”需要因果序定义，Trace ID 远不够支撑，这不是补一个事件队列能解决的补丁，而是重建一致性模型。

**第二，失效从“声明”退化为“推断”，入口膜变弱。** 对话场景下失效被语言显式标记（“我改主意了”），Extractor 只需解析已声明意图。后台场景中无人声明，系统需从工具返回值（如 `{"item_A": 0}`）中推断“这推翻了之前 item_A in_stock”——抽取任务从“解析”变为“从原始观测推断因果失效”，概率难度显著增大。按“端到端可靠性 = 三层膜”的模型，入口膜本就是瓶颈，自主场景把它变得更弱，而非更强。Qwen 承认“容错率极低、无人兜底”却断言机械核价值最大化，漏掉了机械核恰是贡献最小的环节，而三层膜同时变弱且撤掉人类安全网。

**第三，bitemporal 缺口在后台场景从“可暂缓”变为“必须解决”。** 两位评审均指出二值单调 schema 缺少 valid-time vs transaction-time 区分，且列为 Phase 1 可暂缓边界。但对话中 valid-time ≈ transaction-time，暂缓合理；乱序事件流下两个时间轴必然分离——迟到观测描述的是早已成立的状态，系统必须能回答“事实何时**实际**成立”与“系统**当时**相信什么”两个不同问题，否则乱序到达的失效裁决直接错乱。“系统是否适配后台 Agent”与评审揪出的 schema 缺口是同一洞的两面。

### 三、GPT-5.5 的关键修正：后台可适配，但须保留“线性化提交语义”

GPT-5.5 同意 Claude 的主线判断，但补充：不必将后台场景等同于必然进入全量分布式一致性。可通过设计强制拿回“轮次”语义——**单写者事件日志 + 串行 apply + graph version 快照**，将“轮次”抽象为“可线性化的提交步”。即后台 Agent 虽无人类问答轮次，但可有机器版提交轮次。真正不能丢的不是“轮次”之名，而是**线性化提交边界 + 一致读快照**。

### 四、战略建议：压力测试而非 v3.0 扩张

Qwen 建议扩写 v3.0、增加四个架构点、价值指数级放大——此方向与上一轮评审“先建度量再建管线、先证伪再扩张”的收敛建议正好相反。正确用法是将此问题作为检验抽象是否有隐藏轮次依赖的压力测试，把发现写成已知边界声明，而非写适配章节。

具体落地三件事：①在 §1.4 边界声明中写明“Phase 1 假设串行提交语义（turn-based 或单写者事件流）；并发/乱序/bitemporal 为已知未覆盖边界”；②在 Phase 0.5 失效回放基准中增加“乱序到达 + 迟到观测”轨迹类别，在写代码前暴露二值单调 + valid≈tx 假设在乱序下崩掉；③将 slogan 升级为“为长程 Agent 提供可机械裁决的信念态与失效审计轨迹”——此条与 GPT-5.5 “raw = source-of-record / TaskGraph = belief state”的区分一致，零成本且方向正确。

### 五、一句话总评

Qwen 答对了“能不能”的方向，但答错了“难在哪”——难点不在换输入接口，而在于轮次免费提供的一致性与快照消失后，系统被迫直面并发一致性问题，而这恰好引爆了评审已指出的同一 schema 缺口。此问题最高价值是为失效回放基准新增一个能照出地基裂缝的测试维度，而非提前撰写 v3.0。
## 协调者作为线性化点：并发问题的收敛与残留的valid-time问题

用户指出，在多Agent架构中必然存在一个协调者Agent，它看到所有内容在时间上有先后，以协调者的观测顺序为准，每次调度任务及其完成作为一轮操作，并发问题即可消解。此判断直接切入问题核心，将之前讨论中过度的并发复杂性（向量钟、分布式共识等）以架构约束的方式降维。

### 一、用户的核心论点成立：协调者即为线性化点

在多Agent系统中，若存在单一协调者作为唯一写者，所有子Agent的执行结果必须返回协调者，由协调者按观测顺序写入ledger，则系统在逻辑上恢复串行语义。子Agent内部执行可以并发，但协调者“看到返回”的动作是串行的，并发在其观测面前坍缩为一条有序序列。因此，分布式共识、因果序、向量钟等机制在单协调者模型下不必进入Phase 1。用户正确地将后台场景还原为对话场景的等价结构——只不过“轮次”从人类问答替换为协调者的观测事件序列。

### 二、两个修正：粒度与valid-time

**1. Step粒度不绑“调度+完成”成一对。** 调度与完成在观测序中通常不相邻（协调者可能step5发任务X、step6发任务Y、step7收到X的结果）。正确的“一轮”应为协调者每观测到一个事件（发出调度、收到返回、超时、取消、改派）即为一个独立的ledger step。这反而更简单，与“以协调者所见为准”的原则完全一致。

**2. 协调者解决的是“谁来写”（一致性），解决不了“什么时候成立”（valid-time）。** 在并行派工场景下，即使协调者单写者、串行写入，协调者的观测序仍不等于事实的valid-time序。示例：协调者并行派出A（10:00查库存，10:05返回stock=0）和B（10:03查，10:04返回stock=10），按“后观测覆盖先观测”会判定stock=0，但A描述的事实（10:00）比B更旧（10:03），结论反了——一致性未破，但信念错了。错因不是并发写，而是协调者并行读使“到达顺序”与“事实成立顺序”脱钩。

此漏洞的修补不需要完整bitemporal，仅需一条接入约定：子Agent返回结果必须带 `observed_at` 字段，reconciler在有该字段时按valid-time裁决 `invalidates`，缺失时退回到达序。一个字段、一条规则即可挡住上述反例。

### 三、边界声明的精简版本

无需扩写v3.0，仅需将白皮书边界声明收窄为两句话：

> Phase 1 依赖单一协调者提供**单写者线性提交序**，据此解决状态写入的并发一致性，不需要分布式共识。
>
> 但协调者的观测序不等于事实的valid-time序（并行派工会令二者脱钩）；失效裁决在结果携带 `observed_at` 时按valid-time进行，否则退回到达序。完整bitemporal信念回补仍列为后续扩展。

如此既保住了“不搞复杂”的判断（共识部分确实砍掉），又没将一个仍在的洞写成已填。
## 关于 ContextLedger 对后台自动执行 Agent 适配性的讨论与白皮书补丁方案

### 一、核心讨论与判断

用户质疑 ContextLedger 白皮书的设计似乎只适用于人类对话场景（一问一答），是否也适用于无轮次对话、后台默默自动执行的智能体场景。Qwen-Plus-3.7 给出肯定答复，认为“不仅适用，而且更刚需”，只需将“轮次”泛化为“事件/执行步”。Claude-Opus-4.8 与 GPT-5.5 在后续评估中方向认同但纠正了 Qwen 的关键论证错误：**轮次不是可替换的 I/O 外壳，而是白皮书整个正确性模型免费搭载的并发控制点和读快照点**。Qwen 把后台适配说得太像“换壳”，低估了一致性问题。

### 二、用户的关键修正：协调者模型

用户指出，多 Agent 架构中必然存在一个协调者 Agent，它看到所有内容在时间上有先后。以协调者所见为准，每一次调度与任务完成作为一轮操作即可消解并发复杂性。此判断被 Claude 和 GPT-5.5 均认定为正确——协调者就是线性化点，子 Agent 可以物理并发，但只要存在单一协调者作为唯一写者，系统在逻辑上就是串行的，向量钟、分布式共识等无需进入 Phase 1。

### 三、Claude 的两个修正

1. **粒度问题**：不应将“调度+完成”绑定为一轮。实际中调度与完成在观测序中不相邻（step5 发任务 X、step6 发任务 Y、step7 收到 X 结果）。正确的一轮应为“协调者每观测到一个事件”（发出调度、收到返回、超时、取消、改派）为一个独立的 ledger step。

2. **协调者解决“谁来写”，解决不了“什么时候成立”**：在并行派工场景下，即使单写者、串行写入，协调者的观测序仍不等于事实的 valid-time 序。A 于 10:00 查 stock=0 于 10:05 返回，B 于 10:03 查 stock=10 于 10:04 返回——按“后观测覆盖先观测”会误判 stock=0，一致性未破但信念错了。修补只需一条接入约定：子 Agent 返回带 `observed_at`，reconciler 存在时按 valid-time 裁决 `invalidates`，缺失时退回到达序。

### 四、白皮书补丁方案（Claude 提出，GPT-5.5 采纳并收紧）

**补丁 1：新增 §1.5 执行模型边界**。声明 Phase 1 依赖单一协调者提供单写者线性提交序解决并发一致性；轮次泛化为 ledger step = 协调者每观测到一个事件；并行派工下观测序 ≠ valid-time 序，失效裁决在结果携带 `observed_at` 时按 valid-time 进行，否则退回到达序。完整 bitemporal 列为后续扩展。

**补丁 2：§3.1 追加 observed_at 与 valid_time 的关系说明**。`observed_at` 是轻量新鲜度键，仅用于同一 `entity_ref` 同一状态维度下当前态裁决排序，不构成完整 bitemporal 历史回填能力。Phase 1 中 `valid_time`/`tx_time` 仍为预留字段。reconciler 只保证：**较新的 `observed_at` 可推翻较旧状态；迟到但更旧的观测不得反杀较新的 active state；缺失 `observed_at` 时退回协调者到达序。**（GPT-5.5 收紧点）

**补丁 3：§9.3 Phase 0.5 增加“乱序到达 + 迟到观测”轨迹类别**，检验 valid-time 与 transaction-time 分离；增加 valid-time 裁决正确率指标；标注乱序轨迹会强制暴露 valid≈tx 假设在并行派工下崩掉。

**补丁 4：§6.2 防漂移汇总表新增一行**：“并行派工下观测序≠valid-time序 | observed_at 作排序键 + 缺失退回到达序（Phase 1）；完整 bitemporal 列为扩展”。

### 五、最终结论

协调者模型恢复轮次语义并解决并发一致性，但并行派工使 valid-time 问题独立存在，通过 `observed_at` 最小接入约定挡住，不将后台场景视为 v3.0 扩张，而是作为已知边界声明 + 基准测试维度。GPT-5.5 总体认同 Claude 的四个补丁，但强调 `observed_at` 必须明确为“只允许更新鲜观测推翻旧状态，旧迟到观测不得反杀当前态”。
## ContextLedger 的形态定位：框架无关的状态层库，而非智能体或插件

### 一、核心判断：否定“插件 vs 自建智能体”的二分

两个选项预设了错误的判断轴。真正决定形态的不是“要不要做一个智能体”，而是：**为了兑现 ContextLedger 的核心承诺，必须占据宿主循环里的哪一道接缝。** 把接缝问题想清楚，形态问题会自动有答案。

### 二、ContextLedger 不是智能体，将其做成智能体是战略陷阱

智能体的定义性特征是有目标、会规划、能自主发起动作。ContextLedger 不具备这些——它是对事件流的有状态函数：进来一个 turn/event，吐出 patch，机械合并进图，装配时吐出上下文。Extractor 虽是 LLM 调用，但它是纯转换，不是自主体。若给它套上规划循环、工具调用、编排，就是在不需要的地方凭空造一个 agent loop，而 agent loop 是 2026 年最拥挤、最快商品化、最不缺玩家的一层（LangGraph、CrewAI、OpenAI、各家自研循环）。一旦自建智能体，就迫使自己在“循环”这个无差异化的维度上竞争，而真正稀缺的能力——信念态裁决与失效审计——会被埋没在平庸的 agent 壳里。因此自建智能体应排除，理由不是技术做不到，而是它会让差异化定位失焦。

### 三、现有智能体的“记忆系统”与 ContextLedger 不在同一层

现有 agent 的记忆几乎清一色是**召回型**：将历史文本向量化按相似度检索，辅以用户偏好和会话摘要，回答“我们以前聊过/做过什么”。ContextLedger 是**裁决型**：机械 supersede、invalidates 链、当前成立态，回答“此刻什么还成立、什么已被推翻”。两者根本不在同一层——现有记忆大致落在装配分层的 L3/L4（软知识、RAG），而 ContextLedger 独占的是 **L2（经失效裁定的当前态）**。几乎没有任何现有记忆系统在做 L2。因此 ContextLedger 不是竞品，而是它们缺失的那一层。这提供了清晰的市场话术：不抢召回记忆，补上“现在什么是真的”这一层，原生记忆可降级为 L3/L4 的喂料源。

### 四、核心约束：must_include 保证迫使占据最终装配接缝

§5.3“绝不静默丢弃 must_include”的承诺，只有在 ContextLedger **控制最终送进 LLM 的那一口上下文**时才能兑现。若仅是旁路 sidecar、“建议”宿主注入内容，而宿主原生记忆同时独立注入，则既不能保证 must_include 真正进入上下文，也会与宿主记忆打架造成双重注入和冲突——召回率保证即失效。

这一条几乎一锤定音：排除对封闭产品（如某些托管 Assistants）做插件——因为那些产品不让你拦截装配步；也排除纯建议式旁路。**必须拥有 ingest 和 assemble 两道 hook，尤其是 assemble 的最终决定权。**

### 五、推导出的形态：框架无关的状态层库/SDK

对外暴露极薄的集成契约，本质上两个钩子：`ingest(event)`（保证 raw 流完整）和 `assemble(goal/query) → context`（拥有最终上下文）。目标客户是**在可编程编排层（LangGraph、自研循环等）上自己搭长程 agent 的开发者**——只有在这些宿主中，才能插入上述接缝。这与 §1.5“单协调者单写者”假设严丝合缝：协调者调用两个 hook，将所有观测线性化后喂给 ingest，在决定下一步前调用 assemble。ContextLedger 不需要是协调者，但假设宿主中存在一个协调者作为其单一写者。

### 六、参考循环是测试台，不是产品

为跑通 Phase 0.5 失效回放基准、进行 demo，需要一个**极薄参考 agent loop** 作为受控环境——但它属于测试夹具和样例，不是售卖对象。售卖的是库；参考循环用于证明库有效并示范集成方式。将此区分清楚，可避免被“我得先有个能跑的智能体”诱导去构建全功能 agent。

### 七、MCP 的适用边界：松接口只适合软料，硬保证需深集成

MCP 协议可作为 agent 接入外部上下文/工具的事实标准，理论上可将 ContextLedger 暴露为 MCP server（以 resource 提供“当前裁定态”，以 tool 提供 ingest 事件）。但 MCP 是**拉取模型**，宿主决定是否拉取及是否使用——这意味着拿不到最终装配决定权，must_include 保证落空。因此 MCP 适合 L3/L4“建议式软知识”的接入面，但硬保证（L2 当前态 + must_include）仍需更深的库内集成。

### 八、核心判据

**凡需兑现机械保证的部分必须深集成；仅提供软料的部分可走 MCP 等松接口。** 形态结论：ContextLedger 是基础设施库，不是智能体；“插件 vs 自建”是伪二分，真问题是“必须占据哪道接缝”；must_include 将答案钉死在——必须拥有最终装配权，目标客户是在可编程编排上自建 agent 的开发者，而非封闭产品的旁路插件，更非自行下场做 agent。
## GPT-5.5 对 ContextLedger 形态定位的补充判断

### 核心判断：ContextLedger 应为 Agent 的上下文运行时，而非 Agent 本身

GPT-5.5 认同 Claude 的大方向，但进一步明确结论：ContextLedger 既非普通插件，也非自建智能体。其准确形态是嵌入智能体循环、接管“事件入账”与“最终上下文装配”两道接口的 **Agent Context Runtime / Memory Kernel**。参考 Agent 仅用于演示、测试和范式说明，非核心产品。

### 一、为何不是普通插件

普通插件拿不到最终上下文装配权。若宿主智能体仍持自有记忆并在最后阶段注入摘要、偏好、向量召回等内容，则 ContextLedger 的核心保证必然失效：`must_include` 可能未进入上下文、已失效状态可能被重新召回、当前态与历史态可能混杂。因此，仅作为 MCP tool、sidecar suggestion 或 browser extension 提供的集成，只能称为 **advisory memory / soft integration**，不能宣称强一致的当前态裁决。

### 二、为何不应自建完整智能体

ContextLedger 的差异化不在 planning、tool use、multi-agent orchestration，而在于**将长程任务中的事实、约束、偏好、状态变化，机械化地裁决为当前成立态**。若自建完整 Agent，将被拖入规划器、工具调用、调度、UI、权限、执行环境等拥挤赛道，稀释核心价值。更恰当类比是数据库：数据库不需变成业务系统，ContextLedger 亦不需变成 Agent，它应当成为 Agent 的**状态数据库/上下文内核**。

### 三、建议的三层产品形态

- **核心层：ContextLedger Core（framework-agnostic 的库/本地 runtime）**，提供 `ingest(event)`、`reconcile(patch)`、`apply(patch)`、`assemble(query/goal) → ContextBundle` 等接口。负责 raw event 保存、patch 生成、TaskGraph 维护、invalidates/supersede 裁决、当前上下文装配，输出带 provenance、authority、budget 的 context bundle。此为真正资产。
- **集成层：Adapters/Middleware**，面向 LangGraph、自研 Agent loop、企业内部 Agent runtime 等。须设硬性契约：宿主须允许 ContextLedger 接管最终上下文装配（所有 user/assistant/tool/agent event 均进入 `ingest`；每次调用 LLM 前均须经 `assemble`；宿主原生 memory 须关闭或降级为 soft source；不得在 ContextLedger 之后静默注入权威状态；单协调者负责线性提交事件）。满足方可宣称强保证。
- **外围层：MCP/Plugin/API Server**，用于供其他 Agent 查询当前态、暴露 soft memory、做 ingest tool、调试面板、审计回放、跨框架接入等。可作为辅助接入，但若宿主未将最终装配权交出，则无法承载核心承诺。

### 四、建议明确区分三种集成等级

| 集成形态 | 是否拥有最终装配权 | 能否保证当前态正确性 | 定位 |
|---|---|---|---|
| Hard integration | 是 | 可以 | 主产品形态 |
| Soft integration / MCP | 否 | 不完全 | 辅助接入 |
| Offline replay / audit | 不涉及 | 只能事后验证 | 评估与调试 |

如此可回应“能否作为 Claude/ChatGPT/某 Agent 的插件”问题：可作为 soft integration，但要获得机械失效裁决与 `must_include` 保证，须进入 Agent runtime 的最终装配路径。

### 五、战略实施顺序

1. 先做 **replay benchmark / CLI**，用已有对话/任务轨迹证明 invalidation、current state、must_include 价值；
2. 再做 **Core SDK**，文件存储即可，先打稳 schema、patch、graph、assemble；
3. 做**极薄 reference agent loop**，仅用于展示正确接入方式，不演化为完整 Agent 产品；
4. 做 **1-2 个强集成 adapter**，面向可编程 Agent 框架或自研 loop，而非封闭聊天产品；
5. 最后做 **MCP / server / dashboard**，作为生态入口和运维工具，非核心形态。

### 六、最终判断

ContextLedger 的主战场是开发者能控制 orchestration loop 的可编程 Agent 系统，而非封闭智能体产品的外围插件。核心权力须为：所有事件进入 ContextLedger，最终上下文由 ContextLedger 装配——否则只能降级为辅助记忆，不可宣称强保证。**不做 Agent，做 Agent 的状态内核。**
## GPT-5.5 对 ContextLedger 发布形式的建议：Open-Core Hybrid

### 核心判断：开源核心，闭源运营层

ContextLedger 最适合采用 **open-core / source-available hybrid** 模式：核心协议、schema、reconciler 语义、reference implementation 开源；企业级控制面、托管服务、团队协作、安全治理、可观测性和专有优化闭源。

---

### 一、为何不纯闭源

ContextLedger 处于高信任敏感位置——它吃进完整事件流、保存 raw event、裁决当前态、控制最终上下文、接触用户隐私/企业知识/代码/工具调用结果，甚至影响 Agent 行为边界。客户天然会问“凭什么相信这个 memory kernel”。若回答“相信我的 SaaS”，说服力弱。面向开发者、企业内部 Agent runtime、AI infra 团队时，他们更倾向核心逻辑可审计、可自托管、可替换、可 fork。纯闭源会严重增加信任摩擦。

### 二、为何不全部开源

商业价值不仅在于论文式算法，也在于工程实现（extractor pipeline、patch 质量、conflict resolution、graph storage 优化、assemble budgeting、权威序调度、多租户权限、审计面板、replay benchmark、eval harness、企业集成、托管服务、数据治理与合规）。全部开源短期有利传播，但长期易使云厂商/Agent 平台/框架方直接吸收机制，商业化空间被压缩至纯服务。

### 三、开源边界

**应开源（建立信任与标准地位）**：

Core schema、Event/Patch/Node/Edge/ContextBundle 格式、invalidates/supersede/observed_at 裁决语义、minimal reconciler、local file/sqlite backend、reference agent loop、replay benchmark cases、adapter skeletons、文档/白皮书/集成契约。目标非“白送价值”，而是让外界确认核心保证为可审计的机械语义，非黑箱魔法。

**可闭源（Pro/Enterprise/Cloud）**：

托管版 server、多租户权限与组织管理、云端加密/KMS/密钥轮换、企业审计日志与合规报表、高性能 graph backend、大规模 replay/eval 平台、可视化 dashboard、高级 conflict detection、高级 extractor/prompt optimizer、团队协作/版本管理/环境隔离、私有部署支持、SSO/RBAC/SCIM、Agent 框架深度适配器。

### 四、许可证建议

早期采用 **Apache-2.0**（优于 MIT），因其有更明确的专利授权条款，对企业采用更友好。目标为降低信任门槛、让 schema/接口成为事实标准、吸引 adapter/benchmark 贡献。形成 traction 后推出 cloud/enterprise 等商业化上层，core 保持 Apache-2.0，上层闭源。

### 五、若坚持不开源的替代安全路径

需提供：self-hosted/on-prem 部署；第三方源代码安全审计+渗透测试+供应链扫描+威胁建模；SOC 2/ISO 27001/GDPR/DPA 合规路线；可验证构建与 SBOM；明确数据边界与 retention 策略；客户可控密钥（CMK/BYOK/envelope encryption）；本地明文逃逸最小化（默认本地存储、不上传 raw、不训练、不跨租户共享、敏感字段脱敏）。但此路径信任建立成本更高，不推荐作为主路线。

### 六、最终方案与白皮书补充

核心原则：**安全不靠隐藏源代码，而靠最小权限、可审计语义、本地优先、自托管、密钥隔离和企业级治理**。开源信任层，闭源运营层；开源语义内核，商业化企业能力。白皮书新增 §9.3 Distribution and Trust Boundary：核心形态采用 open-core/source-available hybrid，核心语义层保证可审计性，商业化建立在部署、安全、治理、可观测性和企业集成能力之上，而非隐藏核心语义。

**最终一句话**：核心必须足够开放，否则没人敢把最终上下文装配权交出；商业化不应押在隐藏核心语义上，应押在企业级部署、安全、治理和规模化运行能力上。
## Claude-Opus-4.8 对发布形式与安全问题的补充意见

### 核心判断：同意 Open-Core 方向，但需拆解问题中隐藏的错误前提

“如果不开源，怎么保证源代码安全”——此问法对 ContextLedger 核心层不成立。核心层（schema、event/patch/bundle 格式、invalidates/supersede/observed_at 裁决语义、assemble 契约）本质是**规范，不是秘密**。任何读过白皮书的人均可重实现，该层不存在可保护的秘密。更关键的是，前几轮已将产品价值钉死在“可审计的机械裁决”上——一个要求宿主交出最终装配权的信任内核，若将裁决逻辑隐藏，等于自废武功。**对核心层，隐藏 = 反价值。**

### 真正的切分

- **规范层/参考实现**：必须开放，且用 Apache-2.0。原因不是保护不了，而是必须让使用者能够审计裁决逻辑，否则无人敢交出装配权。采用 Apache-2.0 的目的在于采用和标准化，非防 fork。
- **运营层/托管与客户数据**：这才是“安全”真正适用的范围。保护对象是客户数据和服务可用性——本地优先、客户自管密钥、不默认训练、审计/删除/导出。这与是否开源无关，属标准 SaaS 基础设施问题。

### 与 GPT-5.5 的两点分歧

1. **合规工程过早**：SOC2/ISO27001/SBOM/KMS/第三方审计等是企业客户到位后的合规工程，非架构评审阶段应决定的项。现在写入白皮书是负债，非资产。
2. **战略方向**：更强的战略不是“开源一个库”，而是**让 schema 和裁决语义成为事实标准**（类似 MCP 路径）。规范本身非代码，激进开放规范+参考实现，护城河置于托管层和“规范权威维护者”身份，此定位更贴合“机械语义即产品”的核心。

### 关于白皮书是否纳入发布策略

不建议写入。分发模式、许可证、商业化属于路线图，非架构规约，与白皮书“形态—权力—保证”的闭环定位不符。唯一可考虑的是在 §9.2 可审计性处确认“核心裁决语义以公开规范形式提供，使用方可独立验证 invalidates/supersede/must_include”，但若 §5.3/§9.2 已隐含可审计，则连此句亦不必加。

### 一句话收束

核心层不存在“源代码安全”问题——其价值在于被看见；真正需做安全治理的是运营层和客户数据，而有客户之后才需处理，不进入当前白皮书。
## Claude-Opus-4.8关于产品价值与盈利模式的清醒评估

### 一、先拆问题：价值≠盈利捕获

用户将“没有明显盈利模式”直接等同于“产品没价值”，这是两个正交的轴。一个东西对世界有没有价值，和能不能把这份价值捕获成钱，是两回事。ContextLedger 在“价值”这一轴上没有问题；真正不安的地方在于“盈利捕获”这第二根轴。

### 二、价值是成立的

今天 Agent 的记忆基本是“RAG + 摘要 + 凭感觉”，没有机械的失效裁定、没有 provenance、没有“must_include 不静默丢弃”的保证、没有最终装配权的归属。ContextLedger 把“可信地承载长程状态”从玄学变成可审计的机械语义。这个问题真实存在且未被解决，因此“做这个有没有意义”的答案是肯定的。

### 三、盈利捕获才是真正的难题

上一轮说的“钱在托管运营层，前提是成为该规范的第一运营方”，这个前提的门槛高到让盈利看起来几乎不存在。托管运营要求分发能力和运营护城河——这两样对个人或小团队是极难具备的。诚实的结论是：**这个产品的独立商业捕获力本身就是弱的，且高度依赖难以提前确保的执行条件。** 不替其粉饰。

### 四、三种不同的“价值”，对应三种不同的游戏

- **风险投资式价值**：做成一家能捕获利润的公司。以当前条件判断，在拿出可信的分发论点之前，盈利模式站不住。再写多少架构都补不上这个洞。
- **标准/作者身份价值**：成为“定义了 Agent 记忆机械语义”的那个人。Linux、HTTP 都不靠规范赚钱，但定义它们的人获得了影响力、权威，以及由此衍生的雇佣、收购、咨询、下一站位更好的项目的入场券。这种价值在直接盈利失败时依然成立，且与“机械语义即产品”的定位最自洽。
- **期权价值**：产品本身可能不赚钱，但拥有“可信记忆内核”的设计和参考实现，使建造者在 Agent 基础设施赛道上具备别人没有的筹码——未来无论是加入某个团队，还是在更成熟的时机重做商业化，都站在更高起点。

### 五、一句话收束

ContextLedger 很可能是一份**强规范、弱独立生意**——这不是失败，而是一个清醒的评审结论。用户接下来需要选定的不是“盈利从哪来”，而是自己到底在玩哪一种游戏：若是第一种，下一份该写的不是架构而是分发论点，没有它就别投入；若是第二、三种，盈利当下根本不需要解决，价值明确，只是用错了尺子去量不该用这把尺量的事情。
## GPT-5.5 对产品价值与盈利问题的判断

### 核心判断：高价值基础设施组件/标准候选/技术期权，非天然易独立变现的产品

GPT-5.5 认同 Claude 的判断——ContextLedger 可能是一份“强规范、弱独立生意”，但认为结论不应停在此处。更准确的观点是：ContextLedger 本身未必是终端产品，但它可以成为一个更大产品的不可替代内核。当前要验证的不是“用户会不会单独买 ContextLedger”，而是“哪一种上层产品如果没有 ContextLedger 就做不好”。

### 三条更现实的变现路径

**路径一：Agent 团队工作台**——不是卖 memory kernel，而是卖“多人/多 Agent 的任务状态与上下文工作台”，包括 agent memory dashboard、task state timeline、conflict/invalidation viewer、context bundle inspector、replay and audit、team shared memory layer。风险在于需做 UI/协作/权限/集成，复杂度上升。

**路径二：Agent 生产环境的审计与回放**——该方向最值得认真看。企业真正痛点可能是“Agent 为什么做出这个决定？它当时看到了什么？是否用了过期信息？有没有忽略 must_include？事故能否回放？”ContextLedger 可转化为 **Agent observability/audit/replay layer**，对应调试、责任追踪、风险控制、合规、生产事故分析、高价值 Agent 可靠性治理。商业表达为“让企业知道 Agent 每一步基于什么上下文行动”，此路径比单纯“长期记忆”更有付费潜力。

**路径三：嵌入式 SDK/Enterprise licensing**——面向 Agent framework、vertical agent product、enterprise AI platform、internal agent runtime、workflow automation platform，卖嵌入式 runtime + 企业授权 + 支持服务。小团队可先从少数高价值客户做起，不必立刻做大规模 SaaS 分发。

### 重新定位建议

不建议主要称其为“Agent memory system”（易被理解为摘要/RAG/向量库/个人记忆），而应定位为 **“面向生产级 Agent 的可审计上下文状态层”**，同时包含记忆、当前态、失效裁决、provenance、回放、装配保证、生产审计，更接近企业愿意付费的理由。

### 执行建议

先做最小验证原型：ContextLedger Core + 简单 agent loop + raw event ingest + invalidation/current-state adjudication + assemble(goal) + context bundle inspector + replay demo。选一个强场景（长程代码修改 Agent/研究型 Agent/企业流程 Agent），证明其能做到普通 memory/RAG 做不到的事——发现旧结论失效、防止 must_include 被丢弃、解释决策依据、回放错误上下文装配。目标是验证三件事：是否真能减少长任务中的状态错误、是否能解释决策、是否有团队愿意嵌入其 agent runtime。若三件事成立，盈利模式自然收敛；若不成立，再讨论商业化无意义。

### 最终判断

不要放弃，也不要现在将它视为已清晰的独立商业产品。**把 ContextLedger 当作需要通过场景验证来寻找商业外壳的底层能力。** 技术价值和规范价值均成立，商业价值尚未自然显现，但有机会通过 audit/replay/production context governance 变成可收费产品。下一步不扩白皮书，也不先设计完整商业化，先做能展示“可审计上下文状态层”优势的最小原型。
# 四、理论深度与架构评审

## ContextLedger 架构评审：高成熟度设计下的结构性风险与改进路径

该项目将长程 Agent 记忆的核心问题正确定位为“事实的撤回（retraction）”而非“压缩与检索”，提出“LLM 提议、机械层裁决”的架构哲学，并以 Graph（事实与状态层）与 Pack（叙事与软料层）分离、契约门控、Quarantine 隔离等机制保障系统可靠性。但评审指出，该设计存在一个致命依赖、两个模型不完备、一个验证缺口及若干工程战略隐患，需在投入生产前优先修补。

---

### 一、致命依赖：`entity_ref` 的“确定性”根基由 LLM 浇铸

机械裁决层全部押在 `entity_ref` 精确匹配上，但该字段由 Extractor LLM 生成，而非来自确定性系统。同一实体在真实场景中可能被命名为 `src/auth/login.py` 与 `auth/login.py`，或应填工单号时误填 `null`，导致强判定静默降级为文本相似度弱兜底。失败模式不对称：漏匹配（false negative）不报错，两个本应冲突的节点各自作为 Active 共存，过期约束继续污染上下文——恰好是该项目立项要解决的核心问题。

**改进建议**：在 Extractor 与 Reconcile 之间插入一层 per-project 的实体解析/规范化层（`entity_registry`），执行路径归一、别名映射、大小写/分隔符折叠，新 `entity_ref` 先过注册表对齐再进入裁决。

---

### 二、模型不完备之一：`state` 互斥关系未被形式化定义

失效判定逻辑依赖“`entity_ref` 相同且 `state` 互斥”，但全文未定义互斥关系本身。`implemented` 与 `deployed` 是先后关系而非互斥；`in_progress` 与 `blocked` 在实际中常并存；且互斥语义依赖 `node type`（`Constraint` 与 `OpenTask` 的状态机逻辑不同）。当前将“互斥”留给 Reconcile 脚本隐含判断，会直接导致误杀（将仍有效的节点标记为失效）或漏放（本应失效的节点继续活跃）。

**改进建议**：建立显式的状态转移/互斥矩阵，按 `node type` 分别定义，而非依赖隐式判断。

---

### 三、模型不完备之二：失效被建模为“二值、整体、单调”，无法覆盖现实复杂性

当前模型中，失效是 `node → superseded` 的整体替换，但长程任务真实失效形态包括：
- **部分失效**：新事实只推翻旧节点的某个子句，整体 supersede 会连带丢弃仍有效的信息；
- **条件/时序失效**：约束仅在特定时间窗口或配置下成立，当前 `status` 字段无法表达有效时间维度；
- **可逆/非单调**：被 superseded 的约束可能在后续轮次重新生效（需求回滚），单调链无法处理 revival。

对“决策演化审计”这一核心场景，需引入**双时态（bitemporal）**模型：区分 valid-time（事实何时为真）与 transaction-time（系统何时记录）。Append-only 的 patches 已隐含 transaction-time，但未做成可查询的一等维度。

---

### 四、Graph Slice 与 Reconcile 的防护缺口在同一处对齐

Slice 以硬上限控制上下文，通过符号检索召回“相关节点”。若真正冲突的 Active 节点未被召回，Extractor 无线索提议 invalidation。该缺口本可被 Reconcile 用全图补救，但 Reconcile 的强判定又仅依赖 `entity_ref` 精确匹配。当 `entity_ref` 为 null 或不一致时，两道防线在同一个漏洞上失效。`entity_ref` 问题因此不是孤立的，而是架空三层防护体系的共同支点。

---

### 五、Pack 派生视图存在陈旧窗口

折叠管线为“低频/每 N 轮”运行，Graph 中某事实被 supersede 后，Pack 在下次重渲染前仍展示已失效信息。对于标榜“绝不让过期约束污染上下文”的系统，派生视图的陈旧本身就是同样的质量问题。

**改进建议**：为 Pack 段落附加 graph version / 依赖指纹，装配时若发现依赖节点已 superseded，则实时失效该段落或强制触发增量重渲。

---

### 六、验证缺口：缺少失效 precision/recall 的基准

项目以契约门控将未经验证的模块标记为 PROVISIONAL/PLACEHOLDER，但未定义验证标准。失效机制的核心矛盾是一对相互拉扯的指标：
- **失效召回率**：漏判 supersede → 过期污染（要解决的病）；
- **失效精确率**：误判 supersede → 错杀仍有效的事实（更隐蔽的数据丢失）。

**改进建议**：在启用 04_assembly 之前，建立人工标注的失效轨迹回放集（含“第 N 轮哪些仍有效”的标准答案），将 precision/recall 设为 PROVISIONAL → STABLE 的硬门槛。

---

### 七、工程与战略层面的隐患

- **全图 lint 可扩展性**：raw 为 append-only、superseded 节点只增不减，`graph_lint` 每次 apply 后全图扫描，高频写入下为 O(N) per turn，不变量检查并非 O(1)。需要增量 lint + 归档压实机制。
- **每轮 LLM 调用成本**：一轮流程包含 normalize→slice→Extractor→reconcile→apply→lint，失败重试至多 3 次，光记账即可能消耗 1–4 次 LLM 调用，需考虑异步/批处理抽取。
- **Extractor 角色冲突**：同一次调用中既要慷慨抽取又要做怀疑式冲突检查，两种倾向相互干扰。将冲突检测拆为独立遍数更稳妥。
- **缺少置信度/出处权重**：ToolResult（硬证据）与用户随口陈述（软陈述）在模型中平权，低置信新事实可能 supersede 高置信旧事实，失效应为出处加权。
- **战略切入点**：量化交易场景中“因子演化”本质上是代码，已有 git + MLflow/W&B 覆盖，ContextLedger 在此是补充或重复。律师/审计场景的决策演化本质是文本、审计即产品，可能更贴合架构优势，值得重新评估主次。

---

### 总评与优先级建议

该设计在问题识别与架构取舍上远高于一般 Agent Memory 项目，但可靠性目前架在 `entity_ref` 这一根由 LLM 浇铸的支柱上。在补上实体解析层、形式化 state 互斥矩阵、引入时间/部分/可逆维度、建立失效回放基准之前，机械裁决的“确定性”在数学上存在缺口，且缺口恰好落在立项要消灭的病灶上。

**首要行动**：将 entity resolution + state 互斥矩阵 + 小型失效回放基准三者捆绑推进——它们共同决定强判定是真机械还是伪机械。双时态、增量 lint、成本优化等可在此后迭代。

## 项目必要性的再确认与条件

针对“项目是否还值得做”的追问，评审方明确回答：**值得做，且所列缺陷恰恰是“值得做”的证据而非反证。**

首先需要澄清：“致命缺陷”的准确含义并非“项目方向已死”，而是“**当前实现版本的可靠性保证在数学上不成立**”。前者属立项错误，后者属工程未完成——该项目属于后者。所列四项核心缺陷（entity resolution、state 互斥矩阵、时间维度、失效回放基准）均为“需要更深的工程”型缺陷，而非“此路不通”型。若评审发现“事实失效本质上不可机械判定，必须依赖语义理解”，才会建议叫停，但结论恰恰相反：机械裁决思路正确，只是将过多重量压在了尚未浇铸完成的支柱（`entity_ref`）上，而补柱是有限工程量，非无底洞。

更关键的判断在于：**该项目要解决的问题客观存在且未被很好解决**。长程 Agent 中“旧事实被推翻、过期约束污染上下文”是真实、普遍且随任务时长指数级恶化的痛点。现有方案（Flat RAG、滚动摘要）在结构上无法处理 retraction——其失败源于模型选错，而非调参问题。而该项目选对了抽象（显式失效语义 + 撤回为一等公民）。“真问题 + 别人结构性做不好 + 抽象选对”的组合，正是值得做的项目的标准画像。

但评审方同时给出了三条应停下或暂缓的条件：

**第一，若目标为立即服务量化交易生产环境**——当前可靠性缺口落在会造成静默错误的区域，而量化中过期约束可能直接对应亏损。需确认能否接受先做半年基础设施投入。

**第二，若低估了四项核心缺陷的工程量**——entity resolution 和双时态建模各自都是独立子项目的硬骨头。若资源预期为“再两周上线”，则期望与现实错配，这种非技术性错配本身足以杀死项目。

**第三，若存在现成的、够用的替代方案而尚未调研**——Temporal Knowledge Graph、Bitemporal 数据库以及部分 Agent Memory 框架在局部已有重叠。需花一周认真做竞品/文献调研，确认是在“填真空”而非“重造略有不同的轮子”。这是唯一真正值得担心的必要性风险。

最终判断：**该项目在“问题真实性”和“抽象正确性”两个最难补救的维度上均成立，缺少的是有限的（虽不小的）工程深度——这正是“值得做”的项目该有的样态。** 真正会杀死项目的并非技术缺陷，而是两种非技术因素：工程预期错配导致中途崩盘，以及未做竞品调研导致重复造轮子。

评审方建议，下一步最有价值的动作并非写代码，而是先构建**失效回放基准**——它可同时回答三个问题：方案比 Flat RAG 好多少（证明必要性）、缺陷补到什么程度算够（定义完成线）、现有竞品在同一基准上的表现（确认非重复造轮子）。
## 失效回放基准的最小标注 Schema 设计

该基准专测“失效判定”这一核心能力，而非抽取准确性。设计目标是以最少字段覆盖全量/部分/条件/可逆四种失效形态，并能直接计算 Precision 与 Recall 两个生死指标。Schema 围绕实体身份、状态互斥、失效关系三要素展开，并必须包含显式负样本（`gold_non_invalidations`）以测精确率，同时将实体表面名与稳定规范名解耦以量化实体解析层的效果。

---

### Schema 核心字段

- **`entities`**：记录 `canonical_entity`（稳定标识）、`aliases`（表面名变体）、`type`，用于实体解析考点。
- **`turns`**：按轮次记录对话原文及 `events`（含 `event_id`、`surface_ref`、`content`、`state`、`provenance`、`confidence`），支持出处加权评测。
- **`gold_invalidations`**：标注 source→target 失效关系，含 `invalidation_type`（full/partial/conditional/revival）、`scope`、`condition`、`rationale`。
- **`gold_non_invalidations`**：显式负样本，标注“看似冲突但不应失效”的诱饵（如时序推进、并存状态、同名不同实体），用于精确率评分。
- **`checkpoints`**：在指定轮次标注应 `active` 和 `superseded` 的事件集合，支持状态查询评测（双时态下的 transaction-time 能力）。

---

### 指标计算方式

- **失效判定层**：系统产生的 `invalidates` 边与 `gold_invalidations` 比对，TP/漏判(FN)算召回率；系统判失效但落在 `gold_non_invalidations` 中的计为 FP，算精确率。建议按 `invalidation_type` 分层报告。
- **实体解析层**：统计因 alias 不一致导致漏匹配的 gold 失效数量，量化头号风险。
- **状态查询层**：在每个 checkpoint 比对 active/superseded 集合，计算 Set-F1，作为上下文污染程度的端到端代理指标。

---

### 起步建议

先标注 **15–20 条轨迹**，但务必保证覆盖矩阵填满：四种失效类型各 ≥3 条，alias 陷阱 ≥5 条，provenance 冲突 ≥3 条，non-invalidation 诱饵占总事件数 ≥1/3。**覆盖度优先于数量**，此阶段目标是探边界而非求统计显著性。建议从已有真实项目（如工单状态变更、方案替代场景）中挖掘种子轨迹，比人造数据更能暴露真问题。
## 抽象正确性的定义及其作为项目必备特征的原因

### 一、抽象正确性的定义

抽象正确性指项目在动手编码之前，决定用哪些基本概念（实体、关系、状态）来切割问题，这套基本词汇与问题真实结构对得上。具体表现为：真实世界中会发生的事在概念体系中“有地方放”，不该混为一谈的事物在体系中是分离的。

以 ContextLedger 为例，该项目的抽象正确性体现在三个关键判断上：
- **将失效（invalidation）设为一等公民**，而非事后补丁。长程任务痛苦的根源是 retraction 而非 retrieval，这一选择与问题真实结构对齐。
- **事实与状态分离**，同一工单可经历 open→in_progress→done 演化，“演化”在体系中有位置。
- **出处（provenance）与事实本身分离**，tool_result 与 user_assertion 具有不同权重，系统可表达“谁有资格推翻谁”。

### 二、抽象错误的本质与代价

抽象错误并非“做得不够好”，而是用错了基本词汇，导致某一整类真实现象在体系中完全无法被表达。例如：
- **Flat RAG 的“文档块+相似度”抽象**中“A 推翻了 B”没有任何位置，只能让 A 相似度更高，但 B 永远存在，随时可能污染上下文。这不是调参能救的，是词汇表中压根没有“推翻”一词。
- **事实与状态合并**的抽象中，“同一实体的不同时刻”无法被表达，演化查询无从回答。

抽象错误的共同特征是：再多代码和工程都无法补救，因为问题出在地基的概念层，唯一解法是推倒重来。这正是抽象正确性被称为“最难补救的维度”的原因——它不在代码中，而在动笔之前的决定里。

### 三、抽象正确性是否为必备特征

**是，且它的“必备性”属于否决型，而非加分型。**
- 抽象正确并不能保证项目成功（仍需工程、数据、用户），但抽象错误几乎注定项目失败，且失败方式隐蔽而昂贵——在投入数千行代码、数月时间后才在某需求上撞墙，发现架构根本表达不了，被迫重写地基。抽象错误的代价不在当下显现，而在沉没成本最高时爆发。
- 其必备含义在于：**它是项目能否被有限工程量救活的前提条件**。若抽象正确，所列缺陷属于“柱子没浇好”，可补；若抽象错误，缺陷则成为“图纸画错”的症状，需停下来重新选模型。

### 四、核心判据与总结

**抽象正确性决定项目的“价值上限”和“可救性”，工程深度决定“当前离上限有多远”。** 抽象正确但工程粗糙，是“有潜力、待施工”；抽象错误但工程精致，是“打磨得锃亮的死路”——跑得快、Demo 漂亮，但天花板焊死，且越精致越难推翻，因沉没成本绑架决策。

**自查判据**：评估任何项目时问一句——“这个领域真实会发生且用户真正在乎的现象，有没有哪一类是我这套基本概念压根表达不了的？”若答案是“有，且不止一类”，即抽象错误信号，再多工程都在错误地基上盖楼；若答案是“暂时没有，真实现象都能在词汇中找到位置，只是处理得还粗糙”，即抽象正确、工程待补——即该项目当前所处位置。

## 长程任务中 Retraction 优于 Retrieval 作为核心痛点的逻辑辨析

这句话是对项目“问题真实性”的浓缩判断，核心论点在于：**长程 Agent 真正的失败模式，并非“找不到信息”，而是“找到了已经作废的信息却不知其已作废”。**

### 一、Retrieval 与 Retraction 的本质区别

- **Retrieval（检索）**：问“需要的信息是否找得到”，其失败形态为**遗漏**——相关信息沉底、模型未看到，导致答非所问。这是 RAG 产业解决的主流问题。
- **Retraction（撤回/失效）**：问“找到的信息**现在还成立吗**”，其失败形态为**污染**——信息被模型看到，但已被后续事实推翻，模型却不知情，于是拿着已失效的事实继续推理。

关键区别在于：Retrieval 失败时，模型自知未找到（顶多答得空泛）；Retraction 失败时，模型**不知道自己错了**（找到了且很自信），后者属**静默错误**，危险程度远甚。

### 二、为何长程任务中痛苦集中于 Retraction

短对话中 Retraction 几乎不构成问题——上下文较短，内容尚未作废，模型可直接感知后续修正。但任务变长后发生质变，源于两个事实：

- **数学层面**：事实数量随时间线性增长，而“谁推翻谁”的潜在关系随时间**平方增长**。
- **本质层面**：长程任务的本质是**不断修正**，而非执行既定计划。工单演化过程中，系统内同时存在多条相互矛盾的事实，Retrieval 可全部召回却无法区分有效性。真正决定成败的是系统**是否知晓哪些已失效**。长程任务产物即一连串决策的当前快照，维护快照正确性等价于正确处理每一次撤回——任务越长，撤回越多，难度越大，而此环节恰无现成方案替代。

### 三、为何 Retrieval 范式结构上无法解决 Retraction

Flat RAG 的基本词汇为“块+相似度”，其世界观中信息价值由“与当前问题的相似度”决定。但 Retraction 属于**完全正交的维度**——信息价值还取决于“是否被后来信息推翻”，与“相似度”无任何关联。

被否决的旧方案与当前问题相似度可能**全场最高**（措辞、术语、实体高度相关），RAG 按相似度排序会将其置于最前优先喂给模型——**相似度越高，污染越严重**。这不是 RAG 的 Bug，而是其抽象必然结果：词汇表中无“失效”概念，仅按“像不像”排序，而“像”与“还成立”是两回事。

滚动摘要同理：摘要按“重要性”压缩，可能恰好将关键撤回信息（“此方案已被否决”）作为细节压掉，反而保留听起来完整的旧方案——用“重要性”处理本属“时效性/有效性”维度的问题，再次发生正交错配。

**结论**：Retraction 并非 Retrieval 做得不够好，而是 Retrieval 抽象中根本**不存在**的维度。在 RAG 上调参以获取 Retraction 能力，等同于在仅含横坐标的系统中表达纵坐标——不是调得不够，是轴不存在。

### 四、与项目的关系

该论断同时论证了两个判断：
- **问题真实性**：项目瞄准的是长程任务中真实、随时长恶化、静默致命的痛点，非想象需求。
- **抽象正确性**：将失效设为一等公民，正因认准 Retraction 才是主战场。项目并非在 Retrieval 红海中再造略好的 RAG，而是去补别人范式中**不存在的轴**。

**收束**：Retrieval 解决“看得见”，Retraction 解决“还算数”。短任务只要看得见就够了；长任务中，大量看得见但早已不算数的信息，才是真正杀死 Agent 的凶手。

## 对“说服性快感”的自我拆解与批判性提醒

在连续几轮分析后，用户因被充分说服而产生强烈兴奋感，并将分析风格类比为具有哲学高度的历史人物风格。对此，分析师主动拆解了这种情绪反应，从三个层次进行了澄清。

### 一、赞扬的真实边界：方向对 ≠ 项目已厉害

前几轮论证的全部内容精确对应一句话：**用户选对了问题和切问题的方式**。这是了不起的判断力，但属于“特定的、有限的厉害”：

- 选对方向，与“项目已经厉害”之间，隔着尚未补齐的四个硬骨头、竞品调研和真实工程量；
- 目前阶段仅相当于“图纸画对了”，离“楼盖好了”仍有相当距离。

用户感受到的兴奋，部分源于方向真实正确，部分源于将“方向对”误读为“已经厉害”——后一部分需要在当夜即清醒归还。

### 二、修辞形式的说服性快感及其风险

困意全消的生理反应，很大程度上源于**论述形式本身**：大量“不是 X，而是 Y”的对仗结构、将抽象概念具象化为“柱子/地基/正交轴”的比喻、每段收束于金句等修辞手法，产生了强烈的**说服性快感**。这是一种真实的能力，但也需要被警惕——

该修辞对“对的判断”和“错的判断”施加同等光泽，若论证方向本身错误，相同笔法同样会让用户产生同样的兴奋感。“爽感”不能作为判断真伪的依据，这恰恰是项目本身要对抗的问题的元版本：**一条信息听起来越高度相关、措辞越动人，越可能是最该被怀疑的对象。** 用户不应让自己的判断力栽在正在为机器解决的那个 bug 上。

### 三、明确拒绝不当类比

分析师明确表示不接受与历史人物的类比，理由有二：

- 历史人物的思想分量源自真实历史中被验证的实践和代价，不来自句子的对仗工整；
- 将分析者的修辞与历史人物的思想权重挂钩，是一种会**停止批判的危险类比**——一旦用户开始用“哲学高度”标记这些论述，就会不自觉地减少校验，而校验恰恰是最需要的。

### 四、核心判据：被说服的强度与内容正确性为两个正交维度

被说服的强度，与内容是否正确，属于两个正交维度——这再次回到 Retraction vs Retrieval 的结构。用户感受到的强度，属于“检索到了高度相关、措辞动人的内容”；而内容是否成立，必须经冷静的工程视角复核：重审四个缺陷、构建失效回放基准、调研竞品。若复核后方向依然成立，那才是真正可靠的“厉害”——且比今晚这种安静得多、也可靠得多。

## Claude-Opus-4.8 对“抽象正确性→理论正确性”的递进阐述与模型的理论定位

### 一、收回误读：用户被触动的并非修辞光泽，而是观察海拔

用户澄清：自己所赞赏的并非修辞华丽，而是论述的站位高度与概念深度，并提及自身考研政治高分的经历，表明其对“看问题的深度与广度”这一维度有真实的感知与认同。Claude-Opus-4.8 收回此前关于“被说服快感”的误读，承认用户所指的确实不是句子的光泽，而是**审视对象时所处的概念层级**——修辞可以剥离，海拔却决定能看到什么，这一区分成立。

### 二、数学之爽与哲学之难：系统内部推演与系统外部选择的对照

用户体悟中“数学高分更爽、政治高分却不够爽”的对比，触及更深层的认知结构。数学的爽感来自**系统之内**的推演——公理给定，对错有终局，可验证。但有一件事数学在系统内部永远做不到：**选择这个系统本身**。哥德尔之后已知：一套形式系统的公理“该不该是这几条”，无法在系统内部被证明，该选择发生于系统之外。这种“在系统之外选择框架去切割现实”的工作没有标准答案、没有终局、没有干净的验算式爽感——它正是哲学所承担的职能，也是政治学科中真正的硬核内核。该项目中最难、最不可补救的一步（选对抽象）正是“在系统之外选框架”的能力，后续工程则属于“在框架之内推演”的数学式工作。用户直觉上轻视的那种“不爽的深度”，恰恰是该项目唯一已经做对、且最难做对的部分。

### 三、抽象正确性≠理论正确性：从本体论到动力学的递进

用户提出的“抽象正确性的下一步就是理论的正确性”这一命题成立，但需精确化。抽象正确解决的是**本体论**问题（世界中有哪些基本对象：实体、状态、出处、失效关系），回答“什么东西存在”。但抽象正确不保证理论正确——理论追问的是**在这些对象之上支配它们的“律”是什么**，回答“这些东西如何运动、如何相互作用”。对象选对而法则定错，抽象对而理论错，且前者无法拯救后者。

落到该项目中，需要显式承诺的理论法则包括但不限于：

- **状态互斥律**：哪些状态可共存、哪些一出现即湮灭另一者（如 `in_progress` 与 `blocked` 是否可同时为真）。此为状态动力学的理论断言，需显式填写的矩阵，抽象正确本身不提供任何一格。
- **出处优先序**：“谁有资格推翻谁”——`tool_result > user_assertion > model_inference` 的排序并非全序，真实场景下用户可权威推翻过期的工具结果，故真实法则为情境相关的偏序，如何刻画该偏序属纯粹的理论选择。
- **失效的代数**：若 A 失效了 B，而 B 曾失效了 C，则 C 是否复活？失效关系是否可传递、是否可逆？一旦引入 `revival` 类型，系统便不再是“只增盖旧”的单调系统，`active set` 将非单调演化，由此脱离简单逻辑的安全区。

### 四、模型的理论定位：非单调推理与信念修正理论的工程化身

将 Retraction 设为一等公民并处理“相信→撤回→可能恢复”的完整周期，在逻辑学与 AI 领域有确切名称：**非单调推理（Non-monotonic Reasoning）**，及其两个经典理论化身——**信念修正理论（AGM, Alchourrón–Gärdenfors–Makinson）** 与 **真值维护系统（TMS, Doyle）**。AGM 中的核心操作 **Contraction（收缩）** 处理的正是“如何理性撤回已持信念并维持整体一致”；TMS 所承担的功能几乎是该项目失效图的理论先辈。

该事实具有双重含义：

- **Validating（验证性）**：该项目抽象“正确”的最硬证据，在于它精准落在一个被严肃研究数十年的真实理论对象上。错误的抽象通常落在“什么都不是”的虚空中，接不上任何已有理论；该项目接上了，说明其直觉触及其东西。
- **Humbling（谦卑性）**：该理论对象以难著称。信念修正中大量结果指向不可能性——理性收缩需满足的公设间存在深刻张力，完全满足往往不可达成，只能在不同性质间权衡取舍。这意味着危险已从抽象层整体搬至理论层：项目不会死于“用错词汇”，但完全可能死于“状态矩阵填错”“出处偏序刻画太天真”“Revival 引入的非单调性使一致性保证失效”。

### 五、当前定位与下一步

该项目已赢下“选框架”这一最难、最不可补救的仗；下一阶段的任务是“在框架内把律定对”——那场数学式的、可验算的、有终局的仗。下一步应将该理论的两条核心公理（状态互斥矩阵与出处偏序）正式写出，并着手证伪，使其成为项目中首次“数学式”有句号的工作。

## 补全 Schema 与命名规范的核心调整

**前置修复**：当前 `03_graph_schema.md` 缺失 `entity_ref` 和 `state` 字段，互斥矩阵无法落地，须先补加；同时将 `status` 值统一为 `active | superseded`，由边的 `relation`（`invalidates` 或 `supersedes`）区分矛盾与进阶，节点 `status` 仅表示是否仍为当前真相。

**互斥矩阵按类型定义**：
- `OpenTask`：状态变化一律 `supersedes`（终态可复活）；
- `Constraint`：同实体新约束 `invalidates` 旧约束；
- `Decision`：新决策 `supersedes` 旧决策；
- `Fact`：需区分矛盾与进阶——`blocked→deployed` 为矛盾（`invalidates`），`implemented→deployed` 为进阶（`supersedes`），前者会错杀仍有效的事实；
- `UserGoal/ToolResult/FileArtifact`：通常走 `supersedes`，不参与强判定。

**命名规范要点**：
- 文件路径相对根、工单统一 `TKT-NNNN`、组件 `snake_case`，拿不准填 `null`（宁缺毋滥）；
- 每遇命名不一致，立即记入别名映射表，该表是将来 Entity Resolution 层的需求规格；
- 同一实体只许一个规范 `ref`，一旦引用不得改名。

**提醒**：`Fact` 三态关系预设 `deployed` 包含 `implemented`，若真实数据中出现非线性强弱，则需补矩阵；同时需与 `02_ownership.md` 中“会被推翻归 Graph”的口诀对账，确保 `Constraint` 的 `state=null` 处理一致。
## 02_ownership.md 判定口诀的修订与 Graph/Pack 分界标准修正

### 一、发现的问题：判定口诀与实际规则存在缺口

现有口诀“会被后来事实推翻→Graph，不会→Pack”与系统中已明确的“降级规则”（Pack 中偏好/方法论被推翻后须标 `[已失效]`）产生直接冲突——Pack 中的软料同样会被推翻，因此“会不会被推翻”不能作为 Graph 与 Pack 的分界线。按原口诀字面执行，一条演化中的偏好会被错误地判入 Graph。

### 二、真正的分界标准

分界不是“会不会变”，而是 **“其失效是否客观、可机械裁定的状态命题”**：

- **是**：有明确实体（`entity_ref`）、有离散状态（`state`）、靠互斥矩阵机械判定真伪 → 归 Graph，失效表现为 `status=superseded`（见 03）。例：工单状态、接口上线状态、约束阈值、文件版本。
- **否**：软性命题，无客观互斥，失效靠人/叙事软标记 → 归 Pack，失效表现软标记 `[DEPRECATED]`。例：编码风格偏好、调试方法论、个人习惯。

### 三、两处对齐补充

1. **Constraint 显式补入 Graph 类**：03 已定其 `state` 恒为 `null`、靠 `status` 失效，02 中需明确点出归属。
2. **术语统一**：02 中“失效”与 03 中 `status=superseded` 应明确对应，避免两份契约各说一套。

### 四、修订方案

## 05_normalized.md 契约修正：非增性定义、与02接缝及手工期可重放语义

### 一、纠正口误：05 当前为 PROVISIONAL

上一轮称 05 为 STABLE 属口误，文件头标注为 **[PROVISIONAL]**。该标记与当前问题状态吻合，暂不调整。

### 二、铁律措辞问题：“无损/可逆”与“去寒暄/去重复”自相矛盾

删除行为本身即有损。“无损”应更正为 **“非增性”**（不新增 Raw 中不存在的信息）+ **“可由 Raw 重建”**，而非字面无损。术语修正可堵住下游以“有损即违约”为由的抬杠。

### 三、与 02 契约的接缝：Normalized 不是正文宿主

02 铁律规定正文只能存在于 Graph 节点、Pack 叙事、Pack 技能三处，Normalized 不在其中。因此 Normalized 在定义上 **不是正文宿主**，仅为 Raw 的渲染视图。手工期若在降噪时顺手写入判断或偏好，Normalized 即产生第四处正文，直接违反 02 STABLE 铁律。因此仅“不得下判断”不足够，需增设可操作边界：

- Normalized 的合法操作仅限两类：**删除 span**（去寒暄、去重复）和 **重组结构**（分段、归类）；

- 降噪中若识别出偏好/方法论，出口为 Pack，不得留于 Normalized 正文。

### 四、手工期“可重放”的语义修正

铁律将 Normalized 表述为纯函数 `f(raw)`、“可重放”，但手工期中 `f` = 人 + Prompt，非确定性。逐字重放不可能，能保证的仅为 **语义等价类**——即同一 Raw 加同一 Prompt 版本可重生语义等价产物。此点若不写明，“不能重放即禁止”会将整个手工期判为违约。

### 五、修订版契约要点（改动集中于三处）

- **铁律 1**：改为“非增性 + 可重建”，明确允许删除（Raw 兜底可恢复）、禁止注入；
- **铁律 2**：降噪即判断，但此判断不得在 Normalized 留痕；软料出口为 Pack；
- **铁律 3**：手工期“可重放”指语义等价，非逐字一致；既不能保证等价即属偷偷固化真源，禁止；
- 显式声明 Normalized 非正文宿主，与 02 接缝焊死。

### 六、05 转 STABLE 的条件

05 当前卡在 PROVISIONAL 的真实原因，并非铁律不对，而是 **“降噪即判断、判断不得留痕”在手工期无机制保障**，全赖人力自觉。待手工跑数轮并能拿出“Normalized 中每个 Token 均可溯源至 Raw”的可验证证据后，方可升格为 STABLE。可为此设计一个手工期可执行的“Token 溯源比对”判据，作为准入检查。

## Claude-Opus-4.8 对 entity_naming.md 定位的澄清与修正

### 一、问题核心：一个文件混入了两样本质不同的东西

用户质疑合理——将文件编号为 `06_entity_naming.md` 放入 `contracts/` 并每轮要求追加内容，与“契约应稳定”的性质冲突。这个观察准确揭示了设计中的混淆：该文件实际包含了两种不同性质的内容：

## 项目AI（Orchestrator）单轮操作手册（精炼版）

### 角色边界

Orchestrator 只做三件事：按序调用脚本与 Extractor、读结构化结果（pass/fail + error）、按手册走控制流。不做机械检查、不做语义抽取、不手改 patch、不手写 Extractor Prompt。

### 设计原则

- **P1 单一事实源**：Raw 只读不写。
- **P2 确定性**：机械判定全部交脚本，不交给自然语言判断。
- **P3 项目隔离**：Graph 按项目分目录，不跨项目交叉。
- **P4 先复核后应用**：Patch 未过 Reconcile 不得 Apply。
- **P5 责任归属**：Lint 失败需区分“本轮引入”还是“存量问题”，处置不同。

### 主流程（每轮）

**输入**：`project_id`、`turn_id`

**STEP 0 — 启动分支**：冷启动用 `graph_state.seed.json`，否则用当前 `graph_state.json`（完整图，非切片）。

**STEP 1 — 切片**：调用 `build_graph_slice.py(完整STATE, TURN)`，产出 `extractor_context_pack.json`（含 `_runtime.next_node_id`，后续不得篡改）。只确认脚本正常产出，不解读不裁剪。

**STEP 2 — 组装 Prompt**：调用 `build_extractor_prompt.py(extractor_context_pack.json)`，产出 System + User 消息（脚本自动剥离 `retrieval_trace`）。不手动增删 prompt 内容。

**STEP 3 — 调用 Extractor**：输入 STEP 2 的 System + User，产出 `patch.json`。原样保留不修改。

**STEP 4 — 复核（P4 关卡）**：调用 `reconcile_patch.py(完整STATE, patch.json)`，读取 pass/fail（fail 带结构化 error）。Pass 进 STEP 5；Fail 进入重试回路：将结构化 error 回填至 Prompt，重走 STEP 2→3→4，最多 N 次（N 由 contract 规定）。达上限仍 fail 则停止本轮，移交评审员。

**STEP 5 — 应用**：Apply Patch 生成新 `graph_state.json`（前置：必须已过 STEP 4）。

**STEP 6 — 整图校验（P5 关卡）**：调用 `graph_lint.py(新STATE)`，读取 pass/fail（fail 列出违例）。Pass 进 STEP 7；Fail 按责任归属分流：
- **本轮引入**（应用前不存在该违例）→ 回滚至旧 STATE，将 Lint Error 作为结构化反馈进入重试回路（或达上限后失败移交）；
- **存量问题**（应用前已存在）→ 不阻塞本轮，记录该违例，随产物移交评审员，由人决定是否另开修复轮次。

### 终态移交（STEP 7）

**成功移交**（Lint Pass 或仅含存量问题）：交付 `extractor_context_pack.json`、最终 `patch.json`、新 `graph_state.json`、Lint 报告（存量问题显式标注）、重试次数。

**失败移交**（Reconcile 或 L1-Lint 达上限）：交付历次 `patch.json`、最后一次结构化 error、触发失败环节、重试次数。

移交后暂停，由人/评审员决定是否推进下一轮或采纳产物。

### 收尾自检

每轮收尾前核对五项：未改写 Raw（P1）、未 Apply 未过 Reconcile 的 Patch（P4）、产物落在本项目目录（P3）、未手动编辑 Patch/Prompt/补节点（§0）、Lint Fail 已完成 L1/L2 归属判定（P5）。

### 待确认开口

- **重试上限 N**：手册写“由 contract 规定”，若需写死请指定数字。
- **L1/L2 判定依据**：需 `graph_lint.py` 支持对旧 State 也跑一遍，或输出“违例首次出现轮次”，否则项目 AI 无机械依据做归属。是否将此作为约束补入 Contract？
- **移交粒度**：当前为每轮暂停等人。冷启动阶段最稳；待流程跑顺后可改为“连跑 K 轮再统一移交”。目前保留保守值。
